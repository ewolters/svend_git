## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

SLA-001 defines the service level agreements for the Svend platform. Every SLA is declared as a machine-readable `<!-- sla: -->` tag embedded in standards documents. The compliance parser extracts these tags, the `sla_compliance` check measures them against operational data, and results are surfaced on the public compliance page and internal dashboard.

**Core Principle:**

> If you promise it, measure it. If you can't measure it yet, say so.
> SLAs are commitments with teeth — breaches are flagged, escalated, and tracked to resolution.

### **1.2 Scope**

**Applies to:**
- Platform availability and performance
- Data durability and recovery
- Incident response timelines
- Compliance check pass rates
- Change management velocity
- Any future operational commitment declared with `<!-- sla: -->` tags

**Does NOT apply to:**
- Feature development timelines (not operational SLAs)
- LLM response quality (qualitative, not quantifiable as SLA)
- Third-party service availability (Stripe, Anthropic API) — outside our control
- Free tier users (SLAs apply to paid subscribers)

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **SLA** | Service Level Agreement — a formal commitment to a measurable service target. |
| **SLO** | Service Level Objective — the target value within an SLA (e.g., 99.9% uptime). |
| **SLI** | Service Level Indicator — the metric used to measure an SLO (e.g., health check success rate). |
| **Error Budget** | The acceptable amount of SLA non-compliance within a window (e.g., 43.8 minutes downtime/month for 99.9%). |
| **Measurement Window** | The time period over which an SLA is evaluated: monthly, quarterly, per-incident. |
| **Breach** | When the measured SLI falls below the SLO target within the measurement window. |
| **Unmeasurable** | An SLA that cannot currently be measured automatically — honest status, not a failure. |

### **4.2 Required Attributes**

| Attribute | Values | Purpose |
|-----------|--------|---------|
| `metric` | `availability`, `response_time`, `durability`, `incident_response`, `compliance`, `change_velocity` | SLA measurement type |
| `target` | Numeric with unit (e.g., `99.9%`, `2000ms`, `24h`) | The target SLO value |
| `window` | `monthly`, `quarterly`, `annually`, `per_incident` | Measurement period |
| `severity` | `critical`, `high`, `medium`, `low` | Business impact if breached |

### **4.3 Optional Attributes**

| Attribute | Default | Purpose |
|-----------|---------|---------|
| `check` | Auto-generated from description | Explicit SLA ID for deduplication |
| `measurement` | `automated` | How measured: `automated`, `manual`, `derived` |
| `soc2` | — | SOC 2 control mapping |

### **4.4 Deduplication**

When the same SLA appears in SLA-001 and in its source standard (e.g., OPS-001), the parser deduplicates by `sla_id`. Use the `check` attribute to ensure matching IDs across files.

### **5.1 Platform Availability**

<!-- sla: Platform availability | metric=availability | target=99.9% | window=monthly | severity=critical | measurement=automated | soc2=CC9.2 | check=sla-platform-availability -->

**Target:** 99.9% monthly uptime (max 43.8 minutes downtime/month)

**Measurement:** Health endpoint (`GET /api/health/`) pinged every 60 seconds by `audit.health_ping` task. Availability = (successful pings / total pings) × 100 for the current month. Results stored in `HealthPing` model, retained 90 days.

**Alerting:** 3 consecutive ping failures trigger an andon notification to all staff users (SLA-001 §12 breach escalation).

**Exclusions:**
- Planned maintenance announced 48+ hours in advance
- Third-party service outages (Stripe, Anthropic)

### **5.2 Error Budget**

At 99.9%, the monthly error budget is 43.8 minutes. If the budget is consumed:
- No non-critical deployments until next window
- Focus shifts to reliability improvements
- Post-mortem required per §12

### **6.1 API Response Time (p95)**

<!-- sla: API response time p95 | metric=response_time | target=2000ms | window=monthly | severity=high | measurement=automated | check=sla-api-p95 -->

**Target:** 95th percentile response time under 2 seconds for standard API endpoints.

**Measurement:** Automated via `PerformanceMiddleware` → `RequestMetric` bucketed aggregation. Reservoir sampling preserves percentile accuracy across 5-minute buckets. `_measure_response_time()` merges monthly samples for SLA evaluation.

**Exclusions:** DSW long-running statistical computations (which have a 120s Gunicorn timeout), file uploads, synthetic data generation jobs.

### **6.2 API Response Time (p99)**

<!-- sla: API response time p99 | metric=response_time | target=5000ms | window=monthly | severity=medium | measurement=automated | check=sla-api-p99 -->

**Target:** 99th percentile under 5 seconds.

**Measurement:** Automated via `PerformanceMiddleware` → `RequestMetric`. Same pipeline as p95 — merges monthly reservoir samples and computes 99th percentile.

### **12.1 Severity-Based Response**

| Severity | Response Time | Actions |
|----------|---------------|---------|
| `critical` | Immediate | Incident declared, all hands, post-mortem within 48h |
| `high` | 4 hours | Investigation started, remediation CR created |
| `medium` | 24 hours | Remediation CR created, scheduled fix |
| `low` | Next sprint | Tracked as technical debt |

### **12.2 Breach Logging**

Every SLA breach MUST:
1. Be visible in the `sla_compliance` check results
2. Generate a `ChangeRequest` for remediation (linked via `compliance_check_ids`)
3. Be tracked until the SLA returns to compliance

## **13. ANTI-PATTERNS**

### **13.1 Unmeasured Promises**

**PROHIBITED:** Defining an SLA without specifying how it will be measured.

Every `<!-- sla: -->` tag MUST have a `measurement` attribute (defaults to `automated`). If measurement is not yet possible, use `measurement=manual` and document the measurement gap in §11.

### **13.2 Aspirational Targets**

**PROHIBITED:** Setting SLA targets that cannot realistically be met with current infrastructure.

99.99% uptime requires redundancy and failover that a single-server deployment does not provide. 99.9% is honest for the current architecture. Upgrade the infrastructure before upgrading the SLA.

### **13.3 Hidden Breaches**

**PROHIBITED:** Suppressing or ignoring SLA breach indicators.

If the `sla_compliance` check detects a breach, it MUST be reported (not filtered, not downgraded). The remediation path is to fix the issue, not to hide the metric.

### **13.4 SLA Without Escalation**

**PROHIBITED:** Defining an SLA without a breach response plan.

Every SLA severity level has a defined escalation path in §12. New SLAs must fit within these severity tiers.
