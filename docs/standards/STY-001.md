## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

STY-001 codifies all naming, commenting, import ordering, and file structure conventions for the Kjerne/Svend codebase. It consolidates rules from STANDARD.md §2, DAT-001 §7.6, and observed codebase patterns into a single enforceable standard.

**Core Principle:**

> Name it once. Name it right. Name it the same everywhere.

### **1.2 Scope**

**Applies to:**
- All `.py` files in the web application (`services/svend/web/`)
- All Django models, views, URLs, templates, and management commands
- All `syn/` infrastructure code and application code

**Does NOT apply to:**
- Migration files (auto-generated by Django)
- Third-party code and vendored libraries
- `__pycache__/`, `staticfiles/`, `media/` directories
- Non-Python assets (CSS, JS, HTML content) — see FE-001

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **snake_case** | Lowercase words separated by underscores: `my_function`, `user_count` |
| **PascalCase** | Capitalized words concatenated: `MyClass`, `SynaraEntity` |
| **UPPER_SNAKE** | Uppercase words separated by underscores: `MAX_RETRIES`, `ALL_CHECKS` |
| **kebab-case** | Lowercase words separated by hyphens: `paired-t-test`, `gage-rr` |
| **Import group** | A block of import statements separated from other groups by a blank line |
| **Compliance comment** | An inline comment linking code to a standard: `# Compliance: AUD-001 §5` |
| **Module docstring** | The first string literal in a Python file, describing its purpose |

## **5. IMPORT ORDERING**

<!-- rule: mandatory -->

<!-- assert: Imports follow 3-group ordering convention | check=sty-import-order -->
<!-- impl: syn/audit/compliance.py:check_code_style -->
<!-- test: syn.audit.tests.test_code_style.ImportOrderTest.test_key_files_import_order -->
<!-- test: syn.audit.tests.test_code_style.ImportOrderTest.test_no_wildcard_imports -->

### **5.1 Three-Group Convention**

Imports are organized in three groups, separated by blank lines:

<!-- code: correct -->
```python
# Group 1: Python standard library (alphabetical)
import hashlib
import json
import logging
from pathlib import Path

# Group 2: Third-party packages (alphabetical)
from django.conf import settings
from django.db import models
from django.test import SimpleTestCase

# Group 3: Local/project imports (alphabetical)
from syn.audit.models import SysLogEntry
from syn.core.base_models import SynaraEntity
from .models import DSWResult
```

### **5.2 Import Rules**

1. Groups separated by exactly one blank line
2. Each group sorted alphabetically
3. `import X` before `from X import Y` within a group
4. Relative imports (`.models`) in Group 3, after absolute local imports
5. No wildcard imports (`from module import *`)

<!-- code: prohibited -->
```python
# PROHIBITED: wildcard import
from syn.audit.models import *

# PROHIBITED: mixed groups without blank line separator
import json
from django.db import models
from .utils import helper
```

<!-- rule: mandatory -->

<!-- assert: Python files have module-level docstrings | check=sty-module-docstring -->

<!-- test: syn.audit.tests.test_code_style.ModuleDocstringTest.test_key_files_have_docstrings -->

### **6.1 Module Docstrings**

Every `.py` file (except empty `__init__.py`) has a module-level docstring as the first statement:

<!-- code: correct -->
```python
"""
Synara Compliance Automation Engine.

Provides automated compliance checks for SOC 2, ISO 27001, and NIST frameworks.

Compliance: CMP-001 (Compliance Automation), SOC 2 CC4.1
"""
```

### **6.2 Class Docstrings**

Every class has a docstring describing its purpose:

<!-- code: correct -->
```python
class SysLogEntry(SynaraImmutableLog):
    """Hash-chained, immutable audit log entry.

    Standard: AUD-001 §5 (Immutable Audit Log)
    """
```

### **6.3 Compliance Comments**

Code linking to standards uses this format:

<!-- code: correct -->
```python
# Compliance: AUD-001 §5 — immutable audit log
# Standard: SEC-001 §5.2 (Tenant Isolation)
```

### **6.4 Inline Comments**

- Preceded by at least 2 spaces before `#`
- Space after `#`: `# comment` not `#comment`
- Explain *why*, not *what* (the code shows what)

## **7. FILE STRUCTURE**

<!-- rule: mandatory -->

<!-- assert: Module layout follows prescribed order: docstring, imports, constants, logger, helpers, public code | check=sty-module-layout -->
<!-- impl: syn/audit/compliance.py:check_code_style -->
<!-- test: syn.audit.tests.test_code_style.ImportOrderTest.test_key_files_import_order -->
<!-- test: syn.audit.tests.test_code_style.ModuleDocstringTest.test_key_files_have_docstrings -->

### **7.1 Module Layout Order**

Every `.py` file follows this top-to-bottom order:

1. Module docstring
2. Imports (3 groups per §5)
3. Module-level constants (`UPPER_SNAKE`)
4. Logger declaration: `logger = logging.getLogger(__name__)`
5. Private helper functions (`_prefixed`)
6. Public functions and classes
7. Django-specific: Models → Managers → Serializers → Views

### **7.2 File Purpose Patterns**

| Purpose | Pattern | Location |
|---------|---------|----------|
| Models | `models.py` or `models/{domain}.py` | App root or `models/` subdirectory |
| Views | `{feature}_views.py` | App root |
| URLs | `urls.py` or `{feature}_urls.py` | App root |
| Tests | `test_{module}.py` | App root or `tests/` subdirectory |
| Admin | `admin.py` | App root |
| Serializers | `serializers.py` | App root |
| Constants | `constants.py` | App root |
| Utilities | `utils.py` | App root |
| Signals | `signals.py` | App root |
| Events | `events.py` | App root |
| Tasks | `tasks.py` | App root |
| Apps config | `apps.py` | App root |
| Management commands | `management/commands/{verb}_{noun}.py` | Standard Django location |

### **8.1 Naming Violations**

<!-- rule: mandatory -->

<!-- assert: No prohibited naming anti-patterns in model fields (mutable defaults, missing prefixes, FK suffixes) | check=sty-no-field-antipatterns -->
<!-- impl: syn/audit/compliance.py:check_code_style -->
<!-- test: syn.audit.tests.test_code_style.ModelFieldNamingTest.test_no_mutable_jsonfield_defaults -->
<!-- test: syn.audit.tests.test_code_style.ModelFieldNamingTest.test_no_fk_suffix -->
<!-- test: syn.audit.tests.test_code_style.ModelFieldNamingTest.test_boolean_field_is_prefix -->

<!-- code: prohibited -->
```python
# PROHIBITED: camelCase function names
def calculateScore():
    pass

# PROHIBITED: inconsistent boolean naming
active = models.BooleanField()  # Missing is_ prefix
has_data = models.BooleanField()  # Wrong prefix (use is_)

# PROHIBITED: timestamp without _at suffix
created = models.DateTimeField()  # Should be created_at
last_modified = models.DateTimeField()  # Should be updated_at

# PROHIBITED: FK with _fk suffix
project_fk = models.ForeignKey(...)  # Should be just project

# PROHIBITED: mutable JSONField default
data = models.JSONField(default=[])  # Should be default=list
config = models.JSONField(default={})  # Should be default=dict
```

<!-- code: correct -->
```python
# CORRECT
def calculate_score():
    pass

is_active = models.BooleanField()
created_at = models.DateTimeField()
project = models.ForeignKey(...)
data = models.JSONField(default=list)
```

### **8.2 Import Violations**

<!-- code: prohibited -->
```python
# PROHIBITED: wildcard import
from django.db.models import *

# PROHIBITED: unused import (dead code)
import os  # never used

# PROHIBITED: mixed import groups
import json
from django.db import models  # third-party mixed with stdlib
```

### **8.3 Structure Violations**

<!-- code: prohibited -->
```python
# PROHIBITED: imports after code
MY_CONST = 42
import json  # imports must come before constants

# PROHIBITED: CamelCase filename
# File: MyModule.py — should be my_module.py
```

## **9. AUTOMATED ENFORCEMENT**

### **9.1 Linting Toolchain: Ruff**

<!-- rule: recommended -->

**Ruff** is the designated linting and formatting tool for all Python code in the codebase. Configuration lives in `[tool.ruff]` within `services/svend/web/pyproject.toml`.

| Setting | Value | Rationale |
|---------|-------|-----------|
| Rule sets | E (errors), F (pyflakes), W (warnings), I (isort) | Core correctness + import ordering |
| line-length | 120 | Matches existing codebase convention |
| target-version | py310 | Python 3.10+ requirement |
| exclude | migrations, static, node_modules, .venv | Auto-generated and third-party code |

Ruff replaces manual enforcement of §5 (Import Ordering) and supplements the `code_style` compliance check for naming violations.

### **9.2 Pre-Commit Hooks**

<!-- rule: recommended -->

Pre-commit hooks provide **local, development-time enforcement** of style rules. They run on `git commit` and block commits that violate configured checks. Configuration lives in `.pre-commit-config.yaml` at the repository root.

**Configured hooks:**
1. **ruff** — lint check (`ruff check`)
2. **ruff-format** — formatting check (`ruff format --check`)
3. **Django check** — `manage.py check` (syntax, model validation)

Pre-commit hooks are advisory — they catch issues before the compliance runner sees them. They do not replace the `code_style` compliance check or the `standards_compliance` check. See TST-001 §10.4 for the CI/CD layer architecture.

**Installation:** `pip install pre-commit && pre-commit install`
