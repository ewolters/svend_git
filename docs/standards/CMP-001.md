## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

CMP-001 defines the automated compliance testing architecture for Kjerne/Svend. The system provides continuous compliance monitoring through two layers: infrastructure checks (runtime/config verification) and standards-driven checks (parsed directly from standards documents). Standards are the single source of truth — update a standard, the tests follow automatically.

**Core Principle:**

> Compliance is not a checkbox. It is a living system that tests itself against its own standards.
> The same tags that define assertions in this document are parsed and executed by the system it describes.

### **1.2 Scope**

**Applies to:**
- All `docs/standards/*.md` files with machine-readable assertion tags
- Infrastructure compliance checks (SSL, dependencies, encryption, access logging)
- Monthly compliance report generation and publication
- Public compliance landing page

**Does NOT apply to:**
- Manual penetration testing (out of scope for automation)
- SOC 2 Type II audit procedures (external auditor process)
- Agent-level error handling compliance

### **1.3 Implementation**

### **6.1.1 Change Management Check Detail**

<!-- assert: change_management check enforces CHG-001 field requirements, risk assessment gates, and commit traceability | check=cmp-chg-management-detail -->
<!-- impl: syn/audit/compliance.py:check_change_management -->
<!-- rule: mandatory -->

The `change_management` check is the enforcement arm of CHG-001. It runs daily (critical rotation) and is the most comprehensive compliance check in the system — 14 validation rules with FAIL/WARNING severity classification.

**FAIL-level violations (SOC 2 CC8.1/CC3.4):**
1. Emergency changes without retroactive risk assessment (>24h)
2. ALL code-type CRs past `approved` without `RiskAssessment` record
3. Completed code CRs missing `commit_shas` — breaks CR→git traceability
4. Completed code CRs missing `log_md_ref` — breaks CR→log.md traceability
5. CRs with empty `description`

**WARNING-level violations (process gaps):**
6. Changes stuck `in_progress` >7 days
7. Completed changes missing `completed` log entry
8. Submitted+ CRs missing `justification` (non-doc/plan)
9. Approved+ CRs missing `implementation_plan` (non-doc/plan)
10. Approved+ CRs missing `rollback_plan` (feature/migration/infrastructure/security)
11. In_progress+ CRs missing `testing_plan` (non-doc/plan)
12. Enhancement/bugfix/debt past `approved` without single-agent risk assessment
13. ChangeLog `completed` entries missing `commit_sha` in details
14. Completed CRs referencing compliance language but empty `compliance_check_ids`

The check reports a `field_completeness` summary in its details output for dashboard rendering. Any FAIL-level violation sets the overall check status to `fail`. This is a daily critical check — it cannot be disabled, skipped, or overridden.

The `symbol_coverage` check walks all production `.py` files via AST, extracts every public top-level function and class (skipping `_private`, nested, test files, migrations, `__init__.py`), and classifies each symbol into one of three buckets:

| Bucket | Criteria | Risk Weight |
|--------|----------|-------------|
| **Covered** | Symbol has `<!-- impl: file.py:Symbol -->` on an assertion that also has `<!-- test: -->` hooks | 0.0 |
| **Specified, untested** | Symbol has an impl hook but the assertion has no test hooks | 0.5 |
| **Ungoverned** | No impl hook references this symbol | 1.0 |

File-level `<!-- impl: file.py -->` hooks do NOT contribute to symbol coverage. You must name the function or class. `Class.method` hooks resolve to the class (the class is the governance unit).

**Risk score** per file = `ungoverned_loc × 1.0 + specified_untested_loc × 0.5`. Files are sorted by risk descending as a remediation priority list.

Results surface on the internal dashboard Compliance tab as stacked per-module bars (green/yellow/red) and a highest-risk files list.

### **7.5 Severity-Weighted Pass Rate**

<!-- rule: mandatory -->

The compliance system computes a severity-weighted pass rate that prioritizes critical security checks over administrative hygiene checks. This supplements (does not replace) the simple pass rate.

**Weight Tiers:**

| Tier | Weight | Checks |
|------|--------|--------|
| Critical | 3.0 | audit_integrity, security_config, access_logging, change_management, secret_management, session_security, permission_coverage |
| High | 2.0 | encryption_status, ssl_tls, dependency_vuln, password_policy, rate_limiting, security_headers, backup_freshness, incident_readiness, sla_compliance, error_handling |
| Medium | 1.0 | standards_compliance, log_completeness, data_retention, privacy_data_export, caching, architecture, architecture_map, symbol_coverage, statistical_calibration, output_quality, calibration_coverage, endpoint_coverage, complexity_governance |
| Low | 0.5 | code_style, roadmap, policy_review |

**Formula:** `weighted_pass_rate = sum(weight_i × pass_i) / sum(weight_i) × 100`

Only `"pass"` counts — warnings, errors, and failures all reduce the score. Unlisted checks default to medium (1.0).

### **9.3 Layered Verification Architecture**

<!-- assert: The compliance runner (run_compliance) is the authoritative verification system — pre-commit and CI are advisory only | check=cmp-runner-authority -->

The compliance runner is the authoritative verification system (TST-001 §10.4). Pre-commit hooks and GitHub Actions are permitted as **advisory layers** that catch issues earlier in the development cycle:

```
Developer machine          Repository           Production server
┌──────────────┐     ┌──────────────────┐    ┌─────────────────────┐
│ pre-commit   │ ──→ │ GitHub Actions   │ ──→│ Compliance runner   │
│ (local lint) │     │ (lint + subset)  │    │ (AUTHORITATIVE)     │
└──────────────┘     └──────────────────┘    └─────────────────────┘
  Advisory              Advisory               Single source of truth
```

### **9.2 Pre-Commit Integration**

Pre-commit hooks (STY-001 §9.2) enforce a subset of what the compliance runner checks:

| Pre-commit hook | Compliance check equivalent |
|-----------------|----------------------------|
| `ruff check` | `code_style` (partial — naming, imports) |
| `ruff format` | STY-001 §7 file structure (partial) |
| `manage.py check` | Django system checks (not a compliance check) |

Pre-commit hooks **do not** replace:
- `standards_compliance` — requires standards parser + AST verification
- `change_management` — requires DB access to ChangeRequest records
- `audit_integrity` — requires hash chain verification
- Any check that requires ORM/DB access

### **9.3 Dependency Vulnerability Check**

The `dependency_vuln` infrastructure check uses **pip-audit** to scan installed packages against the OSV advisory database. Complemented by GitHub Dependabot for continuous monitoring (SEC-001 §11). This check runs on the Monday/Friday rotation.

## **10. ANTI-PATTERNS**

### **9.1 Hardcoded Compliance Tests**

<!-- rule: mandatory -->

**PROHIBITED:** Writing compliance tests that duplicate what standards tags already define.

Standards are the single source of truth. If a compliance check can be expressed as an `assert`/`impl`/`code` tag in a standards document, it MUST be defined there — not as a separate test file.

### **9.2 Unpublished Public Reports**

<!-- rule: mandatory -->

**PROHIBITED:** Auto-publishing compliance reports without staff review.

Reports MUST be reviewed before publication. `is_published=False` by default enforces manual review of the redacted `public_report` content.

### **9.3 Sensitive Data in Public Reports**

<!-- rule: mandatory -->

**PROHIBITED:** Including file paths, IPs, config values, or credentials in `public_report`.

The redaction boundary is the most important detail in the reporting pipeline. Verify before every publish.

## **11. ACCEPTANCE CRITERIA**

| Criterion | Validation Method |
|-----------|-------------------|
| Standards parser discovers all docs/standards/*.md files | `run_compliance --standards` lists all standards |
| Assertions are extracted from assert/impl/code tags | Parser returns Assertion objects with correct fields |
| verify_impl_exists finds file and symbol via AST | Standards check passes for known implementations |
| Infrastructure checks run on daily rotation | Scheduler executes compliance_daily on schedule |
| Monthly reports contain full_report and public_report | `run_compliance --report` generates both |
| Public report contains no file paths or IPs | Redaction verification on generated report |
| Management command supports all flags | `--all`, `--check`, `--report`, `--standards`, `--run-tests` work |
| Linked tests verified importable | `run_compliance --standards` shows test existence |
| Linked tests execute and pass | `run_compliance --standards --run-tests` shows pass/fail |
| CMP-001 tests itself via its own assertion tags | `run_compliance --standards` includes CMP-001 assertions |
| Failing assertions create DriftViolation records | `DriftViolation.objects.filter(enforcement_check="STD")` |
| Passing assertions auto-resolve open violations | `DriftViolation.objects.filter(resolved_at__isnull=False)` |
| Drift signatures are deterministic and unique | `_compute_drift_signature` unit tests |
