## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

CMP-001 defines the automated compliance testing architecture for Kjerne/Svend. The system provides continuous compliance monitoring through two layers: infrastructure checks (runtime/config verification) and standards-driven checks (parsed directly from standards documents). Standards are the single source of truth — update a standard, the tests follow automatically.

**Core Principle:**

> Compliance is not a checkbox. It is a living system that tests itself against its own standards.
> The same tags that define assertions in this document are parsed and executed by the system it describes.

### **1.2 Scope**

**Applies to:**
- All `docs/standards/*.md` files with machine-readable assertion tags
- Infrastructure compliance checks (SSL, dependencies, encryption, access logging)
- Monthly compliance report generation and publication
- Public compliance landing page

**Does NOT apply to:**
- Manual penetration testing (out of scope for automation)
- SOC 2 Type II audit procedures (external auditor process)
- Agent-level error handling compliance

### **1.3 Implementation**

| Component | Location | Purpose |
|-----------|----------|---------|

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Assertion** | A testable compliance claim extracted from a standards document via `<!-- assert: -->` tags. |
| **Infrastructure check** | A runtime verification of system configuration (SSL, passwords, dependencies, etc.). |
| **Standards check** | A verification derived from parsing machine-readable tags in `docs/standards/*.md`. |
| **Check rotation** | Daily schedule where critical checks run every day and others rotate by weekday. |
| **Public report** | A redacted version of a monthly compliance report safe for external publication. |

## **4. STANDARDS PARSER**

### **4.1 Tag Vocabulary**

<!-- assert: Standards parser recognizes all DOC-001 §7.5 tag types | check=cmp-tag-vocabulary -->
<!-- impl: syn/audit/standards.py:TAG_RE -->

The parser recognizes these HTML comment tags embedded in markdown:

| Tag | Purpose | Example |
|-----|---------|---------|
| `assert` | Testable compliance claim | `<!-- assert: claim \| check=id -->` |
| `impl` | Implementation file link | `<!-- impl: path:symbol -->` |
| `code` | Code pattern (correct/prohibited) | `<!-- code: correct -->` |
| `check` | Named check with control mappings | `<!-- check: id \| soc2=X -->` |
| `control` | External compliance control | `<!-- control: SOC 2 CC7.2 -->` |
| `rule` | Enforcement classification | `<!-- rule: mandatory -->` |

### **4.2 Parser Implementation**

<!-- assert: parse_all_standards() discovers and parses all docs/standards/*.md files | check=cmp-parse-all -->
<!-- impl: syn/audit/standards.py:parse_all_standards -->

<!-- assert: parse_standard() extracts Assertion dataclass instances from markdown | check=cmp-parse-standard -->
<!-- impl: syn/audit/standards.py:parse_standard -->

The parser iterates `docs/standards/*.md`, extracts structured `Assertion` objects from tag sequences, and groups `impl`, `code`, `check`, `control`, and `rule` tags under their preceding `assert` tag.

### **4.3 Assertion Data Structure**

<!-- assert: Assertion dataclass contains text, check_id, impls, code patterns, controls, rule, standard, section | check=cmp-assertion-dataclass -->
<!-- impl: syn/audit/standards.py:Assertion -->

<!-- code: correct -->
```python
@dataclass
class Assertion:
    text: str
    check_id: str
    impls: list = field(default_factory=list)
    code_correct: list = field(default_factory=list)
    code_prohibited: list = field(default_factory=list)
    controls: dict = field(default_factory=dict)
    rule: str = ""
    standard: str = ""
    section: str = ""
```

## **5. VERIFICATION EXECUTORS**

### **5.1 Implementation Verification**

<!-- assert: verify_impl_exists uses AST to check file and symbol existence | check=cmp-verify-impl -->
<!-- impl: syn/audit/standards.py:verify_impl_exists -->

Verifies that `<!-- impl: path:symbol -->` references point to existing files and symbols. Uses Python's `ast` module to find classes, functions, methods, and module-level variables (both `Assign` and `AnnAssign`).

### **5.2 Code Pattern Verification**

<!-- assert: verify_code_pattern checks correct patterns appear in implementation source | check=cmp-verify-code-pattern -->
<!-- impl: syn/audit/standards.py:verify_code_pattern -->

Extracts significant tokens from `<!-- code: correct -->` blocks and verifies they appear in the implementation source. Threshold: ≥60% of key patterns must match, with each pattern requiring ≥50% of significant tokens found.

### **5.3 Prohibited Pattern Verification**

<!-- assert: verify_code_absent checks prohibited patterns do NOT appear in implementation | check=cmp-verify-code-absent -->
<!-- impl: syn/audit/standards.py:verify_code_absent -->

Whitespace-normalized string matching detects anti-patterns from `<!-- code: prohibited -->` blocks. If a prohibited pattern is found, the assertion fails.

### **5.4 Assertion Runner**

<!-- assert: verify_assertion orchestrates impl, code correct, and code prohibited checks | check=cmp-verify-assertion -->
<!-- impl: syn/audit/standards.py:verify_assertion -->

Runs all applicable verifications for a single assertion:
1. Verify all `impl` references exist
2. Verify `code:correct` patterns present in primary impl
3. Verify `code:prohibited` patterns absent from all impls

## **7. COMPLIANCE REPORTING**

### **7.1 Monthly Report Generation**

<!-- assert: generate_monthly_report creates ComplianceReport with full_report and public_report | check=cmp-monthly-report -->
<!-- impl: syn/audit/compliance.py:generate_monthly_report -->

Monthly reports aggregate the previous month's `ComplianceCheck` records into:
- `full_report`: Complete internal details (per-check results, controls, durations)
- `public_report`: Redacted version safe for external publication

### **7.2 Report Redaction**

<!-- rule: mandatory -->

The `public_report` MUST NOT contain:
- File paths (e.g., `syn/audit/models.py`)
- IP addresses
- Internal configuration values
- Database credentials or connection strings

It contains only:
- Category-level pass rates
- SOC 2 control coverage count
- High-level status per check category

### **7.3 Publication Control**

Reports are created with `is_published=False`. Staff must explicitly publish via the internal dashboard after reviewing the redacted content.

## **9. ANTI-PATTERNS**

### **9.1 Hardcoded Compliance Tests**

<!-- rule: mandatory -->

**PROHIBITED:** Writing compliance tests that duplicate what standards tags already define.

Standards are the single source of truth. If a compliance check can be expressed as an `assert`/`impl`/`code` tag in a standards document, it MUST be defined there — not as a separate test file.

### **9.2 Unpublished Public Reports**

<!-- rule: mandatory -->

**PROHIBITED:** Auto-publishing compliance reports without staff review.

Reports MUST be reviewed before publication. `is_published=False` by default enforces manual review of the redacted `public_report` content.

### **9.3 Sensitive Data in Public Reports**

<!-- rule: mandatory -->

**PROHIBITED:** Including file paths, IPs, config values, or credentials in `public_report`.

The redaction boundary is the most important detail in the reporting pipeline. Verify before every publish.
