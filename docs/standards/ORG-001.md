## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

ORG-001 defines the organizational hierarchy, multi-site management, and permission model that governs how users, teams, sites, and resources relate within Svend. It consolidates the org/tenant patterns currently distributed across SEC-001 §5–6 and DAT-001 §6 into a single authoritative reference, and extends them to cover multi-site QMS scoping.

**Core Principle:**

> An organization is a billing entity. A site is an operational boundary.
> Permission flows downward: Org → Site → Resource. A user's effective
> permission on any resource is the intersection of their org role,
> their site access (if applicable), and the resource's ownership scope.

### **1.2 Scope**

**Applies to:**
- Tenant (organization) lifecycle: creation, settings, deactivation
- Membership: role assignment, invitation flow, seat billing
- Site: manufacturing facility management within a tenant
- SiteAccess: per-user, per-site permission grants
- Resource scoping: how Projects, QMS records, and Hoshin artifacts inherit org/site boundaries
- Permission resolution: how org role + site access + tier combine at runtime

**Does NOT apply to:**
- Individual (non-org) user permissions — those remain per SEC-001 §5
- Billing mechanics (Stripe integration) — per BILL-001
- Data model field definitions — per DAT-001 §6 (this standard defines the *rules*, DAT-001 defines the *schema*)
- LLM or AI access controls — per LLM-001

### **2.2 Ownership Rules**

Every resource in Svend has exactly one owner context:

| **Individual** | Set | NULL | NULL | Only that user |

### **2.3 Individual-to-Org Promotion**

When an individual user creates or joins an organization:
1. Existing individual resources (Projects, DSW results, etc.) remain **user-owned**
2. New resources created within the org context are **tenant-owned**
3. There is no automatic migration of individual resources into the org
4. Users can explicitly transfer projects to the org (future feature)

## **3. MEMBERSHIP & ROLES**

### **3.1 Org Roles**

Four roles with strictly hierarchical permissions:

| Role | View | Edit | Admin | Billing | Assign Roles |
|------|------|------|-------|---------|--------------|
| **VIEWER** | ✓ | — | — | — | — |
| **MEMBER** | ✓ | ✓ | — | — | — |
| **ADMIN** | ✓ | ✓ | ✓ | — | Up to MEMBER |
| **OWNER** | ✓ | ✓ | ✓ | ✓ | Any role |

<!-- assert: Membership model enforces four-tier role hierarchy (OWNER > ADMIN > MEMBER > VIEWER) with can_edit and can_admin properties | check=org-role-hierarchy -->
<!-- impl: core/models/tenant.py:Membership -->

**Admin** includes:
- Manage members (invite, remove, change roles up to MEMBER)
- Edit org settings
- Access ALL sites in the org (implicit SiteAccess)
- Manage SiteAccess grants for other users

**Owner** adds:
- Billing management (Stripe portal, plan changes)
- Assign ADMIN and OWNER roles
- Delete the organization
- Transfer ownership

### **3.2 Invitation Flow**

```
Owner/Admin creates invite
  → OrgInvitation(status=PENDING, expires_at=+7d)
  → Stripe seat added (prorated)
  → Email sent with token link

User accepts:
  → Membership created (role from invitation)
  → OrgInvitation.status → ACCEPTED
  → User.tier does NOT change (org features via Membership)

If expired/cancelled:
  → Stripe seat removed
  → OrgInvitation.status → EXPIRED | CANCELLED
```

### **3.3 Constraints**

- One membership per user per tenant (`unique_together`)
- Cannot remove the last OWNER
- Cannot demote the last OWNER
- ADMIN can only assign roles up to MEMBER (not ADMIN or OWNER)
- User can belong to multiple orgs (each is a separate Membership)

## **4. MULTI-SITE ARCHITECTURE**

### **4.1 What a Site Represents**

A **Site** is a physical or logical operational boundary — typically a manufacturing plant, warehouse, distribution center, or office. Sites exist within a single Tenant. A Tenant may have one or many Sites.

Sites carry:
- Identity: name, code (e.g., "PLT-01"), business unit
- Contacts: plant manager, CI leader, controller (text fields — not FKs to User because plant floor personnel may not have Svend accounts)
- Status: `is_active` flag for decommissioned sites

<!-- assert: Site model represents an operational boundary within a tenant with identity, contacts, and active status | check=org-site-model -->
<!-- impl: agents_api/models.py:Site -->

### **4.2 Site Access Control**

Site access is a **separate layer** from org membership. Being an org member does NOT automatically grant access to all sites (unless you are OWNER or ADMIN).

**SiteAccess** grants a specific user access to a specific site, with a site-level role:

| Site Role | Read | Write | Site Admin |
|-----------|------|-------|------------|
| **VIEWER** | ✓ | — | — |
| **MEMBER** | ✓ | ✓ | — |
| **ADMIN** | ✓ | ✓ | ✓ |

**Org role overrides:**
- Org OWNER/ADMIN → implicit access to ALL sites (read + write + site admin)
- Org MEMBER/VIEWER → access ONLY to sites with explicit SiteAccess grants

### **4.3 Permission Resolution**

The effective permission for any site-scoped resource is resolved by this algorithm:

```
1. Is user authenticated?                     → No: 401
2. Does user have required tier/feature?      → No: 403 (upgrade)
3. Does user have active Membership in tenant? → No: 403 (not a member)
4. Is user org OWNER or ADMIN?                → Yes: full access (skip step 5)
5. Does user have SiteAccess for this site?   → No: 404 (not visible)
   5a. What is the SiteAccess.role?
       VIEWER → read only
       MEMBER → read + write
       ADMIN  → read + write + site admin
```

Resources that are site-scoped:

## **5. QMS MULTI-SITE EXTENSION**

### **5.1 Principle**

QMS records (NCR, CAPA, FMEA, RCA, A3) are currently user-scoped (`owner` FK). For multi-site organizations, these records need to be **optionally site-scoped** so that:
- A site manager sees only their site's NCRs/CAPAs
- An org admin sees NCRs/CAPAs across all sites
- Management reviews can aggregate metrics per-site or org-wide
- Recurrence detection operates within the org boundary, not just per-user

### **5.2 Site-Scoping Pattern for QMS Records**

QMS models that need site context add an **optional** `site` FK and a **`created_by`** audit field:

```python
site = models.ForeignKey(
    "agents_api.Site",
    on_delete=models.SET_NULL,
    null=True,
    blank=True,
    related_name="<model>_records",
)
created_by = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.SET_NULL,
    null=True,
    related_name="<model>_created",
)
```

**Ownership adheres to §2.2:**
- `site=NULL, owner=Set` → individual user-scoped (no org context)
- `site=SET, owner=NULL` → site-scoped, tenant inferred from `site.tenant`
- `created_by` preserves authorship for audit trail regardless of ownership context
- When a record is assigned to a site, `owner` is cleared (ownership transfers to the org/site)
- `owner` becomes nullable on all QMS models to support this transition

<!-- assert: QMS records support optional site FK with ownership transfer per §2.2 and created_by audit field | check=org-qms-site-scoping -->

### **5.3 QMS Query Patterns**

For org users querying QMS records:

```python
# Individual user (no org):
records = Model.objects.filter(owner=user)

# Org member (site-scoped view):
accessible_sites = _get_accessible_sites(user, tenant)
records = Model.objects.filter(
    Q(created_by=user) |  # Records I created (read-only if site not accessible)
    Q(site__in=accessible_sites)  # Records at my accessible sites
)

# Org admin (all sites):
records = Model.objects.filter(
    Q(site__tenant=tenant) |
    Q(created_by__in=tenant_members, site__isnull=True)
)
```

**Edit permissions** are separate from visibility:
- A user can always **see** records they created (`created_by=user`)
- A user can only **edit** records at sites where they have write access (`_check_site_write`)
- Org OWNER/ADMIN can edit any record in the org
- Records without a site (`owner=user`) can only be edited by the owner

### **5.4 Models Requiring Site FK**

| Model | Current Scoping | Site FK Needed | Rationale |
|-------|----------------|----------------|-----------|
| `NonconformanceRecord` | `owner` FK | Yes | NCRs originate at a specific site |
| `CAPAReport` | `owner` FK | Yes | CAPAs are corrective actions at a site |
| `FMEA` | `user` FK | Yes | FMEAs assess risk at a specific process/site |
| `RCASession` | `owner` FK | Yes | Root cause analysis for a site-specific event |
| `A3Report` | `user` FK | Yes | Problem solving at a site |
| `InternalAudit` | `owner` FK | Yes | Audits target a specific site |

### **5.5 Cross-Site Analytics**

Management review auto-populate (FEAT-008) and NCR analytics (FEAT-010) extend to support site filtering:

- `GET /api/iso/ncrs/analytics/?site=<uuid>` — Pareto for a specific site
- `GET /api/capa/copq/?site=<uuid>` — CoPQ for a specific site

## **7. EMPLOYEE & RESOURCE MANAGEMENT**

### **7.1 Employee vs. User**

Not all plant personnel have Svend accounts. The `Employee` model represents anyone who participates in CI activities:

| Entity | Has Svend Account | Svend Access | Used For |
|--------|-------------------|--------------|----------|
| User | Yes | Full platform | Dashboard, analysis, QMS |
| Employee | Optional (`user_link` FK) | None directly | Hoshin assignments, training matrix |

**When `user_link` is set:**
- Employee becomes a linked shadow of the User
- Resource commitments, training records, and action items can reference either
- The User can see their own assignments in the dashboard

### **7.2 Resource Commitments**

Resource commitments track who is assigned to which Hoshin project, at what capacity:

```
Employee → ResourceCommitment → HoshinProject
           ├── role (facilitator/team_member/sponsor/process_owner/subject_expert)
           ├── hours_per_day
           ├── start_date / end_date
           └── status (requested → confirmed → active → completed)
```

**Availability check:** Before confirming a commitment, check for overlapping active commitments on the same employee. Warn (don't block) if an employee is over-committed.

## **9. MIDDLEWARE & REQUEST CONTEXT**

### **9.1 Tenant Resolution**

The `TenantIsolationMiddleware` runs after authentication and sets:
- `request.tenant` — the user's active tenant (or None for individual users)
- `request.membership` — the user's Membership in that tenant (or None)

This context is available to all views without additional queries.

### **9.2 Site Context**

Site context is resolved per-endpoint, not in middleware. Views that operate on site-scoped resources call:
- `_require_tenant(user)` → returns `(tenant, error_response)`
- `_get_accessible_sites(user, tenant)` → returns `(sites_qs, is_org_admin)`

This pattern avoids middleware overhead for endpoints that don't need site context.

## **11. MIGRATION PATH**

### **11.1 Current State**

QMS records are user-scoped. Multi-site is implemented only for Hoshin Kanri.

### **11.2 Target State**

QMS records support optional site FK. Queries respect org/site boundaries. Management reviews aggregate per-site.

### **11.3 Migration Steps**

1. Extract site permission helpers (`_get_accessible_sites`, `_check_site_read`, `_check_site_write`, `_require_tenant`) from `hoshin_views.py` to shared `agents_api/permissions.py`
2. Add optional `site` FK + `created_by` FK to QMS models (NCR, CAPA, FMEA, RCA, A3, InternalAudit, ControlledDocument, TrainingRequirement); make `owner` nullable
3. Add site-aware query helpers to QMS views with read/write permission separation
4. Extend analytics endpoints with `?site=<uuid>` filter
5. Extend management review auto-populate with per-site breakdowns
6. Add SiteAccess management endpoints (§8.3)
7. Update recurrence detection to operate org-wide

Each step is a separate ChangeRequest per CHG-001.
