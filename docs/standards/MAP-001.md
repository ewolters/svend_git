## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

MAP-001 defines the machine-readable architecture map for Kjerne/Svend. It is the single source of truth for what standards exist, what status they have, and which codebase modules they govern. The compliance system parses this document and enforces it.

**Core Principle:**

> If it exists, it's mapped. If it's mapped, it's enforced. If it drifts, it's caught.

### **1.2 Scope**

**Applies to:**
- All standard IDs referenced anywhere in the codebase
- All Python modules under `syn/` and top-level Django apps
- Git-tracked changes and their relationship to standards
- Compliance check `architecture_map` enforcement

**Does NOT apply to:**
- External dependencies or third-party libraries
- Non-Python assets (CSS, JS, HTML) — see FE-001
- Standards content or quality — see individual standards

### **1.3 Architecture**

The architecture map operates on three levels:

| Level | What | How |
|-------|------|-----|
| **Standards Registry** | Every standard ID with status | §4 table parsed by compliance check |
| **Module Map** | Every codebase module with governing standards | §5 table parsed by compliance check |
| **Git Tracking** | Changed files mapped to affected standards | §6 rules enforced by compliance check |

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Standards Registry** | Machine-readable table of all known standard IDs with their lifecycle status |
| **Module Map** | Machine-readable table mapping codebase directories to their governing standards |
| **Phantom Standard** | A standard ID referenced in code but with no corresponding `.md` file |
| **Registry Drift** | An APPROVED standard whose `.md` file is missing from disk |
| **Phantom Drift** | A standard ID found in code that does not appear in the registry |
| **Coverage Drift** | A codebase directory containing `.py` files that is not listed in the module map |
| **File Drift** | A module map entry whose directory no longer exists on disk |
| **Affected Standards** | The set of standards whose governing modules contain files changed in a git commit |

## **4. STANDARDS REGISTRY**

### **4.1 Registry Rules**

<!-- rule: mandatory -->

1. Every standard ID that appears in any `.md` or `.py` file MUST have an entry in the registry table
2. Every `.md` file in `docs/standards/` MUST have an APPROVED entry
3. Status values: `APPROVED` (has `.md` file), `PLANNED` (genuine future need, no `.md` yet), `PHANTOM` (referenced in code, no `.md`), `DEPRECATED` (superseded, absorbed into parent standard, or legacy)
4. The registry is the canonical list — if an ID is not here, it does not exist

### **6.1 Change Detection**

When the `architecture_map` check runs, it:

1. Calls `git diff --name-only` to identify files changed since the last compliance run
2. Maps each changed file to its module map entry by path prefix matching
3. Collects the union of governing standard IDs across all changed files
4. Reports which standards are "affected" by recent changes

### **6.2 Git Metadata on Drift Records**

DriftViolation records created by the architecture map check populate:
- `git_commit_sha` — current HEAD commit hash
- `git_author` — author of the HEAD commit

These fields were previously unpopulated across all compliance checks.

### **6.3 Graceful Degradation**

If git is unavailable (not installed, not a git repo, subprocess timeout), the check continues without git data. Change tracking is informational, not enforcement — the check does not fail due to git unavailability.

### **7.2 Phantom Reference Scanning**

<!-- assert: Architecture map check detects phantom references not in registry | check=map-phantom-detection -->
<!-- impl: syn/audit/compliance.py:_scan_phantom_references -->
<!-- test: syn.audit.tests.test_architecture_map.PhantomScannerTest.test_no_unregistered_phantoms -->
<!-- test: syn.audit.tests.test_architecture_map.PhantomScannerTest.test_registered_standards_not_flagged -->

The scanner searches `.py` files for patterns matching `[A-Z]{2,6}-\d{3,4}` and compares against the registry. Known non-standard codes (SHA-256, AES-128, HTTP-200, etc.) are excluded.

Registered PHANTOM entries are expected — they acknowledge the reference exists. Only **unregistered** references trigger drift warnings.

### **7.3 DriftViolation Records**

Architecture map findings create DriftViolation records with:
- `enforcement_check = "STD"`
- `detected_by = "architecture_map_check"`
- `git_commit_sha` and `git_author` populated from current HEAD
- Signature-based deduplication (same as CMP-001 §5.6)

## **8. ANTI-PATTERNS**

### **8.1 Unregistered Standard References**

<!-- rule: mandatory -->

**PROHIBITED:** Referencing a standard ID in code without registering it in §4.

```python
# Wrong — SCH-099 doesn't exist in MAP-001 registry
class MyTask:
    """Standard: SCH-099 §4 (Custom Task Patterns)"""
    pass
```

**CORRECT:** Add the ID to the §4 registry table first (as PHANTOM or PLANNED), then reference it.

```python
# Right — SCH-002 is registered as PHANTOM in MAP-001 §4
class MyTask:
    """Standard: SCH-002 §4 (Scheduler Implementation)"""
    pass
```

### **8.2 Unmapped Modules**

<!-- rule: mandatory -->

**PROHIBITED:** Creating a new directory with Python code without adding it to §5.

**CORRECT:** When adding a new module directory, add a row to the §5 module map table with at least one governing standard.

### **8.3 Stale Map Entries**

<!-- rule: mandatory -->

**PROHIBITED:** Leaving module map entries for directories that have been deleted or moved.

**CORRECT:** When removing or relocating a module, update the §5 module map table in the same change.
