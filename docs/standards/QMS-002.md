## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

This standard defines the resource management subsystem for Hoshin Kanri deployment. It governs Employee/Contact registry, ResourceCommitment tracking, token-based non-user participation, and site event calendar visualization.

### **1.2 Scope**

Covers roadmap items D-039 through D-052 (Phase 2.6). Phase 4B implements sub-phases 2a and 2b (Employee, ResourceCommitment, ActionToken). Calendar and workload views (2c) are deferred.

### **1.3 Relationship to QMS-001**

QMS-001 §4.5 defines Hoshin Kanri projects, KPIs, and the X-Matrix. QMS-002 adds the **people side** — who is committed to which project, when, and whether they're available. Resource management is a prerequisite for credible Hoshin deployment at scale.

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Employee** | A non-user personnel record representing a CI participant. May or may not have a linked Svend user account. |
| **ResourceCommitment** | A time-bounded assignment of an Employee to a HoshinProject with a defined role and capacity allocation. |
| **ActionToken** | A cryptographically random, time-limited, single-use, action-scoped token enabling non-user participation via email links. |
| **Facilitator** | An Employee assigned to a HoshinProject in the `facilitator` role, responsible for driving CI deployment. |
| **Availability** | The absence of overlapping ResourceCommitment date ranges for a given Employee, indicating capacity for new assignments. |
| **Feature Gating** | Restriction of endpoint access to tenants with the `hoshin_kanri` feature flag (Enterprise tier). |

**Security requirements (SEC-001 review required):**
- Tokens MUST be cryptographically random (≥32 bytes, `secrets.token_urlsafe`)
- Tokens MUST expire after 72h or first use (whichever first)
- Tokens MUST be action-scoped — a `confirm_availability` token cannot update project data
- Token endpoints MUST NOT expose any data beyond the scoped action

## **9. ANTI-PATTERNS**

### **9.1 Roles Without Feature Gating**

**PROHIBITED:** Exposing resource management endpoints without checking the `hoshin_kanri` feature flag.

All Employee, ResourceCommitment, and Calendar endpoints MUST be gated behind the Enterprise tier feature flag. Only ActionToken endpoints (`/action/<token>/`) are exempt because they serve non-user participants who have no account context.

### **9.2 Email Templates With Hardcoded URLs**

**PROHIBITED:** Embedding absolute URLs (e.g., `https://svend.ai/action/abc123`) directly in email template strings.

All email links MUST be constructed using Django's `{% url %}` tag or `reverse()` with the `ActionToken.token` value. Hardcoded domains break across environments (local, staging, production) and create a security risk if the domain changes.

### **9.3 ActionTokens Without Scope Enforcement**

**PROHIBITED:** Accepting any action on an ActionToken endpoint without verifying the token's `action_type` and `scoped_to` fields match the requested operation.

A `confirm_availability` token MUST NOT be usable to `update_progress`. Every token action handler MUST validate that the requested action matches the token's declared scope. Failure to enforce scope is a privilege escalation vulnerability.
