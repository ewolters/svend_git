## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

This standard defines the resource management subsystem for Hoshin Kanri deployment. It governs Employee/Contact registry, ResourceCommitment tracking, and site event calendar visualization. All participants have Svend accounts (viewer role or above) — non-user token participation is deprecated in favor of viewer accounts (CANON-001 §5.4).

### **1.2 Scope**

Covers roadmap items D-039 through D-052 (Phase 2.6). Phase 4B implements sub-phases 2a and 2b (Employee, ResourceCommitment, viewer role integration). Calendar and workload views (2c) are deferred.

### **1.3 Relationship to QMS-001**

QMS-001 §4.5 defines Hoshin Kanri projects, KPIs, and the X-Matrix. QMS-002 adds the **people side** — who is committed to which project, when, and whether they're available. Resource management is a prerequisite for credible Hoshin deployment at scale.

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Employee** | A personnel record representing a CI participant. Always linked to a Svend user account (viewer role or above). |
| **ResourceCommitment** | A time-bounded assignment of an Employee to a HoshinProject with a defined role and capacity allocation. |
| **Facilitator** | An Employee assigned to a HoshinProject in the `facilitator` role, responsible for driving CI deployment. |
| **Availability** | The absence of overlapping ResourceCommitment date ranges for a given Employee, indicating capacity for new assignments. |
| **Feature Gating** | Restriction of endpoint access to tenants with the `hoshin_kanri` feature flag (Enterprise tier). |

```

  ├── employee      FK → Employee
  ├── project       FK → HoshinProject
  ├── role          (facilitator / team_member / sponsor / process_owner / subject_expert)
```

Status lifecycle: `requested → confirmed → active → completed` (or `declined` from `requested`).

Availability check: before creating a commitment, query for overlapping date ranges on the same employee. Return conflicts. Allow override with acknowledgment.

### **4.3 ActionToken — DEPRECATED**

**Decision (2026-03-07):** ActionToken is deprecated. Non-user participation via token-authenticated links is replaced by viewer-role accounts (CANON-001 §5.4). All participants now have Svend accounts with appropriate role-based access. Email-based quick actions for authenticated users are handled by NotificationToken (NTF-001 §5.2).

The ActionToken model, endpoints, and tests should be removed when the viewer role migration is complete.

## **7. EMAIL TEMPLATES**

### **7.1 Availability Request**

Triggered when `ResourceCommitment` is created with status `requested`. Contains confirm/decline/suggest-alternative links via NotificationToken (NTF-001).

### **7.2 Progress Update**

Weekly digest to committed team members with project status and task update link.

### **7.3 Completion Notification**

Sent when HoshinProject transitions to `completed`. Contains savings summary and KPI impact.

## **9. ANTI-PATTERNS**

### **9.1 Roles Without Feature Gating**

**PROHIBITED:** Exposing resource management endpoints without checking the `hoshin_kanri` feature flag.

All Employee, ResourceCommitment, and Calendar endpoints MUST be gated behind the Enterprise tier feature flag.

### **9.2 Email Templates With Hardcoded URLs**

**PROHIBITED:** Embedding absolute URLs (e.g., `https://svend.ai/action/abc123`) directly in email template strings.

All email links MUST be constructed using Django's `{% url %}` tag or `reverse()` with the NotificationToken value (NTF-001). Hardcoded domains break across environments (local, staging, production) and create a security risk if the domain changes.

### **9.3 ActionTokens Without Scope Enforcement — DEPRECATED**

Removed with ActionToken (§4.3). Scope enforcement for email quick-actions is handled by NotificationToken (NTF-001 §5.2).
