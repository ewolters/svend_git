## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

XRF-001 defines how standards reference each other, how those references are formatted, and how they can be validated — manually today, automatically when tooling is built.

**Core Principle:**

> Cross-references are the nervous system of standards. A broken reference is a broken contract.
> Every reference must be parseable by a human in context and by a regex in CI.

### **1.2 Scope**

**Applies to:**
- All standards in `docs/standards/` (*.md files)
- Python code with compliance comments (`# Compliance: MOD-001 §4.1`)
- Architecture documents referencing standards
- CLAUDE.md and STANDARD.md when citing standards

**Does NOT apply to:**
- External URLs (separate concern)
- Informal documentation (READMEs, inline comments)
- Planning documents unless they cite standards
- Vendor documentation in `docs/`

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Cross-reference** | A link from one standard/section to another, using § notation. |
| **Internal reference** | Reference within the same document: `§5.1` |
| **External reference** | Reference to another standard: `API-001 §3.2` |
| **Dependency** | Standard A depends on Standard B if A references B in its Compliance header. |
| **Broken reference** | Reference to a non-existent standard or section. |
| **Circular dependency** | A references B, B references A — prohibited at the Compliance level. |
| **Cardinality** | Number of incoming references to a section. High cardinality = high-impact changes. |

## **4. CROSS-REFERENCE SYNTAX**

### **4.1 Internal Section References**

**Format:** `§[SECTION-NUMBER]`

```
§1          → Section 1
§1.2        → Section 1, Subsection 2
§3.4.1      → Section 3, Subsection 4, Sub-subsection 1
```

**Usage in text:**
```markdown
See §7.2 for implementation details.
This requirement is defined in §4.1 above.
```

### **4.2 External Standard References**

**Format:** `[STANDARD-CODE] §[SECTION-NUMBER]`

```
API-001 §5.3     → API Standard, Section 5.3
DOC-001 §4       → Documentation Standard, Section 4
AUD-001 §6.1     → Audit Standard, Section 6.1
```

**With version requirement (in Compliance headers):**
```
DOC-001 ≥ 1.0 §4      → Documentation v1.0+, Section 4
SEC-001 ≥ 2.0          → Security v2.0+ (no specific section)
```

### **4.3 ISO and External Framework References**

**ISO Standards:**
```
ISO 9001:2015 §7.5      → ISO 9001, Clause 7.5
ISO 27001:2022 A.9.2    → ISO 27001, Annex A Control 9.2
```

**SOC 2 Controls:**
```
SOC 2 CC6.1             → Common Criteria 6.1
SOC 2 CC8.1             → Common Criteria 8.1
```

**NIST:**
```
NIST SP 800-53 AU-2     → NIST Security Control AU-2
```

### **4.4 Compliance Comments in Code**

Python code MAY include compliance references as comments:

```python
# Compliance: AUD-001 §4.2 — immutable audit entry
class SysLogEntry(SynaraImmutableLog):
    ...

# Compliance: SEC-001 §5.2 — tenant isolation at query level
def get_queryset(self):
    qs = super().get_queryset()
    tenant = get_current_tenant()
    if tenant:
        qs = qs.filter(tenant=tenant)
    return qs
```

**Format:** `# Compliance: [STANDARD-CODE] §[SECTION] — [brief description]`

These are informational today. When XRF tooling is built, they become machine-verifiable links.

## **5. DEPENDENCY RULES**

### **5.1 Dependency Graph**

Standards form a directed acyclic graph (DAG) through their Compliance headers.

```
DOC-001 ← XRF-001 ← (all other standards)
```

DOC-001 is the root — it has no Kjerne standard dependencies. All other standards depend on DOC-001 (for structure) and XRF-001 (for reference syntax).

### **5.2 Circular Dependencies**

**PROHIBITED** at the Compliance (dependency) level. Standard A MUST NOT require Standard B if B requires A.

Cross-references in body text are fine — only the `Compliance:` header creates a hard dependency.

**Example of allowed pattern:**
```
DOC-001 Compliance: (none — root)
XRF-001 Compliance: DOC-001 ≥ 1.0
API-001 Compliance: DOC-001 ≥ 1.0, XRF-001 ≥ 1.0
```

**Example of prohibited pattern:**
```
API-001 Compliance: ERR-001 ≥ 1.0
ERR-001 Compliance: API-001 ≥ 1.0    ← CIRCULAR
```

**Resolution:** Extract shared concepts into a lower-level standard that both can depend on.

### **5.3 Version Requirements**

The `≥` operator in Compliance headers means "this version or newer."

When a standard bumps its MAJOR version (breaking change), all dependents MUST be reviewed and their version requirements updated if needed.

## **6. VALIDATION**

### **6.1 Manual Validation (Current)**

When creating or updating a standard:

1. **Check all `§N` references** — does that section exist in the target document?
2. **Check all `CODE-NNN` references** — does that standard exist in `docs/standards/`?
3. **Check Compliance header** — are all listed standards at the required version or newer?
4. **Check for circular dependencies** — trace the Compliance headers; no cycles allowed.

### **6.2 Automated Validation (Future)**

When tooling is built (Phase 2), XRF validation will:

1. **Scan** all `.md` files in `docs/standards/` for `§` and `[A-Z]+-[0-9]+` patterns
2. **Parse** reference tokens: standard code, section number, version requirement
3. **Validate** targets exist and versions are satisfied
4. **Report** broken references, version mismatches, and circular dependencies
5. **Integrate** with pre-commit hooks and CI

**Implementation location:** `services/svend/web/syn/xrf/` (when built)

## **7. ANTI-PATTERNS**

### **7.1 Vague References**

**PROHIBITED:**
```markdown
See the API standard for details.
```

**CORRECT:**
```markdown
See API-001 §5.3 for endpoint naming conventions.
```

### **7.2 Hardcoded Section Numbers in Prose**

**PROHIBITED:**
```markdown
As mentioned in section 4.2 of DOC-001...
```

**CORRECT:**
```markdown
As defined in DOC-001 §4.2...
```

The `§` notation is machine-parseable. "section 4.2" is not.

### **7.3 Missing Version in Dependencies**

**PROHIBITED (in Compliance headers):**
```markdown
**Compliance:**
- DOC-001 (Documentation Structure)
```

**CORRECT:**
```markdown
**Compliance:**
- DOC-001 ≥ 1.0 (Documentation Structure — §4 Section Structure)
```

### **7.4 Orphan Standards**

**PROHIBITED:** A standard that is never referenced by any other standard and references nothing.

Every standard must either depend on or be depended upon. Orphan standards indicate either misplacement or missing integration.
