## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

NTF-001 defines the notification system architecture for Svend. It covers the data model, delivery channels, real-time transport (SSE), user preference management, notification lifecycle, and security constraints. The system provides timely, actionable alerts for QMS events, approvals, deadlines, and system conditions.

**Core Principle:**

> A notification system that ignores user preference is spam.
> A notification system that loses messages is broken.
> Every notification is delivered, tracked, and under the user's control.

### **1.2 Scope**

**Applies to:**
- Notification model and lifecycle (create, deliver, read, archive)
- In-app delivery via Server-Sent Events (SSE)
- Email delivery (FEAT-002 — defined here, implemented later)
- Bell icon UI in application navbar
- User notification preferences
- All notification trigger sources across QMS, DSW, and platform modules

**Does NOT apply to:**
- Marketing emails and onboarding drip campaigns (separate CRM system)
- System logging and audit trail entries (LOG-001, AUD-001)
- Internal dashboard alerts for staff (internal_views.py)
- Browser push notifications (out of scope for v1)

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Notification** | An actionable message created for a specific user in response to a system event |
| **Notification Type** | Category classifying the event that triggered the notification (e.g., `capa_status`, `ncr_overdue`) |
| **Channel** | Delivery method: `in_app` (bell icon + SSE), `email` (FEAT-002). Extensible to webhook, SMS |
| **SSE** | Server-Sent Events — W3C standard for server-to-client push over HTTP. Unidirectional |
| **Entity Reference** | Generic foreign key pattern: `entity_type` (string) + `entity_id` (UUID) pointing to the source record |
| **Preference** | User-configurable setting controlling which notification types are delivered via which channels |
| **Delivery Status** | Lifecycle state of a notification: `pending` → `delivered` → `read` (in-app) or `pending` → `sent` / `failed` (email) |
| **Unread Count** | Number of notifications where `is_read=False` for a user — displayed as badge on bell icon |
| **Stream** | A long-lived SSE connection that pushes new notifications to the client in real time |
| **Keepalive** | Periodic `ping` event sent over SSE to prevent proxy/firewall timeout |
| **Reconnection** | Client-side automatic reconnection using `Last-Event-ID` header per W3C SSE spec |

## **5. DELIVERY CHANNELS**

### **5.1 In-App (SSE)**

<!-- assert: In-app notifications are delivered via Server-Sent Events with DB polling at 3-second intervals | check=ntf-sse-delivery -->
<!-- impl: notifications/views.py:notification_stream -->

<!-- rule: mandatory -->

**Transport:** Server-Sent Events (SSE) over HTTP. Client connects to `/api/notifications/stream/` using the `EventSource` API.

**Polling Strategy:** The SSE endpoint polls the database every 3 seconds for new notifications where `recipient = user` and `created_at > last_seen_timestamp`. This avoids in-process event bus complexity while providing near-real-time delivery.

**Connection Lifecycle:**

```
Client                              Server
  |                                   |
  |-- GET /api/notifications/stream/ →|
  |   Accept: text/event-stream       |
  |   Cookie: sessionid=...           |
  |                                   |-- 200 OK
  |                                   |   Content-Type: text/event-stream
  |                                   |   Cache-Control: no-cache
  |                                   |   X-Accel-Buffering: no
  |                                   |
  |← event: ping                      |  (every 15 seconds)
  |   data: {}                        |
  |                                   |
  |← event: notification              |  (when new notification exists)
  |   id: <created_at ISO>            |
  |   data: {"id":..., "title":...}   |
  |                                   |
  |                                   |-- close after 110 seconds
  |                                   |
  |-- (auto-reconnect) ------------->|  (EventSource handles this)
  |   Last-Event-ID: <timestamp>      |  (resume from last seen)
```

**SSE Event Format:**

```
event: notification
id: 2026-03-05T12:01:55.396000
data: {"id": "uuid", "type": "capa_status", "title": "CAPA-042 moved to verification", "entity_type": "capa", "entity_id": "uuid", "created_at": "2026-03-05T12:01:55Z"}

```

**Keepalive:** Send `event: ping\ndata: {}\n\n` every 15 seconds to prevent proxy/CDN/firewall connection timeout.

**Connection Timeout:** Close the stream after 110 seconds (10 seconds under gunicorn's 120s worker timeout). The `EventSource` API automatically reconnects with `Last-Event-ID`.

### **5.2 Email (FEAT-002)**

<!-- impl: notifications/tokens.py:NotificationToken -->

#### **5.2.1 NotificationToken Model**

<!-- assert: NotificationToken uses UUID PK, user FK, notification FK, cryptographic token, 72h expiry, and single-use enforcement | check=ntf-token-model -->
<!-- impl: notifications/tokens.py:NotificationToken -->

Each notification email contains a one-click action link backed by a `NotificationToken`. The model is separate from `agents_api.ActionToken` (QMS-002) by design — ActionToken serves non-user employees in Hoshin; NotificationToken serves authenticated users for notification acknowledgement.

| Field | Type | Constraints | Purpose |
|-------|------|-------------|---------|
| `id` | UUIDField | PK, default=uuid4 | Unique identifier |
| `user` | ForeignKey(User) | CASCADE | Token owner |
| `notification` | ForeignKey(Notification) | CASCADE | Associated notification |
| `action_type` | CharField(20) | choices: acknowledge, view | Scoped action |
| `token` | CharField(64) | unique, db_index | Cryptographic credential (secrets.token_urlsafe(32)) |
| `expires_at` | DateTimeField | auto-set 72h | Token expiry |
| `used_at` | DateTimeField | null=True | Single-use enforcement |
| `created_at` | DateTimeField | auto_now_add | Creation timestamp |
| `email_sent_at` | DateTimeField | null=True | Delivery tracking |
| `email_failed_at` | DateTimeField | null=True | Failure tracking |
| `email_opened_at` | DateTimeField | null=True | Open tracking |

Meta:
- `db_table = "notification_tokens"`
- Indexes: `(token)`, `(user, -created_at)`

Security:
- Token entropy: 256 bits (`secrets.token_urlsafe(32)` → 43+ chars)
- TTL: 72 hours from creation
- Single-use: `used_at` set on first POST, subsequent requests receive HTTP 410 Gone
- Action-scoped: `action_type` restricts what the token can do
- No authentication required — the token IS the credential

#### **5.2.2 Delivery Modes**

<!-- assert: Users can configure email delivery mode as immediate, daily digest, or weekly digest via preferences | check=ntf-email-modes -->
<!-- impl: notifications/views.py:notification_preferences -->

Users select a delivery mode via `user.preferences["notifications"]["email_mode"]`:

| Mode | Behavior | Schedule |
|------|----------|----------|
| `immediate` | One email per notification, scheduled within seconds of creation | On-demand via `schedule_task()` |
| `daily` | Batch all unread notifications into one digest email | `0 8 * * *` (8:00 AM UTC daily) |
| `weekly` | Batch all unread notifications into one digest email | `0 8 * * 1` (8:00 AM Monday UTC) |

Default when `email_enabled=True` and no mode set: `immediate`.

Extended preferences structure:

```json
{
    "muted_types": ["system"],
    "email_enabled": true,
    "email_mode": "immediate"
}
```

#### **5.2.3 Email Content**

Each notification email contains:

1. **Notification title** as heading
2. **Notification message** as body text
3. **Entity link** — "View in Svend" button linking to the entity URL per §4.3 routing table
4. **Acknowledge button** — one-click token link: `https://svend.ai/ntf/{token}/`
5. **Per-type unsubscribe link** — signed URL that mutes the specific notification type
6. **Global unsubscribe link** — existing `make_unsubscribe_url()` in email footer

All emails are wrapped in the existing `EMAIL_TEMPLATE` from `api/internal_views.py` for consistent branding (responsive, Svend green #4a9f6e, dark header #1a2a1a).

Digest emails additionally include:
- User greeting
- Unread count summary
- Notification list grouped by type, each with title, timestamp, entity link, and acknowledge link
- "View all notifications" link to `/app/`

#### **5.2.4 Token Action Endpoint**

<!-- assert: Token action endpoint at /ntf/<token>/ allows unauthenticated GET (HTML) and POST (acknowledge + mark read) with 410 on expired/used tokens | check=ntf-token-endpoint -->
<!-- impl: notifications/token_views.py -->

| Method | Path | Purpose | Auth |
|--------|------|---------|------|
| GET | `/ntf/<token>/` | Render notification details + acknowledge form | None (token is credential) |
| POST | `/ntf/<token>/` | Execute action: mark notification as read, mark token as used | None (token is credential) |

This endpoint lives outside `/api/` — same pattern as `/action/<token>/` (QMS-002 ActionToken). Invalid/expired/used tokens return HTTP 410 Gone.

#### **5.2.5 Task Handlers**

<!-- assert: Notification email tasks are registered with syn.sched for immediate send, daily digest, weekly digest, and token cleanup | check=ntf-email-tasks -->
<!-- impl: notifications/tasks.py -->

| Task Name | Purpose | Schedule | Timeout | Retries |
|-----------|---------|----------|---------|---------|
| `notifications.send_email` | Send immediate email for single notification | On-demand | 30s | 3 |
| `notifications.daily_digest` | Batch digest for daily-mode users | `0 8 * * *` | 120s | 2 |
| `notifications.weekly_digest` | Batch digest for weekly-mode users | `0 8 * * 1` | 120s | 2 |
| `notifications.cleanup_tokens` | Delete expired tokens older than 7 days | `0 4 * * 0` | 60s | 1 |

**Immediate flow:** `notify()` → create `NotificationToken` → `schedule_task("notifications.send_email")` → task sends email, updates `email_sent_at`.

**Digest flow:** Cron task queries users with matching `email_mode` → for each user, fetch unread notifications since cutoff → skip if all already read → create tokens, build digest, send.

#### **5.2.6 Anti-Spam and Safety**

<!-- rule: mandatory -->

- **Async delivery:** `notify()` schedules email via `syn.sched` — never blocks the request path
- **Global opt-out:** Check `user.is_email_opted_out` before any send (CAN-SPAM compliance)
- **Type opt-out:** Check `muted_types` in preferences (already enforced by `notify()`)
- **Rate limit:** Max 20 immediate emails per user per hour. Exceeding skips with warning log
- **Digest dedup:** If a notification is already `is_read=True` at digest time, skip it
- **Failure isolation:** Email send failures update `email_failed_at` on the token but never block notification creation. `notify()` wraps all email scheduling in try/except
- **No PII in tokens:** Token URLs contain only the cryptographic token string, not user IDs or emails
- **Batch cap:** Digest tasks process max 100 users per run, re-schedule overflow

#### **5.2.7 Per-Type Unsubscribe**

<!-- assert: Per-type unsubscribe endpoint validates signed token and adds notification type to user's muted_types preference | check=ntf-type-unsub -->
<!-- impl: notifications/views.py:notification_type_unsubscribe -->

Each notification email includes a per-type unsubscribe link using Django's `Signer`:

```
https://svend.ai/api/notifications/unsubscribe/?token=<signed_payload>
```

Payload format: `{user_id}:{notification_type}`, signed with salt `"notification-type-unsub"`.

The endpoint:
1. Unsigns and validates the token
2. Adds the notification type to `user.preferences["notifications"]["muted_types"]`
3. Returns a simple HTML confirmation page

### **5.3 Channel Priority**

<!-- rule: mandatory -->

In-app notifications are always created. Email is additive — it does not replace in-app. A user who disables email still sees in-app notifications. A user who disables a notification type receives it on no channel.

### **7.1 notify() Function**

<!-- assert: notify() helper creates a notification respecting user preferences and returning the created instance or None if muted | check=ntf-helper -->
<!-- impl: notifications/helpers.py:notify -->

<!-- rule: mandatory -->

All notification creation MUST go through the `notify()` helper. Direct `Notification.objects.create()` is prohibited outside of `helpers.py`.

<!-- code: correct -->
```python
def notify(recipient, notification_type, title, message="", entity_type="", entity_id=None):
    """Create a notification, respecting user preferences.

    Returns Notification instance, or None if the type is muted.
    """
```

Behavior:
1. Check `recipient.preferences.get("notifications", {}).get("muted_types", [])` — if `notification_type` is muted, return `None`
2. Create `Notification` with provided fields
3. Return the instance

The helper does NOT trigger email or any other channel — channels read from the Notification table independently. This keeps the helper fast and side-effect-free for the calling view.

### **7.2 Usage Pattern**

<!-- code: correct -->
```python
from notifications.helpers import notify

    recipient=capa.owner,
    notification_type="capa_status",
    title=f"CAPA {capa.title} moved to {new_status}",
    message=f"Status changed from {old_status} to {new_status}.",
    entity_type="capa",
    entity_id=capa.id,
```

## **9. SECURITY**

### **9.1 Authentication**

<!-- assert: All notification endpoints require authentication — unauthenticated requests receive 401/403 | check=ntf-auth -->
<!-- impl: notifications/views.py -->

<!-- rule: mandatory -->

Every endpoint uses `@require_auth`. The SSE stream endpoint validates the session cookie before entering the generator loop.

### **9.2 User Isolation**

<!-- assert: Users can only access their own notifications — no cross-user or cross-tenant leakage | check=ntf-isolation -->
<!-- impl: notifications/views.py -->

<!-- rule: mandatory -->

All queries filter by `recipient=request.user`. Mark-read operations verify ownership before updating. The SSE stream only delivers notifications for the authenticated user.

### **9.3 SSE Connection Limits**

<!-- rule: recommended -->

Rate limit SSE connections to 2 per user (prevent tab-spam consuming all gunicorn workers). Additional connections receive 429 Too Many Requests.

Implementation: Track active SSE connections per user in an in-process dict (not DB — transient state). Clean up on connection close.

### **9.4 Payload Security**

<!-- rule: mandatory -->

Notification payloads MUST NOT contain:
- Passwords, tokens, or secrets
- Full email addresses of other users
- PII beyond what the recipient already has access to
- Raw database queries or stack traces

Notifications reference entities by UUID. The frontend resolves details by navigating to the entity page, which enforces its own access controls.

## **11. GUNICORN AND INFRASTRUCTURE**

### **11.1 Worker Compatibility**

<!-- rule: recommended -->

Current gunicorn config (`worker_class = "sync"`, `timeout = 120`, 4 workers) supports SSE:

- Each SSE connection holds 1 worker for up to 110 seconds
- Practical limit: 2-3 concurrent SSE users (need 1+ worker for regular requests)
- Acceptable for Team tier usage (< 10 concurrent users)

**Scaling path** (no code changes required):
1. `worker_class = "gthread"` — thread-based workers, ~20 concurrent SSE per process
2. `worker_class = "gevent"` — green threads, ~1000 concurrent SSE per process
3. Dedicated SSE process on separate port — complete isolation from request workers

### **11.2 Cloudflare Tunnel**

SSE works through Cloudflare Tunnel without modification. Cloudflare buffers SSE events but streams them to the client. The `X-Accel-Buffering: no` header and `Cache-Control: no-cache` disable proxy buffering.

## **12. ANTI-PATTERNS**

### **12.1 Direct Model Creation**

<!-- rule: mandatory -->

**PROHIBITED:** Creating notifications directly, bypassing the helper.

<!-- code: prohibited -->
```python
Notification.objects.create(
    recipient=user,
    notification_type="capa_status",
    title="CAPA moved",
)
```

**CORRECT:** Use the `notify()` helper, which respects preferences.

<!-- code: correct -->
```python
from notifications.helpers import notify

notify(
    recipient=user,
    notification_type="capa_status",
    title="CAPA moved",
    entity_type="capa",
    entity_id=capa.id,
)
```

### **12.2 Sensitive Data in Payload**

<!-- rule: mandatory -->

**PROHIBITED:** Including credentials or other users' PII in notification content.

<!-- code: prohibited -->
```python
notify(user, "system", f"Password reset token: {token}")
notify(user, "assignment", f"Assigned by {assigner.email} ({assigner.phone})")
```

**CORRECT:** Reference entities by ID. Include only the recipient's own context.

<!-- code: correct -->
```python
notify(user, "system", "Your password has been reset.")
notify(user, "assignment", f"You were assigned NCR: {ncr.title}", entity_type="ncr", entity_id=ncr.id)
```

### **12.3 Blocking Notification in Request Path**

<!-- rule: mandatory -->

**PROHIBITED:** Making notification creation block the primary request or adding network calls in the notify path.

<!-- code: prohibited -->
```python
def capa_transition(request, capa_id):
    # ... transition logic ...
    send_email(user.email, template)  # Blocks response
    send_slack_webhook(channel, message)  # External call
    return JsonResponse(result)
```

**CORRECT:** `notify()` is a fast DB insert only. Email/webhook delivery runs asynchronously via syn.sched.

<!-- code: correct -->
```python
def capa_transition(request, capa_id):
    # ... transition logic ...
    notify(capa.owner, "capa_status", f"CAPA moved to {new_status}",
           entity_type="capa", entity_id=capa.id)
    return JsonResponse(result)
```

### **12.4 Ad-Hoc Type Strings**

<!-- rule: mandatory -->

**PROHIBITED:** Using arbitrary strings for notification types.

<!-- code: prohibited -->
```python
notify(user, "custom_alert", "Something happened")
notify(user, "my_new_type", "New thing")
```

**CORRECT:** All types must be members of the `NotificationType` enum.

<!-- code: correct -->
```python
notify(user, NotificationType.SYSTEM, "Maintenance scheduled for tonight.")
```
