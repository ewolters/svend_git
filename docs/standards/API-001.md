## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

API-001 defines the API design conventions for Kjerne/Svend. All HTTP API endpoints follow a consistent URL structure, request parsing pattern, response format, error envelope, pagination scheme, and request tracing model. The middleware stack enforces these conventions at the infrastructure level.

**Core Principle:**

> Every API endpoint is a contract. URL shape, request format, response envelope, error codes, and tracing headers are non-negotiable.
> If a client knows one endpoint, they know all of them. Consistency is the feature.

### **1.2 Scope**

**Applies to:**
- All endpoints under `/api/` paths
- Django function-based views using `accounts/permissions.py` decorators
- DRF-based views using serializers and `@api_view`
- Middleware in `syn/api/middleware.py`
- Pagination in `syn/api/pagination.py`

**Does NOT apply to:**
- Django admin (`/admin/`)
- Health check endpoints (`/health/`, `/ready`, `/live`, `/metrics`)
- Billing/Stripe webhook endpoints (`/billing/`)
- Static file serving (`/static/`, `/media/`)
- Django template-rendered HTML pages

### **1.3 Implementation**

| Component | Location | Purpose |
|-----------|----------|---------|
| View decorators | `accounts/permissions.py` | Auth, tier gating, rate limiting |
| Request ID | `syn/api/middleware.py:SynRequestIdMiddleware` | ULID generation, W3C traceparent |
| API headers | `syn/api/middleware.py:APIHeadersMiddleware` | Accept validation, charset, Vary, timing |
| Idempotency | `syn/api/middleware.py:IdempotencyMiddleware` | POST replay, conflict detection |
| Error envelope | `syn/api/middleware.py:ErrorEnvelopeMiddleware` | Standard error wrapping |
| Pagination | `syn/api/pagination.py:SynaraCursorPagination` | Cursor-based pagination |
| URL routing | `agents_api/urls.py`, `api/urls.py`, `svend_web/urls.py` | Feature-based sub-routes |

## **3. TERMINOLOGY**

<!-- table: terminology -->

| Term | Definition |
|------|------------|
| **Endpoint** | A URL path + HTTP method combination that performs a defined action. |
| **Decorator stack** | Ordered set of Python decorators on a view function enforcing method, auth, tier, and rate limit checks. |
| **Error envelope** | Standard JSON wrapper for all error responses: `{"error": {"code", "message", "retryable", "request_id"}}`. |
| **Cursor** | Opaque base64-encoded pagination token. Clients pass it back to retrieve the next page. |
| **Idempotency key** | Client-supplied ULID in the `Idempotency-Key` header ensuring POST requests can be safely retried. |
| **Request ID** | ULID stored as `request.syn_request_id`, echoed in `Syn-Request-Id` response header. |
| **Tier gating** | Access control based on subscription tier: Free, Founder, Pro, Team, Enterprise. |

### **4.3 Naming Conventions**

<!-- rule: mandatory -->

- **Kebab-case** for multi-word path segments: `/api/dsw/paired-t-test/`, not `/api/dsw/paired_t_test/`
- **Plural nouns** for resource collections: `/api/problems/`, `/api/reports/`
- **UUID** for resource identifiers: `/api/problems/{id}/`
- **Verb suffixes** for actions on resources: `/api/problems/{id}/approve/`

### **4.4 URL Patterns**

<!-- assert: URL patterns follow list/detail/action convention | check=api-url-patterns -->

| Pattern | Example | Purpose |
|---------|---------|---------|
| List | `GET /api/problems/` | Retrieve collection |
| Detail | `GET /api/problems/{id}/` | Retrieve single resource |
| Create | `POST /api/problems/` | Create new resource |
| Update | `PUT /api/problems/{id}/` | Full resource replacement |
| Patch | `PATCH /api/problems/{id}/` | Partial resource update |
| Delete | `DELETE /api/problems/{id}/` | Remove resource |
| Action | `POST /api/problems/{id}/approve/` | Perform action on resource |
| RPC-style | `POST /api/dsw/paired-t-test/` | Stateless computation |

**Note:** DSW and SPC endpoints are predominantly RPC-style (stateless computation) rather than RESTful CRUD. The statistical engine receives data and parameters, returns results. This is by design — analysis endpoints are functions, not resources.

### **7.3 SynaraError Integration**

<!-- assert: ErrorEnvelopeMiddleware.process_exception converts SynaraError to envelope via to_envelope() | check=api-synara-error -->
<!-- impl: syn/api/middleware.py:ErrorEnvelopeMiddleware.process_exception -->

The middleware's `process_exception` handles two paths:

1. **SynaraError** (from `syn/err/`): calls `exception.to_envelope(request_id=request_id)` and returns the error's own HTTP status. See ERR-001 §4-5 for the exception hierarchy.
2. **Non-SynaraError**: returns 500 with generic `internal_error` code. Internal details are never exposed to the client (POL-002 §8 redaction).

### **7.4 Retryable Errors**

<!-- assert: Retryable status codes are 408, 429, 500, 502, 503, 504 | check=api-retryable -->
<!-- impl: syn/api/middleware.py:ErrorEnvelopeMiddleware.RETRYABLE_STATUSES -->

Retryable status codes: `{408, 429, 500, 502, 503, 504}`.

The `retryable` field in the error envelope tells clients whether the request can be safely retried. Clients SHOULD implement exponential backoff for retryable errors (see ERR-001 §6.1).

## **8. REQUEST TRACING**

### **8.1 Syn-Request-Id**

<!-- assert: SynRequestIdMiddleware generates ULID or extracts from Syn-Request-Id header | check=api-request-id -->
<!-- impl: syn/api/middleware.py:SynRequestIdMiddleware -->

`SynRequestIdMiddleware` processes every request:

1. **Extract** `Syn-Request-Id` from incoming request header
2. **Generate** a new ULID if header is absent
3. **Store** on `request.syn_request_id` for downstream access
4. **Echo** the ID in the `Syn-Request-Id` response header

<!-- code: correct -->
```python
# Accessing request ID in view code
request_id = request.syn_request_id
```

### **8.2 W3C Traceparent**

<!-- assert: SynRequestIdMiddleware validates and extracts W3C traceparent header | check=api-traceparent -->
<!-- impl: syn/api/middleware.py:SynRequestIdMiddleware -->

W3C `traceparent` header is supported for distributed tracing integration:

```
traceparent: {version}-{trace-id}-{span-id}-{trace-flags}
```

Pattern: `^[0-9a-f]{2}-[0-9a-f]{32}-[0-9a-f]{16}-[0-9a-f]{2}$`

When present and valid, the middleware extracts:
- `request.syn_trace_id` — 32-character hex trace ID
- `request.syn_span_id` — 16-character hex span ID
- `request.syn_trace_flags` — 2-character hex flags
- `request.syn_tracestate` — vendor-specific trace state (if provided)

The traceparent is echoed in the response for correlation.

### **8.3 Timing**

<!-- assert: SynRequestIdMiddleware records request start time for duration tracking | check=api-timing -->
<!-- impl: syn/api/middleware.py:SynRequestIdMiddleware -->
<!-- impl: syn/api/middleware.py:APIHeadersMiddleware -->

Request timing is tracked across two middleware:

1. `SynRequestIdMiddleware` stores `request.syn_start_time = time.time()`
2. `APIHeadersMiddleware` computes duration and adds `X-Response-Time: {N}ms` header

## **9. PAGINATION**

<!-- assert: SynaraCursorPagination uses cursor-based pagination, not offset | check=api-cursor-pagination -->
<!-- impl: syn/api/pagination.py:SynaraCursorPagination -->

All paginated endpoints MUST use cursor-based pagination via `SynaraCursorPagination`. Offset-based pagination is prohibited for performance and consistency reasons.

<!-- code: correct -->

| Parameter | Type | Default | Bounds | Description |
|-----------|------|---------|--------|-------------|
| `limit` | integer | 50 | 1-200 | Number of items per page |
| `cursor` | string | (none) | -- | Opaque base64 cursor from previous response |

<!-- assert: Paginated responses use {data, next_cursor, total_estimate} envelope | check=api-pagination-response -->

## **10. IDEMPOTENCY**

### **10.1 Idempotency-Key Header**

<!-- assert: IdempotencyMiddleware validates Idempotency-Key on POST /api/ requests | check=api-idempotency-key -->
<!-- impl: syn/api/middleware.py:IdempotencyMiddleware -->

<!-- rule: recommended -->

Clients SHOULD include an `Idempotency-Key` header on POST requests. The key MUST be a ULID (26-character Crockford base32).

Pattern: `^[0-9A-HJKMNP-TV-Z]{26}$`

### **10.2 Behavior**

<!-- assert: IdempotencyMiddleware returns cached response on key replay and 409 on payload conflict | check=api-idempotency-behavior -->
<!-- impl: syn/api/middleware.py:IdempotencyMiddleware -->

| Scenario | Behavior |
|----------|----------|
| No `Idempotency-Key` | Request processed normally (warning logged) |
| New key | Request processed; response cached on 2xx |
| Same key, same payload | Cached response replayed |
| Same key, different payload | 409 Conflict with `idempotency_conflict` error |

### **10.3 Payload Hash**

<!-- assert: Idempotency payload hash is SHA-256 of raw request body | check=api-payload-hash -->
<!-- impl: syn/api/middleware.py:IdempotencyMiddleware -->

Conflict detection uses SHA-256 hash of the raw request body. Two requests with the same idempotency key but different body content produce different hashes, triggering a 409 response.

### **10.4 Cache TTL**

<!-- assert: Idempotency cache TTL is 24 hours | check=api-idempotency-ttl -->
<!-- impl: syn/api/middleware.py:IdempotencyMiddleware._cache_response -->

Idempotency keys are cached for **24 hours** (`IDEMPOTENCY_TTL_HOURS = 24`). After expiry, the same key can be reused.

Cache key format: `idempotency:{tenant_id}:{idempotency_key}` — scoped per tenant.

## **11. ANTI-PATTERNS**

### **11.1 Returning Raw Exceptions to Clients**

<!-- rule: mandatory -->

**PROHIBITED:**
<!-- code: prohibited -->
```python
@csrf_exempt
def my_view(request):
    try:
        result = do_work()
    except Exception as e:
        return JsonResponse({"error": str(e)}, status=500)
```

**CORRECT:** Let `ErrorEnvelopeMiddleware` handle it, or raise a `SynaraError`:
<!-- code: correct -->
```python
from syn.err.exceptions import ValidationError

@csrf_exempt
@require_http_methods(["POST"])
@gated_paid
def my_view(request):
    try:
        data = json.loads(request.body)
    except json.JSONDecodeError:
        return JsonResponse({"error": "Invalid JSON"}, status=400)

    if not data.get("values"):
        raise ValidationError("Missing required field: values")

    result = do_work(data)
    return JsonResponse(result, status=200)
```

### **11.2 Offset-Based Pagination**

<!-- rule: mandatory -->

**PROHIBITED:**
<!-- code: prohibited -->
```python
page = int(request.GET.get("page", 1))
offset = (page - 1) * 50
items = queryset[offset:offset + 50]
```

**CORRECT:** Use `SynaraCursorPagination` (§9.1). Offset pagination is O(n) on large tables and produces inconsistent results during concurrent writes.

### **11.3 Missing Method Enforcement**

<!-- rule: mandatory -->

**PROHIBITED:**
<!-- code: prohibited -->
```python
@csrf_exempt
@gated_paid
def my_view(request):
    if request.method == "POST":
        ...
    elif request.method == "GET":
        ...
```

**CORRECT:** Use `@require_http_methods` to declare allowed methods explicitly. Separate views for separate methods when logic differs.

### **11.4 Inventing Custom Error Formats**

<!-- rule: mandatory -->

**PROHIBITED:**
<!-- code: prohibited -->
```python
return JsonResponse({"success": False, "msg": "Something went wrong"}, status=400)
```

**CORRECT:** Use the standard error envelope format. Return `{"error": "message"}` with the appropriate status code, and let `ErrorEnvelopeMiddleware` standardize it, or raise a `SynaraError`.

### **11.5 Ignoring Request ID**

<!-- rule: mandatory -->

**PROHIBITED:** Logging errors or returning error responses without including `request.syn_request_id`.

**CORRECT:** The middleware handles this automatically. When manually constructing error responses, include the request ID:

<!-- code: correct -->
```python
return JsonResponse({
    "error": {
        "code": "custom_error",
        "message": "Description",
        "request_id": request.syn_request_id,
    }
}, status=400)
```

**CORRECT:**

Kebab-case is the URL convention. Underscores are for Python identifiers, not URL paths.
