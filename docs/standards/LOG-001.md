## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

LOG-001 defines the logging and observability architecture for Kjerne/Svend. The logging system provides structured, correlated, multi-tenant log capture with stream-based organization, threshold alerting, time-bucketed metrics, and configurable retention policies.

**Core Principle:**

> Logs are structured data, not free-form text. Every log entry has a correlation ID, a tenant scope, a severity level, and a stream.
> If you cannot query it, filter it, and correlate it to a request in under 10 seconds, the logging is broken.

### **1.2 Scope**

**Applies to:**
- All application-level logging in `syn/` infrastructure code
- HTTP request/response logging via middleware
- Correlation ID propagation across request lifecycle
- Log stream configuration and retention policies
- Threshold-based alerting on log patterns
- Time-bucketed metric aggregation
- Django LOGGING configuration

**Does NOT apply to:**
- Audit trail entries (see AUD-001) -- audit has its own immutable, hash-chained model
- Agent-level logging in `agents/` (uses Python logging directly)
- Frontend JavaScript console logging
- External SIEM configuration (SIEM forwarding is toggled here, configured externally)

## **3. TERMINOLOGY**

<!-- table: terminology -->

| Term | Definition |
|------|------------|
| **LogStream** | Logical grouping of log entries with its own retention policy, minimum level, and optional aggregation sink. |
| **LogEntry** | A single structured log record with timestamp, level, logger, message, correlation_id, tenant_id, and optional context/metadata/exception fields. |
| **Correlation ID** | A UUID propagated through the entire request lifecycle, linking all log entries, audit records, and error traces to a single originating request. |
| **ContextVar** | Python `contextvars.ContextVar` -- thread-safe, async-safe variable scoped to the current execution context. Used for correlation_id, tenant_id, and actor_id. |
| **Hot storage** | Primary database storage with full query capability. Retention controlled by `LogStream.retention_days`. |
| **Archive storage** | Cold storage for aged-out entries. Retention controlled by `LogStream.archive_days`. |
| **Aggregation sink** | External system receiving forwarded log data: ELK, Datadog, Splunk, or CloudWatch. |
| **SIEM** | Security Information and Event Management system. Streams with `is_siem_enabled=True` forward security-relevant logs. |
| **LogAlert** | Threshold-based alert configuration that fires when a stream accumulates N entries at a given level within a time window. |
| **LogMetric** | Pre-computed time-bucketed aggregation storing count and error_count per stream/logger/level combination. |
| **W3C traceparent** | Distributed tracing header format: `version-trace_id-parent_id-flags`. CorrelationMiddleware extracts `trace_id` as correlation_id. |

### **5.1 CorrelationMiddleware**

<!-- assert: CorrelationMiddleware extracts or generates correlation_id and sets it on contextvars and response headers | check=log-correlation-middleware -->

<!-- test: syn.audit.tests.test_logging.CorrelationMiddlewareTest.test_generates_correlation_id_when_missing -->
<!-- test: syn.audit.tests.test_logging.CorrelationMiddlewareTest.test_extracts_from_x_correlation_id_header -->

**Location:** `syn/log/middleware.py`

Propagates correlation ID through the full request lifecycle.

**Header extraction order:**

<!-- table: field-definitions -->

| Priority | Header | Format |
|----------|--------|--------|
| 1 | `X-Correlation-Id` | UUID string |
| 2 | `Syn-Request-Id` | UUID string |
| 3 | `traceparent` | W3C: `version-trace_id-parent_id-flags` (extracts `trace_id`) |
| 4 | `request.syn_request_id` | Already set by upstream middleware |
| 5 | (none) | Generates new `uuid.uuid4()` |

**Lifecycle:**

1. Extract correlation_id from headers (or generate)
2. Set `correlation_id_var` via `set_correlation_id()`
3. Extract `tenant_id` from request attributes or authenticated user
4. Extract `actor_id` from authenticated user (email preferred)
5. Store `request.correlation_id` and `request.syn_correlation_id` on the request object
6. Process the request
7. Add `X-Correlation-Id` and `Syn-Request-Id` to response headers
8. Clear all context variables (`set_correlation_id(None)`, etc.)

<!-- assert: CorrelationMiddleware extracts tenant_id from request.tenant_id, user.tenant_id, or X-Tenant-Id header | check=log-tenant-extraction -->
<!-- impl: syn/log/middleware.py:CorrelationMiddleware._get_tenant_id -->
<!-- test: syn.audit.tests.test_logging.CorrelationMiddlewareTest.test_extracts_tenant_from_header -->

**Tenant extraction order:**
1. `request.tenant_id` attribute (set by tenant middleware)
2. `request.user.tenant_id` (authenticated user's tenant)
3. `X-Tenant-Id` header

## **8. CORRELATION ID PROPAGATION**

### **8.1 Lifecycle**

<!-- assert: Correlation ID is propagated from request headers through ContextVar to log records to response headers | check=log-correlation-lifecycle -->
<!-- impl: syn/log/middleware.py:CorrelationMiddleware -->
<!-- impl: syn/log/handlers.py:CorrelationFilter -->

The correlation ID flows through the following path:

```
Incoming Request
    |
    v
CorrelationMiddleware._get_correlation_id()
    |-- X-Correlation-Id header
    |-- Syn-Request-Id header
    |-- traceparent header (W3C, extract trace_id)
    |-- request.syn_request_id attribute
    |-- uuid.uuid4() (generated)
    |
    v
set_correlation_id() --> correlation_id_var (ContextVar)
    |
    v
request.correlation_id = correlation_id
request.syn_correlation_id = correlation_id
    |
    v
[Request processing]
    |-- CorrelationFilter enriches log records
    |-- SynaraLogHandler persists to LogEntry.correlation_id
    |-- AuditLoggingMiddleware passes to SysLogEntry.correlation_id
    |-- ErrorEnvelopeMiddleware includes in error response
    |
    v
Response headers:
    X-Correlation-Id: <correlation_id>
    Syn-Request-Id: <correlation_id>
    |
    v
set_correlation_id(None)  -- clear context
set_tenant_id(None)
set_actor_id(None)
```

### **8.2 W3C Traceparent Support**

<!-- assert: CorrelationMiddleware extracts trace_id from W3C traceparent header format version-trace_id-parent_id-flags | check=log-w3c-traceparent -->
<!-- impl: syn/log/middleware.py:CorrelationMiddleware._get_correlation_id -->
<!-- test: syn.audit.tests.test_logging.CorrelationMiddlewareTest.test_extracts_from_traceparent -->

The `traceparent` header follows the W3C Trace Context specification:

```
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
              ^   ^                                ^                ^
              |   trace_id (extracted)              parent_id        flags
              version
```

The middleware splits on `-` and extracts `parts[1]` as the correlation ID. This allows Kjerne to participate in distributed traces initiated by external systems.

### **8.3 Context Variable Thread Safety**

<!-- rule: mandatory -->

All context propagation uses Python `contextvars.ContextVar`, which is:
- **Thread-safe:** Each thread gets its own value
- **Async-safe:** Each asyncio task gets its own copy
- **Restorable:** `LogContext` uses `ContextVar.reset()` tokens to restore previous values

This means correlation IDs do not leak between concurrent requests, even under Gunicorn with threaded workers.

## **9. ANTI-PATTERNS**

### **9.1 Unstructured Log Messages**

<!-- rule: mandatory -->

**PROHIBITED:**
<!-- code: prohibited -->
```python
logger.info(f"User {user.email} did {action} on {resource} at {time}")
```

**CORRECT:**
<!-- code: correct -->
```python
logger.info(
    "Action performed",
    extra={
        "user_id": user.email,
        "operation": action,
        "resource_type": resource.__class__.__name__,
        "resource_id": str(resource.pk),
    },
)
```

Log messages should be static strings. Variable data goes in `extra` for structured querying.

### **9.2 Missing Correlation ID**

<!-- rule: mandatory -->

**PROHIBITED:** Creating log entries or error responses without a correlation ID.

If `CorrelationMiddleware` is in the middleware stack, `CorrelationFilter` ensures every log record gets a correlation_id. For code outside the request lifecycle (background tasks, management commands), use `LogContext`:

<!-- code: correct -->
```python
from syn.log.handlers import LogContext
import uuid

with LogContext(correlation_id=str(uuid.uuid4())):
    process_batch()
```

### **9.3 Logging Sensitive Data**

<!-- rule: mandatory -->

**PROHIBITED:** Logging passwords, tokens, API keys, or PII in log messages or context.

`RequestLoggingMiddleware` masks `authorization`, `cookie`, `x-api-key`, and `x-auth-token` headers. `AuditLoggingMiddleware` strips `password`, `token`, and `key` from query params. Application code MUST apply the same discipline.

<!-- code: prohibited -->
```python
logger.info("Auth attempt", extra={"password": request.POST["password"]})
```

### **9.4 Catching and Suppressing Handler Errors**

<!-- rule: mandatory -->

**PROHIBITED:** Wrapping `SynaraLogHandler` in try/except that silently discards failures.

The handler itself degrades gracefully (logs to stderr on database failure). Do not add additional suppression layers. If logging fails, it MUST be visible -- even if only on stderr.

### **9.5 Bypassing the Stream Model**

<!-- rule: mandatory -->

**PROHIBITED:** Writing directly to `LogEntry.objects.create()` without going through `SynaraLogHandler` or `get_synara_logger()`.

<!-- code: prohibited -->
```python
from syn.log.models import LogEntry
LogEntry.objects.create(
    stream=some_stream, level="INFO", logger="my.module",
    message="Direct write", correlation_id=uuid.uuid4(),
)
```

**CORRECT:** Use the handler or logger factory, which ensures filters run, context is enriched, and exceptions are captured:

<!-- code: correct -->
```python
from syn.log.handlers import get_synara_logger

logger = get_synara_logger(__name__)
logger.info("Correct write pattern")
```

### **9.6 Logging in Tight Loops Without Batching**

<!-- rule: recommended -->

**DISCOURAGED:** Using `batch_size=0` (immediate write) inside high-frequency loops.

For bulk operations, configure `SynaraLogHandler` with `batch_size > 0` or use a separate logger with batching enabled to avoid per-iteration database writes.
