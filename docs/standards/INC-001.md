## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

INC-001 defines the incident response management architecture for Kjerne/Svend. Every production incident — outages, degraded service, security events, data integrity issues — follows a defined lifecycle, receives severity classification, generates an immutable log chain, and is measured against SLA targets defined in SLA-001 §8.

**Core Principle:**

> Every incident has a severity, a lifecycle, an owner, and a log.
> Detection is logged. Acknowledgement is logged. Investigation is logged. Resolution is logged.
> The audit trail for an incident is a linked chain — from detection through every checkpoint to closure and post-mortem.

### **1.2 Scope**

**Applies to:**
- All production service outages or degraded availability
- Security incidents (unauthorized access, data exposure, vulnerability exploitation)
- Data integrity incidents (corruption, loss, unauthorized modification)
- Dependency failures (third-party service outages affecting Svend)
- Performance degradation affecting user experience

**Does NOT apply to:**
- Development environment issues
- Planned maintenance windows (tracked via CHG-001 as infrastructure changes)
- Feature requests or enhancement suggestions
- Non-production test failures

## **4. DETECTION AND ALERTING**

### **4.1 Automated Detection**

<!-- assert: Health ping auto-detection creates incidents on consecutive failures | check=inc-auto-detection -->
<!-- impl: syn/sched/svend_tasks.py:health_ping_handler -->

The health ping system (SLA-001 §5.1) provides automated incident detection:

- **Health endpoint** (`/api/health/`) is pinged every 60 seconds by `audit.health_ping` task
- **3 consecutive failures** trigger an availability alert via `_fire_availability_andon()`
- Staff receive `system` notifications with failure details

Future enhancement: Auto-create `Incident` records from 3+ consecutive health ping failures.

### **4.2 Manual Reporting**

Staff can create incidents manually via:
- Internal dashboard → Operations → Incidents tab → Create Incident
- API: `POST /api/internal/incidents/create/`

Required fields: `title`, `description`, `severity`, `category`.

## **5. INCIDENT LIFECYCLE**

### **5.1 Lifecycle States**

<!-- assert: Incident model contains status field tracking lifecycle states | check=inc-lifecycle-states -->
<!-- impl: syn/audit/models.py:Incident -->
<!-- test: syn.audit.tests.test_incidents.IncidentModelTest.test_status_choices -->

```
detected → acknowledged → investigating → mitigating → resolved → post_mortem → closed
```

| State | Description | Timestamp Field |
|-------|-------------|-----------------|
| `detected` | Incident identified (auto or manual) | `detected_at` |
| `acknowledged` | Staff aware and responding | `acknowledged_at` |
| `investigating` | Root cause analysis underway | `investigating_at` |
| `mitigating` | Fix being applied or workaround in place | `mitigating_at` |
| `resolved` | Service restored, incident contained | `resolved_at` |
| `post_mortem` | Post-incident review in progress | — |
| `closed` | Post-mortem complete, remediation tracked | `closed_at` |

Intermediate states may be skipped (e.g., `detected → investigating` if acknowledgement is implicit).

<!-- assert: Invalid incident state transitions are rejected by the transition API | check=inc-transition-validation -->

### **5.2 State Transition Logging**

<!-- assert: Every Incident state transition creates an immutable IncidentLog entry | check=inc-state-logging -->
<!-- impl: syn/audit/models.py:IncidentLog -->
<!-- test: syn.audit.tests.test_incidents.IncidentLogTest.test_model_exists -->
<!-- test: syn.audit.tests.test_incidents.IncidentLogTest.test_immutability -->

Every state transition MUST create an `IncidentLog` entry with:
- Timestamp (auto-set)
- Actor (who made the transition)
- Previous state → new state
- Message/notes for the transition

IncidentLog entries are **immutable** — `save()` raises `ValidationError` on update, `delete()` raises `ValueError`. This mirrors the ChangeLog pattern from CHG-001 §5.2.

Each lifecycle state has a corresponding `DateTimeField` that is set automatically when the incident transitions to that state. These timestamps are the basis for SLA measurement.

## **6. SLA TARGETS**

### **6.1 Acknowledgement SLA**

<!-- assert: Acknowledgement SLA breach threshold is 1 hour for critical/high severity per SLA-001 §8 | check=inc-ack-threshold -->

Per SLA-001 §8.1: All incidents MUST be acknowledged within the target window for their severity level.

| Severity | Ack Target |
|----------|------------|
| Critical | 1 hour |
| High | 1 hour |
| Medium | 4 hours |
| Low | 8 hours |

**Measurement:** `Incident.ack_elapsed_hours` property computes hours between `detected_at` and `acknowledged_at`. If not yet acknowledged, elapsed time is measured from `detected_at` to now.

### **6.2 Resolution SLA**

Per SLA-001 §8.2-§8.4: Incidents MUST be resolved within the target window for their severity level.

| Severity | Resolution Target |
|----------|-------------------|
| Critical | 8 hours |
| High | 24 hours |
| Medium | 72 hours |
| Low | 168 hours (7 days) |

**Measurement:** `Incident.resolution_elapsed_hours` property computes hours between `detected_at` and `resolved_at`.

### **6.3 SLA Breach Detection**

<!-- assert: Incident model has SLA breach detection properties | check=inc-sla-breach -->
<!-- impl: syn/audit/models.py:Incident -->
<!-- test: syn.audit.tests.test_incidents.IncidentModelTest.test_sla_properties -->

`Incident.is_ack_sla_breached` and `Incident.is_resolution_sla_breached` return `True` when the elapsed time exceeds the target for the incident's severity. These properties drive the compliance check and dashboard indicators.

## **7. COMMUNICATION AND ESCALATION**

### **7.1 Notification Triggers**

| Event | Notification Type | Recipients |
|-------|-------------------|------------|
| Incident created | `incident_created` | All staff |
| SLA breach imminent (75% of target elapsed) | `incident_escalated` | All staff |
| Incident resolved | `incident_resolved` | All staff |

### **7.2 Escalation Policy**

When an incident's SLA target is breached:
1. `incident_escalated` notification sent to all staff
2. IncidentLog entry created with action `escalated`
3. Dashboard shows breach indicator (red badge)

## **8. POST-MORTEM**

### **8.1 Post-Mortem Requirements**

| Severity | Post-Mortem Required | Content |
|----------|---------------------|---------|
| Critical | **Mandatory** | Root cause, timeline, remediation actions, prevention measures |
| High | **Mandatory** | Root cause, remediation actions |
| Medium | Recommended | Brief summary of cause and fix |
| Low | Optional | — |

### **8.2 Post-Mortem Fields**

- `root_cause`: What caused the incident
- `resolution_summary`: How it was resolved
- `post_mortem_notes`: Detailed analysis, lessons learned, prevention measures

### **8.3 Incident-Driven Changes**

When an incident requires code changes:
1. Create `ChangeRequest` with `change_type=hotfix` or `bugfix` per CHG-001
2. Link the CR to the Incident via `Incident.change_request`
3. For critical/high: use CHG-001 §9 emergency change process
4. Post-incident review creates a `plan` type CR within 48h per CHG-001 §9.1

## **9. METRICS AND REPORTING**

### **9.1 Key Metrics**

| Metric | Calculation | Purpose |
|--------|-------------|---------|
| **MTTR** | Mean of `resolution_elapsed_hours` for resolved incidents | Service reliability indicator |
| **SLA Breach Rate** | Breached incidents / total incidents × 100 | Compliance indicator |
| **Incident Frequency** | Count per month by severity | Trend indicator |
| **Severity Distribution** | % by severity level | Risk profile indicator |

### **9.2 Dashboard Integration**

The internal dashboard (`/internal/dashboard/` → Operations → Incidents) displays:
- KPI bar: Active, Resolved (30d), Critical Active, MTTR, SLA Breaches
- Split-pane: filterable incident list (left) + detail view (right)
- Timeline: IncidentLog entries with actor and message
- SLA indicators: Elapsed time vs target with breach warnings

## **11. ANTI-PATTERNS**

### **11.1 Silent Resolution**

<!-- rule: mandatory -->
<!-- assert: Incident resolution requires resolution_summary in IncidentLog — silent resolution is prohibited | check=inc-no-silent-resolution -->

**PROHIBITED:** Resolving an incident without creating an IncidentLog entry documenting the resolution.

Every resolution MUST include a `resolution_summary` explaining what was done.

### **11.2 Severity Downgrade Without Justification**

<!-- rule: mandatory -->

**PROHIBITED:** Reducing an incident's severity without an IncidentLog entry documenting the justification.

Severity changes MUST create an IncidentLog entry with action `severity_changed` including the rationale.

### **11.3 Skipping Post-Mortem for Critical/High**

<!-- rule: mandatory -->
<!-- assert: Closing critical/high incidents requires post_mortem_notes — transition blocked without them | check=inc-postmortem-required -->

**PROHIBITED:** Closing a critical or high severity incident without post-mortem notes.

The transition to `closed` from `post_mortem` MUST require `post_mortem_notes` for critical/high severity incidents.

### **11.4 Unlinked Remediation**

<!-- rule: mandatory -->

**PROHIBITED:** Making code changes to fix an incident without linking the ChangeRequest to the Incident.

If a CR remediates an incident, `Incident.change_request` MUST be set before the incident is closed.
