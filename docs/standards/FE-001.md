## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

FE-001 defines the frontend architecture, template hierarchy, theme system, JavaScript conventions, and UI patterns for the Svend web application. There is no SPA framework — this is vanilla JavaScript with Django templates.

**Core Principle:**

> No build step. No transpiler. No node_modules. Templates render HTML, inline JS handles interaction, CSS variables handle theming. If it needs a framework, the feature is too complex for the frontend.

### **1.2 Scope**

**Applies to:**
- All templates in `services/svend/web/templates/`
- Static files in `services/svend/web/static/`
- Inline JavaScript and CSS in templates
- CDN library usage
- Theme system implementation

**Does NOT apply to:**
- Backend Python code
- API response formatting (see API-001)
- Static marketing pages (`services/svend/site/`)

| Term | Definition |
|------|------------|
| **base_app.html** | Master template (1,900+ lines). All authenticated pages extend this. |
| **Theme** | CSS variable set applied via `data-theme` attribute on `<html>`. 6 themes available. |
| **SvendTheme** | Global JS object exposing theme-aware colors for Chart.js integration. |
| **SvendChart** | Global JS wrapper for Chart.js with theme-aware defaults. |
| **CSRF monkey patch** | `window.fetch` override that auto-injects X-CSRFToken on non-GET requests. |

### **5.4 Theme Switching**


<!-- assert: Theme persists via localStorage and user profile API | check=fe-theme-persist -->

### **5.6 Canvas Background Consistency**

<!-- assert: All canvas-based views use var(--bg-primary) for canvas background, not hardcoded colors or feature-specific variables | check=fe-canvas-bg -->
<!-- impl: templates/whiteboard.html -->
<!-- impl: templates/simulator.html -->
<!-- impl: templates/vsm.html -->
<!-- test: syn.audit.tests.test_frontend.CanvasConsistencyTest.test_canvas_backgrounds_use_theme_var -->
<!-- test: syn.audit.tests.test_frontend.CanvasConsistencyTest.test_canvas_no_hardcoded_bg_colors -->

All canvas-based views (whiteboard, plant simulator, VSM) MUST use `var(--bg-primary)` for the canvas background color, either directly or via a local alias that maps to `var(--bg-primary)`. Feature-specific hardcoded background colors (e.g., `--wb-bg: #1a1a1a`) are prohibited — they break theme switching and visual consistency.

Grid patterns SHOULD use theme-aware colors (`var(--border)`, `var(--bg-hover)`, or `rgba()` with low opacity).

### **6.6 DSW Column Model Invariant**

<!-- assert: DSW frontend column objects use c.dtype property (not c.type) with values 'numeric', 'text', 'datetime' — no aliases | check=fe-column-dtype -->
<!-- impl: templates/analysis_workbench.html -->

Column metadata objects in the DSW workbench (`analysis_workbench.html`) MUST use `c.dtype` as the property name, with exactly three allowed values:
- `'numeric'` — continuous or integer measurement data
- `'text'` — categorical, nominal, or string data
- `'datetime'` — temporal data

**Prohibited patterns:**
- `c.type` (ambiguous — conflicts with HTML element `.type` property)
- `'number'` as a dtype value (use `'numeric'`)
- Bare `c` in template literals where `c` is a column object (must be `c.name`)

### **6.7 Rendering Safety**

<!-- assert: Frontend uses safeStr() for error envelope objects — never render raw API error objects | check=fe-safe-render -->
<!-- impl: templates/base_app.html:safeStr -->

API responses may contain `ErrorEnvelopeMiddleware` objects (`{code, message, retryable, request_id}`). Frontend code that displays API results MUST use `safeStr(val, fallback)` to safely coerce values to display strings, preventing `[object Object]` rendering.

## **9. BRAND IDENTITY**

### **9.1 Favicon**

<!-- assert: All templates use the Svend logo SVG as favicon, not emoji | check=fe-favicon -->
<!-- impl: templates/base_app.html -->
<!-- impl: templates/base_guest.html -->
<!-- impl: templates/landing.html -->

All pages — landing, app, guest, tool — MUST use the Svend logo as the browser tab icon:

```html
<link rel="icon" type="image/svg+xml" href="/static/favicon.svg">
```

The favicon is a black SVG silhouette (`static/favicon.svg`). Landing pages MAY additionally include PNG and Apple Touch Icon fallbacks for broader compatibility:

```html
<link rel="icon" type="image/png" sizes="32x32" href="/static/favicon-32x32.png">
<link rel="apple-touch-icon" sizes="180x180" href="/static/apple-touch-icon.png">
```

**PROHIBITED:** Emoji favicons via data URIs. The logo is the brand — it must be consistent across every surface.

### **9.2 Emoji Prohibition**

<!-- assert: No emoji or Unicode pictographs in templates — use inline SVG icons | check=fe-no-emoji -->
<!-- impl: templates/ -->
<!-- test: syn.audit.tests.test_frontend.EmojiComplianceTest.test_no_emoji_in_dashboard -->
<!-- test: syn.audit.tests.test_frontend.EmojiComplianceTest.test_no_emoji_in_app_templates -->

Emoji and Unicode pictographs (e.g. ⚠, ✓, ✗, 🔒, 📊) are **PROHIBITED** in all templates and inline JavaScript. Use Svend-style inline SVG icons instead.

**Svend icon pattern:**
```html
<svg viewBox="0 0 24 24" width="12" height="12" fill="none" stroke="currentColor"
     stroke-width="2" stroke-linecap="round" stroke-linejoin="round">
    <polyline points="20 6 9 17 4 12"/>
</svg>
```

**Why:** Emoji render inconsistently across platforms (font variation, size, baseline alignment). They dilute the Svend visual language and reduce perceived professionalism in a B2B/scientific context. Inline SVGs inherit `currentColor`, scale uniformly, and respect theme CSS variables.

**Prohibited:**
- Emoji or Unicode pictographs as status indicators, icons, or decorative elements
- Unicode symbols as icon replacements (use `_svIcons` helper or inline SVG)

**Allowed:**
- Standard punctuation and typographic characters in prose (em dash, bullet, etc.)

### **9.3 Logo Usage**

The Svend logo appears in three contexts:

| Context | File | Format |
|---------|------|--------|
| Browser tab (favicon) | `static/favicon.svg` | Black SVG silhouette |
| Landing page header | `static/favicon.svg` (inline) | Same SVG, sized per layout |
| Navigation | Text wordmark "SVEND" | Inter font, uppercase |

## **11. ANTI-PATTERNS**

### **11.1 Adding a Build Step**

**PROHIBITED:** Introducing webpack, vite, esbuild, or any transpilation. The frontend must remain zero-build.

### **11.2 Adding an SPA Framework**

**PROHIBITED:** Adding React, Vue, Angular, or Svelte. Django templates are the rendering layer.

### **11.3 Inline Styles Over CSS Variables**

<!-- code: prohibited -->
```html
<div style="background: #0a0f0a; color: #ffffff;">
```

<!-- code: correct -->
```html
<div style="background: var(--bg-primary); color: var(--text-primary);">
```

All colors MUST use CSS variables for theme compatibility.

### **11.4 Unapproved CDN Sources**

**PROHIBITED:** Loading scripts from CDN origins not listed in §7.1. New libraries require CSP update and review.

### **11.5 Direct DOM Manipulation Libraries**

<!-- assert: No jQuery or direct DOM manipulation libraries — only native DOM APIs | check=fe-no-jquery -->
