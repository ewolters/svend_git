## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

SYS-001 defines how Svend behaves **from the user's perspective**. Every critical user path — page loads, form submissions, email links, file downloads, webhook callbacks — is documented with its expected response type, status code, authentication requirement, and error behavior. This is the contract between the system and its users.

**Core Principle:**

> The user doesn't care about your middleware stack. They care that clicking a link works, downloading a PDF works, and errors tell them what to do next.
> If a path isn't documented here, it doesn't have a contract. If it doesn't have a contract, it can break silently.

### **1.2 Scope**

**Applies to:**
- All browser-navigable pages under `/`, `/app/`, `/tools/`, content pages
- All JSON API endpoints under `/api/`
- All file download/export endpoints (PDF, CSV, DOCX, SVG, PNG)
- All email link targets (tracking, unsubscribe)
- All webhook endpoints
- All beacon/analytics endpoints
- The middleware stack that governs all of the above

**Does NOT apply to:**
- Django admin (`/admin/`)
- Static files (`/static/`, `/media/`)
- Deployment scripts and ops tooling
- Internal scheduler tasks (syn.sched)

### **1.3 Implementation**

| Component | Location | Purpose |
|-----------|----------|---------|
| Root URL config | `svend/urls.py` | Route dispatch |
| API URL configs | `api/urls.py`, `agents_api/urls.py` | Feature-based sub-routes |
| Middleware stack | `svend/settings.py:MIDDLEWARE` | Request/response pipeline |
| Auth decorators | `accounts/permissions.py` | Access control |
| Error envelope | `syn/api/middleware.py:ErrorEnvelopeMiddleware` | Standard error wrapping |
| Browser-facing bypass | `syn/api/middleware.py:_is_browser_facing_api()` | Accept header bypass |
| Templates | `templates/` | HTML page rendering |

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **User path** | A specific URL + method + trigger combination that a human or browser initiates |
| **Trigger type** | How the request originates: `BROWSER_NAV`, `JS_FETCH`, `EMAIL_LINK`, `WEBHOOK`, `BEACON`, `FILE_DOWNLOAD` |
| **Response contract** | The expected Content-Type, status code range, and body format for a path |
| **Boundary** | The point where two components interact (browser↔server, email↔server, webhook↔server) |
| **Error envelope** | Standard JSON error format per ERR-001: `{"error": {"code", "message", "retryable", "request_id"}}` |
| **Browser-facing API** | An endpoint under `/api/` that returns non-JSON to browsers (redirects, files, pixels) |

## **4. TRIGGER TYPES**

Every user-facing path falls into exactly one trigger type. The trigger type determines what the middleware stack does with the request.

### **4.1 BROWSER_NAV — Browser Page Navigation**

User clicks a link or types a URL. Browser sends `Accept: text/html`. Server returns rendered HTML.

<!-- assert: Browser navigation pages return HTML with 200 status | check=sys-browser-nav -->
<!-- test: syn.audit.tests.test_sys_integration.BrowserNavTest.test_public_pages_return_html -->
<!-- test: syn.audit.tests.test_sys_integration.BrowserNavTest.test_app_pages_require_auth -->

**Contract:**
- Response: `Content-Type: text/html; charset=utf-8`
- Status: `200 OK` (app pages serve HTML shell even without auth — auth check is client-side JS)
- Error: Django error page (not JSON envelope)
- CSRF: Token embedded in template via `{% csrf_token %}`

### **4.2 JS_FETCH — JavaScript API Calls**

Frontend JS calls `fetch('/api/...')` with `Accept: application/json`. Server returns JSON.

<!-- assert: JS fetch endpoints return JSON with error envelope on failure | check=sys-js-fetch -->
<!-- test: syn.audit.tests.test_sys_integration.JSFetchTest.test_api_endpoints_return_json -->
<!-- test: syn.audit.tests.test_sys_integration.JSFetchTest.test_api_errors_use_envelope -->

**Contract:**
- Request: `Content-Type: application/json`, `Accept: application/json`
- Response: `Content-Type: application/json; charset=utf-8`
- Success: `200 OK` with feature-specific JSON body
- Auth failure: `401` with error envelope
- Validation: `400` with error envelope
- Server error: `500` with error envelope
- Headers: `Syn-Request-Id`, `Vary: Accept, Cookie`

### **4.3 EMAIL_LINK — Email Click/Open/Unsubscribe**

User clicks a link in an automated email. Browser sends `Accept: text/html`. These paths are under `/api/` but must NOT be rejected by Accept validation.

<!-- assert: Email link endpoints bypass Accept validation and return appropriate non-JSON responses | check=sys-email-link -->
<!-- test: syn.audit.tests.test_sys_integration.EmailLinkTest.test_open_tracking_returns_pixel -->
<!-- test: syn.audit.tests.test_sys_integration.EmailLinkTest.test_click_tracking_redirects -->
<!-- test: syn.audit.tests.test_sys_integration.EmailLinkTest.test_unsubscribe_returns_html -->

**Contract:**
- `/api/email/open/<uuid>/` → `200` with 1×1 transparent GIF (`image/gif`)
- `/api/email/click/<uuid>/` → `302` redirect to target URL
- `/api/email/unsubscribe/` → `200` with styled HTML page (GET shows form, POST processes)
- No `Accept: application/json` required
- No authentication required
- Bypass: `_is_browser_facing_api()` in `syn/api/middleware.py`

### **4.4 FILE_DOWNLOAD — Export and Download Endpoints**

User clicks "Export PDF" or "Download CSV". Browser sends `Accept: text/html` or `*/*`. These paths are under `/api/` but return binary content.

<!-- assert: File download endpoints bypass Accept validation and return correct Content-Type | check=sys-file-download -->
<!-- test: syn.audit.tests.test_sys_integration.FileDownloadTest.test_pdf_export_returns_pdf -->
<!-- test: syn.audit.tests.test_sys_integration.FileDownloadTest.test_csv_download_returns_csv -->
<!-- test: syn.audit.tests.test_sys_integration.FileDownloadTest.test_model_download_returns_binary -->

**Contract:**
- PDF exports → `Content-Type: application/pdf`, `Content-Disposition: attachment`
- CSV/Excel downloads → `Content-Type: text/csv` or `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`
- DOCX exports → `Content-Type: application/vnd.openxmlformats-officedocument.wordprocessingml.document`
- SVG/PNG exports → `Content-Type: image/svg+xml` or `image/png`
- Model downloads → `Content-Type: application/octet-stream`
- Authentication required (except whitepaper PDFs)
- Bypass: `_is_browser_facing_api()` suffix/segment matching

### **4.5 WEBHOOK — External Service Callbacks**

External services (Stripe) POST to webhook endpoints. Not under `/api/`, so middleware Accept validation does not apply.

<!-- assert: Webhook endpoints verify signatures and are not under /api/ path | check=sys-webhook -->
<!-- test: syn.audit.tests.test_sys_integration.WebhookTest.test_stripe_webhook_path_not_under_api -->

**Contract:**
- Paths: `/webhooks/stripe/`, `/billing/webhook/` (and trailing-slash variants)
- Method: POST only
- Auth: Stripe signature verification (not session/token auth)
- Response: `200 OK` with minimal JSON on success
- No Accept header validation (not under `/api/`)

### **4.6 BEACON — Analytics and Tracking**

Frontend sends `navigator.sendBeacon()` or lightweight POST for engagement tracking. Low-priority, fire-and-forget.

<!-- assert: Beacon endpoints accept POST without strict content validation | check=sys-beacon -->
<!-- test: syn.audit.tests.test_sys_integration.BeaconTest.test_site_duration_accepts_beacon -->

**Contract:**
- `/api/site-duration/` → `204 No Content` or `200 OK`
- `/api/funnel-event/` → `200 OK`
- `/api/events/` → `200 OK`
- No authentication required
- Accept `application/json` or `text/plain` (sendBeacon sends text/plain by default)

## **6. RESPONSE CONTRACTS**

### **6.1 HTML Pages (BROWSER_NAV)**

<!-- assert: All browser-navigable pages extend base template and include CSRF | check=sys-html-contract -->
<!-- test: syn.audit.tests.test_sys_integration.ResponseContractTest.test_html_pages_have_csrf -->

All pages under `/app/` MUST:
- Extend `base_app.html`
- Include `{% csrf_token %}` in any form
- Return `200 OK` with `Content-Type: text/html; charset=utf-8`
- Redirect to `/login/` if authentication is required and user is not authenticated

Public pages (under `/`, `/tools/`, content pages) MUST:
- Return `200 OK`
- Not require authentication

### **6.2 JSON API Responses (JS_FETCH)**

<!-- assert: All JSON API responses follow consistent envelope format | check=sys-json-contract -->
<!-- test: syn.audit.tests.test_sys_integration.ResponseContractTest.test_json_errors_use_envelope -->
<!-- test: syn.audit.tests.test_sys_integration.ResponseContractTest.test_json_success_consistent -->

**Success responses** MUST:
- Return `Content-Type: application/json; charset=utf-8`
- Return `200 OK` for retrieval/update, `201 Created` for creation, `204 No Content` for deletion
- Include `Syn-Request-Id` header

**Error responses** MUST use the error envelope (ERR-001 §5):
```json
{
  "error": {
    "code": "snake_case_error_code",
    "message": "Human-readable message",
    "retryable": false,
    "request_id": "req_..."
  }
}
```

**Known gaps** (to be closed — tracked as debt):
- `workbench/views.py` uses `@login_required` + `JsonResponse({"error": "msg"})` instead of DRF + envelope
- `notifications/views.py` returns bare `{"error": "msg"}` in some paths
- ~62 endpoints across `core/`, `notifications/`, `forge/` return non-envelope errors

### **6.3 File Downloads (FILE_DOWNLOAD)**

<!-- assert: File download responses include Content-Disposition header | check=sys-download-contract -->
<!-- test: syn.audit.tests.test_sys_integration.ResponseContractTest.test_downloads_have_disposition -->

All file download responses MUST:
- Include `Content-Disposition: attachment; filename="..."` or `inline; filename="..."`
- Set correct `Content-Type` for the file format
- Return `200 OK` on success
- Return error envelope JSON (not bare error) on failure

### **6.4 Error Behavior at Boundaries**

<!-- assert: Errors at system boundaries return appropriate format for the trigger type | check=sys-error-boundary -->
<!-- test: syn.audit.tests.test_sys_integration.ErrorBoundaryTest.test_html_page_error_is_html -->
<!-- test: syn.audit.tests.test_sys_integration.ErrorBoundaryTest.test_api_error_is_json_envelope -->
<!-- test: syn.audit.tests.test_sys_integration.ErrorBoundaryTest.test_email_link_error_is_not_json -->

Errors MUST match the trigger type's expected response format:
- BROWSER_NAV errors → HTML error page (not JSON)
- JS_FETCH errors → JSON error envelope (not HTML)
- EMAIL_LINK errors → HTML page or redirect (not JSON)
- FILE_DOWNLOAD errors → JSON error envelope (caller is JS that initiated the download)
- WEBHOOK errors → JSON (caller is external service)

### **7.1 Auth Patterns**

<!-- assert: Authentication uses consistent decorator patterns per module type | check=sys-auth-patterns -->
<!-- test: syn.audit.tests.test_sys_integration.AuthContractTest.test_api_endpoints_use_require_auth -->

|-------------|-------------|-----------|

**Known gaps** (to be closed):
- `workbench/views.py` uses `@login_required` on JSON API views (returns HTML redirect instead of 401 JSON)
- Three distinct auth patterns across modules — should converge to two: `@require_auth` for API, `@login_required` for pages

## **8. MIDDLEWARE BEHAVIOR**

### **8.1 Request Pipeline**

<!-- assert: Middleware processes requests in documented order | check=sys-middleware-order -->
<!-- test: syn.audit.tests.test_sys_integration.MiddlewareTest.test_cors_before_api_headers -->
<!-- test: syn.audit.tests.test_sys_integration.MiddlewareTest.test_request_id_on_all_api_responses -->

Middleware order (from `settings.py:MIDDLEWARE`):

| Order | Middleware | Effect |
|-------|-----------|--------|
| 1 | SecurityMiddleware | HTTPS redirect, HSTS |
| 2 | SynRequestIdMiddleware | Generate `Syn-Request-Id` (ULID), set `request.syn_request_id` |
| 3 | CorsMiddleware | CORS headers (before API surface — ensures error responses include CORS) |
| 4 | APIHeadersMiddleware | Accept validation on `/api/` (bypass for browser-facing), charset, Vary, timing |
| 5 | SessionMiddleware | Session handling |
| 6 | CommonMiddleware | Trailing slash, etc. |
| 7 | CsrfViewMiddleware | CSRF protection |
| 8 | AuthenticationMiddleware | User authentication |
| 9 | AuditLoggingMiddleware | Mutation audit trail |
| 10 | ErrorEnvelopeMiddleware | Wrap exceptions as error envelope JSON |
| 11 | IdempotencyMiddleware | POST replay, conflict detection |

**Critical ordering rules:**
- CORS MUST be before APIHeaders (so 406 responses include CORS headers)
- SynRequestId MUST be before APIHeaders (so error responses include request ID)
- ErrorEnvelope wraps exceptions ONLY for `/api/` paths

### **8.2 Accept Header Validation**

<!-- assert: Accept header validation correctly routes browser-facing vs API requests | check=sys-accept-validation -->
<!-- test: syn.api.tests.test_api_infrastructure.APIHeadersTest.test_missing_accept_header_rejected -->
<!-- test: syn.api.tests.test_api_infrastructure.APIHeadersTest.test_export_download_paths_bypass_accept_check -->
<!-- test: syn.api.tests.test_api_infrastructure.APIHeadersTest.test_accept_with_quality_params -->

For requests to `/api/` paths:
- If path matches `_is_browser_facing_api()` → skip Accept validation
- If `Accept` header is missing or does not include `application/json` media type → `406 Not Acceptable`
- Accept header parsed by media type splitting (not substring match): `{p.split(";")[0].strip() for p in accept.split(",")}`

Browser-facing API path bypass patterns:
- **Prefixes:** `/api/email/click/`, `/api/email/open/`, `/api/email/unsubscribe/`, `/api/notifications/unsubscribe/`
- **Suffixes:** `/export/pdf/`, `/export/docx/`, `/download/`
- **Segments:** `/forge/download/`, `/dsw/models/`

## **10. KNOWN INCONSISTENCIES**

This section documents verified inconsistencies found during the system audit. Each item is a candidate for remediation via CHG-001.

### **10.1 Error Response Format**

<!-- assert: All API error responses use the standard error envelope | check=sys-error-envelope-consistency -->

**Gap:** ~62 endpoints return bare `{"error": "message"}` instead of the standard envelope `{"error": {"code", "message", "retryable", "request_id"}}`.

| Module | Pattern | Count | Severity |
|--------|---------|-------|----------|
| `workbench/views.py` | `JsonResponse({"error": "msg"}, status=N)` | ~15 | HIGH — entire module |
| `notifications/views.py` | `JsonResponse({"error": "msg"})` | ~5 | MEDIUM |
| `forge/views.py` | `JsonResponse({"error": "msg"})` | ~8 | MEDIUM |
| `core/views.py` | Mixed DRF + bare JsonResponse | ~10 | MEDIUM |
| Various | `return JsonResponse({"error": ...}, status=400)` | ~24 | LOW — scattered |

**Impact:** Clients cannot reliably parse error responses. Frontend must handle two formats. `retryable` and `request_id` missing from non-envelope errors.

### **10.2 Authentication Decorator Inconsistency**

<!-- assert: Authentication decorators are consistent within module types | check=sys-auth-consistency -->

**Gap:** Three distinct auth patterns coexist.

| Pattern | Module | Effect on 401 |
|---------|--------|---------------|
| `@api_view + @permission_classes([IsAuthenticated])` | DRF views (most of agents_api) | 401 JSON (DRF format) |
| `@require_auth + @csrf_exempt` | Function-based API (api/views.py) | 401 JSON (custom) |
| `@login_required` | workbench/views.py | 302 redirect to `/login/` (HTML) |

**Impact:** `workbench/views.py` returns HTML redirect on auth failure for JSON API calls. Frontend JS gets a 302 redirect instead of a 401 JSON response.

### **10.3 Status Code Usage**

<!-- assert: HTTP status codes follow RFC 9110 conventions consistently | check=sys-status-codes -->

**Gap:** ~30 endpoints use raw integer status codes or inconsistent codes.

| Issue | Example | Expected |
|-------|---------|----------|
| `status=200` for creation | `POST /api/a3/create/` returns 200 | Should be 201 |
| `status=200` for deletion | Some DELETE views return 200 | Should be 204 |
| Raw integers | `JsonResponse({...}, status=403)` without error envelope | Should use `SynaraError` |

### **10.4 Notification Unsubscribe Response**

**Fixed in CR-0da6 (this session):** `/api/notifications/unsubscribe/` previously returned JSON to browser clicks. Now returns styled HTML matching email unsubscribe page.
