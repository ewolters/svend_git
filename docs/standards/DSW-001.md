## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

DSW-001 defines the architecture, integration patterns, and invariants for the Decision Science Workbench (DSW) subsystem. DSW is a stateless statistical analysis engine comprising 200+ analyses, dispatch routing, result persistence, and Bayesian hypothesis integration via the Synara belief engine.

> **Core Principle:** DSW is a calculator, not a database. Analysis requests are stateless: data in, result out. Persistent state lives in `core.Project`, `core.Hypothesis`, and `core.Evidence`. The only DSW-owned persistence is `DSWResult` for deferred import. Hypothesis and evidence operations MUST have a valid project context — creating hypotheses without a project violates the data model.

This standard covers architecture and integration only. Statistical methodology, formulas, and mathematical correctness are out of scope.

### **1.2 Scope**

**Applies to:**
- DSW monolith (`agents_api/dsw_views.py`)
- DSW dispatch module (`agents_api/dsw/dispatch.py`)
- DSW sub-modules (`agents_api/dsw/*.py`)
- Synara belief engine views (`agents_api/synara_views.py`)
- Synara engine classes (`agents_api/synara/`)
- SPC engine (`agents_api/spc.py`, `agents_api/spc_views.py`)
- DOE engine (`agents_api/experimenter_views.py`)
- DSW result model (`agents_api/models.py:DSWResult`)
- Evidence bridge (`agents_api/evidence_bridge.py:create_tool_evidence`)
- Frontend workbench (`templates/workbench_new.html`)

**Does NOT apply to:**
- Statistical formulas and methodology (future STAT-001)
- QMS tools: FMEA, RCA, A3, VSM, Hoshin Kanri (QMS-001)
- Whiteboard collaboration (integration point only)
- Learn module sandbox sessions (consumes DSW, is not DSW)
- Knowledge graph (DAT-001)

DSW analysis requests follow a stateless dispatch pattern. Each request carries its own data (inline, uploaded file, or triage result reference) and configuration. The dispatcher routes by `analysis_type` string to the appropriate sub-module handler. Handlers receive a pandas DataFrame and config dict, return a result dict with plots, summary, and `guide_observation` fields. No server-side session state is maintained between analysis requests.

Dispatch routing:

| `analysis_type` | Module | Handler |
|-----------------|--------|---------|
| `stats` | `dsw/stats.py` | `run_statistical_analysis()` |
| `ml` | `dsw/ml.py` | `run_ml_analysis()` |
| `spc` | `dsw/spc.py` | `run_spc_analysis()` |
| `bayesian` | `dsw/bayesian.py` | `run_bayesian_analysis()` |
| `reliability` | `dsw/reliability.py` | `run_reliability_analysis()` |
| `simulation` | `dsw/simulation.py` | `run_simulation()` |
| `viz` | `dsw/viz.py` | `run_visualization()` |
| `d_type` | `dsw/d_type.py` | `run_d_type()` |

Synara uses a bounded dict cache (`_synara_cache`, max 128 entries) keyed by `workbench_id`. On cache miss, state loads from `core.Project.synara_state` JSON field. On any mutation, `save_synara()` persists state back to the Project. When the cache is full, the oldest entry is evicted.

### **5.4 Save Failure Semantics**

<!-- rule: mandatory -->

<!-- assert: save_synara() failure MUST propagate to the caller as an error response, never silently succeed | check=dsw-save-failure -->
<!-- impl: agents_api/synara_views.py:save_synara -->
<!-- test: syn.audit.tests.test_dsw.SaveFailureFunctionalTest.test_save_returns_false_without_project -->

`save_synara()` returns `False` when `_resolve_project()` returns `None`. All callers MUST check this return value and return an HTTP error to the client. A Synara endpoint that returns `{"success": true}` when `save_synara()` returned `False` is a data integrity violation — the user believes their data was saved, but it exists only in a volatile cache that will be evicted.

## **9. ANTI-PATTERNS**

### **9.1 Silent Save Failure**

<!-- rule: mandatory -->

<!-- assert: No Synara endpoint returns success when save_synara() fails — all callers check return value | check=dsw-no-silent-save -->
<!-- impl: agents_api/synara_views.py -->
<!-- test: syn.audit.tests.test_dsw.SaveFailureFunctionalTest.test_save_returns_false_without_project -->
<!-- test: syn.audit.tests.test_dsw.SaveFailureFunctionalTest.test_require_project_returns_400 -->

**PROHIBITED:** Calling `save_synara()` without checking its return value. When `save_synara()` returns `False`, the endpoint has failed to persist the user's data. Returning success is a data integrity violation.

<!-- code: prohibited -->

```python
save_synara(workbench_id, synara, user=request.user)
return JsonResponse({"success": True, "hypothesis": h.to_dict()})
```

<!-- code: correct -->

```python
if not save_synara(workbench_id, synara, user=request.user):
    return JsonResponse({"error": "Failed to persist state."}, status=500)
return JsonResponse({"success": True, "hypothesis": h.to_dict()})
```

### **9.2 Hypothesis Without Project Validation**

<!-- rule: mandatory -->

**PROHIBITED:** Creating hypotheses, evidence, or links without first validating that a project resolves for the given `workbench_id`. The `_require_project()` helper MUST be called before any Synara mutation.

<!-- code: prohibited -->

```python
synara = get_synara(workbench_id, user=request.user)
h = synara.create_hypothesis(description=desc, prior=0.5, source="user")
save_synara(workbench_id, synara, user=request.user)
return JsonResponse({"success": True})
```

<!-- code: correct -->

```python
project, err = _require_project(workbench_id, user=request.user)
if err:
    return err
synara = get_synara(workbench_id, user=request.user)
h = synara.create_hypothesis(description=desc, prior=0.5, source="user")
if not save_synara(workbench_id, synara, user=request.user):
    return JsonResponse({"error": "Failed to persist hypothesis."}, status=500)
return JsonResponse({"success": True, "hypothesis": h.to_dict()})
```

### **9.3 Frontend Actions Without Project Check**

<!-- rule: mandatory -->

<!-- assert: Frontend MUST NOT enable mutation UI without valid currentProjectId | check=dsw-no-frontend-mutation-without-project -->
<!-- impl: templates/workbench_new.html -->
<!-- test: syn.audit.tests.test_dsw.FrontendRenderingTest.test_hypothesis_button_requires_project_id -->

**PROHIBITED:** Enabling hypothesis or evidence creation UI elements when no project is loaded. The frontend MUST check `currentProjectId` before allowing mutation actions.

<!-- code: prohibited -->

```javascript
if (currentWorkbench && currentWorkbench.id) {
    expHtml += '<button class="dsw-create-hyp">Create Hypothesis</button>';
}
```

<!-- code: correct -->

```javascript
if (currentWorkbench && currentWorkbench.id && currentProjectId) {
    expHtml += '<button class="dsw-create-hyp">Create Hypothesis</button>';
}
```

### **9.4 Unlinked Analysis Results**

<!-- rule: recommended -->

**DISCOURAGED:** Running analysis with a project loaded but not passing `project_id` in the request body. When a project is active in the frontend, the `project_id` should be included so results can be linked as evidence. The frontend includes this automatically via `currentProjectId`.
