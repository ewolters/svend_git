## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

CHG-001 defines the change management architecture for Kjerne/Svend. Every codebase change — feature additions, updates, bug fixes, hotfixes, infrastructure changes, and planning activities — follows a defined process, receives a risk classification, and generates an immutable log chain linking the change request to its authorization, risk assessment, implementation, and verification.

**Core Principle:**

> Every change has a type, a risk level, an authorization chain, and a log.
> Plans are logged. Decisions are logged. Implementation is logged. Verification is logged.
> The audit trail for a change is a linked chain — from the source issue through every checkpoint to closure.

### **1.2 Scope**

**Applies to:**
- All code changes to the Kjerne repository
- Infrastructure and configuration changes
- Database migrations
- Feature additions, updates, bug fixes, and hotfixes
- Planning and architectural decisions
- Emergency changes with retroactive documentation
- CI/CD pipeline modifications

**Does NOT apply to:**
- Documentation-only changes (typo fixes, README updates) — exempt from risk assessment but still logged
- External dependency updates handled by automated tooling (e.g., Dependabot) — tracked via dependency_vuln compliance check
- SOC 2 Type II audit procedures (external auditor process)

| Term | Definition |

| State | Description | Who Transitions |
|-------|-------------|-----------------|
| `draft` | Change being planned — plans are logged here | Author |
| `submitted` | Formally submitted for review | Author |
| `risk_assessed` | Risk assessment completed (mandatory gate for `feature`/`migration` — see §6.4.1) | System (after agent votes) |
| `approved` | Authorized for implementation (requires `RiskAssessment` for `feature`/`migration`) | Approver (staff or system for standard changes) |
| `rejected` | Change denied — reason logged | Approver |
| `in_progress` | Implementation underway | Author |
| `testing` | Verification in progress | Author |
| `completed` | Successfully implemented and verified | Author |
| `failed` | Implementation failed — rollback may be needed | Author or system |
| `rolled_back` | Change reverted | Author |
| `cancelled` | Change abandoned before implementation | Author |

### **5.3 Standards Review**

<!-- rule: mandatory -->

**Every change MUST begin with a standards review.** Before writing code, modifying configuration, or creating implementation plans, the author identifies which standards from the Kjerne Standards Library (25 standards) apply to the change and reviews the relevant sections.

**Purpose:** Standards encode hard-won decisions about architecture, security, compliance, and quality. A change that violates a standard creates drift, breaks compliance checks, and generates rework. Reviewing standards first prevents this — the standard tells you *how* to make the change before you make it.

**Process:**

1. **Identify applicable standards.** Every change touches at least one standard. Use this guide:

   | Change Involves | Review These Standards |
   |-----------------|----------------------|
   | Any code change | CHG-001 (this document), STY-001 (code style) |
   | API endpoints | API-001 (URL design, error envelope), SEC-001 (auth, CSP) |
   | Database models or migrations | DAT-001 (field patterns, UUIDs), ARCH-001 (layer boundaries) |
   | Frontend / templates | FE-001 (themes, CSRF, CDN), CACHE-001 (browser storage, SRI) |
   | Error handling or logging | ERR-001 (error hierarchy), LOG-001 (structured logging) |
   | Statistical analysis | STAT-001 (methodology), DSW-001 (pipeline), QUAL-001 (output quality) |
   | Quality tools (FMEA, RCA, etc.) | QMS-001 (tool architecture), QUAL-001 (output validation) |
   | LLM / AI features | LLM-001 (rate limits, encryption, prompt patterns) |
   | Billing or subscriptions | BILL-001 (tiers, Stripe, feature gating) |
   | Deployment or infrastructure | OPS-001 (systemd, backup, TLS), SCH-001 (scheduler) |
   | Tests | TST-001 (testing patterns, fixtures) |
   | Compliance or audit | CMP-001 (automation), AUD-001 (audit trail) |
   | Documentation or standards | DOC-001 (structure, hooks), XRF-001 (cross-references) |

2. **Review the relevant sections.** Read the sections that apply to your change — not the entire standard, but the parts that govern what you're doing. Pay attention to:
   - Anti-patterns (§ "Anti-Patterns" in each standard)
   - Assertion hooks (`<!-- assert: -->`) that will be verified by compliance checks
   - Required patterns and prohibited practices

3. **Record in the ChangeRequest.** The `implementation_plan` field (§7.1) MUST note which standards were reviewed. This is not bureaucracy — it creates the record that the change was informed by the standards, which is what SOC 2 CC8.1 requires.

**When standards conflict or are silent:** If the change involves something not covered by existing standards, or if two standards give contradictory guidance, note this in the CR. This is how standards evolve — gaps discovered during real work are the best input for standard updates.

### **5.4 Process Enforcement**

<!-- rule: mandatory -->

**ALL codebase changes MUST follow the CHG-001 process.** This is enforced via:

1. **CLAUDE.md linkage**: Development instructions reference CHG-001 as the required change process
2. **log.md integration**: Every change creates an entry in `log.md` with a link to the ChangeRequest
3. **Compliance check**: The `change_management` compliance check (14 checks, daily) verifies change process adherence
4. **Commit linkage**: Every commit MUST be linked back to its CR via `commit_shas` (see §5.4.2)
5. **Transition gates**: `validate_for_transition()` blocks completion without `commit_shas` and `log_md_ref`

### **5.4.1 CR Creation Sequencing**

<!-- rule: mandatory -->
<!-- assert: ChangeRequest must be created before any code is committed for non-emergency changes | check=chg-cr-before-code -->
<!-- impl: syn/audit/compliance.py -->

**For non-emergency changes, a ChangeRequest MUST exist in `draft` or `submitted` state BEFORE any code is committed.** The CR is the authoritative record of *why* code is changing — not a post-hoc artifact. The lifecycle state machine (`draft` → `approved` → `in_progress`) encodes this: implementation begins at `in_progress`, which requires prior authorization.

**Database migrations carry elevated risk.** Schema changes are difficult or impossible to reverse in production. A migration applied without a CR:

- Has no rollback plan on record
- Has no risk assessment (multi-agent required for `migration` type per §4.1)
- Breaks the audit chain required by SOC 2 CC8.1
- Cannot be verified during compliance review

The **only** exception is the emergency hotfix path (§9), which permits immediate action with retroactive documentation SLAs (1h for logs, 24h for risk assessment, 48h for post-incident review). Emergency changes MUST use the `hotfix` type — retroactive CRs for normal change types are a compliance violation.

If a CR is created after code has already been committed, the log chain MUST include a `comment` action entry honestly documenting the sequencing gap. Falsifying timestamps or omitting the gap is prohibited.

### **5.4.2 Commit Linkage**

<!-- rule: mandatory -->

**Every git commit that implements a change MUST be linked back to its ChangeRequest via the `commit_shas` field.** This is the bridge between the CR audit trail and the git history. Without it, the chain is broken — you can find the CR but not the code, or the code but not the authorization.

**How to link commits:**

After creating a commit, append its SHA to the CR:

```python
cr.commit_shas.append(commit_sha)
cr.save()
```

Or via the transition API when completing:

```json
{
    "action": "completed",
    "details": {"commit_sha": "abc123def456"}
}
```

**This is enforced at completion.** `validate_for_transition('completed')` rejects any code-touching CR with an empty `commit_shas` list. The transition will return HTTP 400 with `"commit_shas required before completion"`.

**Commit message convention:** Include the CR short ID in the commit message for grep-ability:

```
[CR:5231a289] CHG-001 lockdown: mandatory field enforcement
```

This is not yet enforced by a git hook, but it enables future automation and makes `git log --grep` useful for traceability.

**Who is responsible:** The author completing the CR. For Claude-authored changes, Claude MUST populate `commit_shas` after committing and before transitioning the CR to `completed`. This is not optional — it is part of the completion checklist:

1. Commit code → capture SHA
2. Set `cr.commit_shas = [sha]`
3. Set `cr.log_md_ref` to the log.md section reference
4. Transition CR to `completed`

### **5.5 Plan Logging**

<!-- rule: mandatory -->

**Plans and architectural decisions MUST be logged as change requests with type `plan`.** Planning activities include:
- Architectural design decisions
- Technology selection
- Migration planning
- Risk assessment deliberations
- Scope changes during implementation

Plans enter the system as `draft` → `submitted` → `completed` (no approval gate, but fully logged).

### **5.6 Planning Pipeline**

<!-- rule: mandatory -->

**PlanDocuments provide the deliberation layer that precedes ChangeRequests.** The full pipeline:

```
PlanDocument (spec/plan/rfc/retro)
    │  Draft → Active → Archived
    │
    ├── Links to one or more ChangeRequests via change_request_ids
    │
    ▼
ChangeRequest (feature/enhancement/bugfix/...)
    │  draft → submitted → risk_assessed → approved → in_progress → testing → completed
    │
    ├── Risk assessment (multi-agent voting for features/migrations)
    ├── ChangeLog entries at each transition
    │
    ▼
Implementation
    │  Code changes, migrations, tests
    │
    ▼
Verification
    │  Compliance checks, test execution
    │
    ▼
Deployment
       Gunicorn reload, collectstatic
```

**Model**: `PlanDocument` (`api.models`) — fields: `id` (UUID), `title`, `body` (markdown), `status` (draft/active/archived), `category` (spec/plan/rfc/retro), `change_request_ids` (JSONField of CR UUIDs).

**Dashboard**: Internal Dashboard → Product → Planning tab. Two-pane editor with live markdown preview. Linked CRs shown as status-colored chips.

**API**:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/internal/plans/` | GET | List plans (filter: status, category) |
| `/api/internal/plans/save/` | POST | Create or update plan |
| `/api/internal/plans/<id>/` | GET | Single plan with full body |
| `/api/internal/plans/<id>/delete/` | POST | Delete plan |

**Rationale**: Plans are written over days or weeks — scope definition, architectural decisions, risk considerations. When a plan is ready for execution, one or more CRs are created and linked back. The plan provides the "why" and the CR provides the "how" and the audit chain. This replaces ad-hoc planning with a deliberate spec → execute → verify loop.

### **5.6.1 Planning Linkage**

<!-- rule: recommended -->

Every ChangeRequest SHOULD link to the Feature or PlanTask that motivated it via `feature_id` and/or `task_id`. This creates bidirectional traceability:

- **Planning → CR:** `Feature.change_request_ids` / `PlanTask.change_request_id` (existing)
- **CR → Planning:** `ChangeRequest.feature_id` / `ChangeRequest.task_id`

The `link_planning()` method on ChangeRequest writes both directions automatically — setting the CR's fields and writing back to the planning model's CR reference field.

The `change_management` compliance check (check #15) flags active code-type CRs without planning linkage as WARNING. Exempt: `documentation`, `plan`, `hotfix` types.

**Full traceability chain:** `Initiative → Feature → PlanTask → ChangeRequest → ChangeLog → commit_shas → git history`

For `feature`, `migration`, and `critical` risk changes, risk assessment uses multi-agent voting where each agent evaluates from a different role perspective:

| Agent Role | Perspective | Focus Areas |
|------------|-------------|-------------|
| `security_analyst` | Security & compliance | Auth changes, data exposure, SOC 2 impact |
| `architect` | System design & integrity | Architecture drift, coupling, scalability |
| `operations` | Availability & deployment | Downtime risk, rollback plan, monitoring |
| `quality` | Testing & correctness | Test coverage, regression risk, edge cases |

Each agent provides:
- **Risk score** (1-5 per dimension)
- **Recommendation** (approve / approve_with_conditions / reject)
- **Rationale** (structured justification)
- **Conditions** (required mitigations if approve_with_conditions)

### **6.3 Risk Score Aggregation**

The overall risk level is computed from agent votes:

1. **Per-dimension**: Average scores across agents → dimension risk (1-5)
2. **Overall**: Maximum dimension score → overall risk level
3. **Recommendation**: Majority vote with security_analyst having veto power on security-related changes

| Score Range | Risk Level | Required Action |
|-------------|-----------|-----------------|
| 1.0 - 2.0 | `low` | Auto-approve eligible |
| 2.1 - 3.0 | `medium` | Single approver required |
| 3.1 - 4.0 | `high` | Staff review required |
| 4.1 - 5.0 | `critical` | Staff review + rollback plan mandatory |

### **6.4 Assessment Timing**

#### **6.4.1 Standard Path (feature, migration, critical risk)**

A `RiskAssessment` with multi-agent votes MUST be created **before** the CR transitions to `approved`. The compliance check (`change_management`) enforces this: any `feature` or `migration` CR in `approved`, `in_progress`, `testing`, or `completed` status without an associated `RiskAssessment` record is flagged as a compliance failure.

```
submitted → [RiskAssessment created + AgentVotes] → risk_assessed → approved
```

This is not a suggestion — it is a hard gate. The assessment is what justifies the approval.

#### **6.4.2 Expedited Path (hotfix, security)**

For `hotfix` and `security` type changes:
- Single agent assessment (security_analyst for security, operations for hotfix)
- Assessment and approval can happen simultaneously
- Full multi-agent review MUST happen retroactively within 24 hours

#### **6.4.3 Retroactive Assessment**

If a `feature` or `migration` CR reaches `completed` without a `RiskAssessment` (process gap), a retroactive assessment MUST be created to clear the compliance finding:

- Set `is_retroactive=True` on the `RiskAssessment`
- Set `assessment_type='expedited'`
- Create all 4 `AgentVote` records with honest rationales noting the retroactive nature
- Add a `risk_assessed` `ChangeLog` entry documenting the gap

Retroactive assessments clear the compliance check but the gap is permanently visible in the audit trail. This is by design — the honest record matters more than a clean timeline.

|-------|-----------|-----------------|
| **Model** | `ChangeRequest.clean()` | Empty title, description, or author at creation |
| **API** | `ChangeRequest.validate_for_transition(target_state)` | Missing fields for the target lifecycle state |
| **Compliance** | `check_change_management()` — 14 checks, daily | All gaps on existing records, field completeness reporting |

There is no way to create a CR with an empty title or description. There is no way to transition a CR to `approved` without an implementation plan. There is no way to complete a CR without commit SHAs. Violations that predate this enforcement are permanently visible in the audit trail via `[BACKFILL]` markers.

### **7.2 Testing Requirements**

<!-- rule: mandatory -->

Per SOC 2 CC8.1 and STANDARD.md 5S methodology, all code changes MUST pass:

1. **Syntax check**: `python manage.py check`
2. **Migration check**: `python manage.py makemigrations --check --dry-run`
3. **Unit tests**: `python manage.py test` (relevant modules)
4. **Compliance check**: `python manage.py run_compliance --standards` (if standards affected)

Test results are logged as a `ChangeLog` entry with action `testing_completed`.

### **7.3 Post-Implementation Verification & Completion Checklist**

<!-- rule: mandatory -->

**Before transitioning a CR to `completed`, the author MUST:**

1. **Link commits**: Set `commit_shas` to the list of git SHAs that implement the change (see §5.4.2)
2. **Link log.md**: Set `log_md_ref` to the relevant section in `log.md`
3. **Run tests**: Execute relevant test suites and log results
4. **Run compliance**: If standards were affected, run `python manage.py run_compliance --standards`

`validate_for_transition('completed')` enforces items 1 and 2 — the CR cannot reach `completed` without them. Items 3 and 4 are verified by the `testing_completed` log entry.

After transitioning to `completed`, the system records:
- Commit SHA(s) of the changes (from `commit_shas`)
- Test results summary (from `testing_completed` log entry)
- Compliance check results (if applicable)
- Link to `log.md` entry (from `log_md_ref`)

#### **7.3.1 Deployment Linkage**

For changes that require production deployment, the `ops/deploy.sh` script (OPS-001 §4.4) provides automated CR integration via the `--cr <uuid>` flag:

- Creates a `ChangeLog` entry with `action='completed'` on successful deploy
- Records commit SHAs, deploy duration, and previous/new commit in `details`
- On rollback, creates a `ChangeLog` entry with `action='rolled_back'`

This closes the loop: CR → code → commit → deploy → ChangeLog. The full audit trail is traversable from any node.

Deploying without `--cr` is permitted only for emergency situations (§9). The deploy script logs a warning, and the `change_management` compliance check will flag unlinked deployments.

### **8.4 UUID Linking (Synara Convention)**

<!-- assert: ChangeRequest contains UUID-linked fields for cross-referencing compliance checks, drift violations, and audit entries | check=chg-uuid-linking -->
<!-- impl: syn/audit/models.py:ChangeRequest -->
<!-- test: syn.audit.tests.test_change_management.UUIDLinkingTest.test_uuid_link_fields_exist -->
<!-- test: syn.audit.tests.test_change_management.UUIDLinkingTest.test_parent_change_field -->
<!-- test: syn.audit.tests.test_change_management.ChangeRequestFieldsTest.test_correlation_id_field -->

All cross-references between change management and other audit subsystems use UUID-based linking per Synara convention:

| Field | Type | Links To |
|-------|------|----------|
| `ChangeRequest.correlation_id` | UUID | SysLogEntry.correlation_id — audit trail linkage |
| `ChangeRequest.compliance_check_ids` | JSON list of UUIDs | ComplianceCheck records that triggered or relate to change |
| `ChangeRequest.drift_violation_ids` | JSON list of UUIDs | DriftViolation records this change remediates |
| `ChangeRequest.audit_entry_ids` | JSON list of IDs | SysLogEntry records related to this change |
| `ChangeRequest.parent_change_id` | UUID | Parent ChangeRequest for sub-task hierarchy |
| `ChangeRequest.related_change_ids` | JSON list of UUIDs | Related ChangeRequest records |

The chain is **bidirectional** — findings link forward to their remediation, and changes link back to their source findings:

| Field | Type | Direction |
|-------|------|-----------|
| `ComplianceCheck.remediation_change_id` | UUID | Finding → ChangeRequest |
| `DriftViolation.remediation_change_id` | UUID | Finding → ChangeRequest |
| `ChangeRequest.compliance_check_ids` | UUID list | ChangeRequest → Findings |
| `ChangeRequest.drift_violation_ids` | UUID list | ChangeRequest → Findings |
| `ChangeRequest.related_change_ids` | UUID list | ChangeRequest ←→ ChangeRequest |

Every entity is discoverable by UUID. Start from any node in the chain — a compliance check failure, a drift violation, a change request, an audit log entry — and traverse to any related entity via UUID lookup. Dashboard badges are clickable links that resolve to the linked entity's detail view.

#### **8.4.1 Enforcement — Model Methods**

<!-- rule: mandatory -->

**All UUID linking MUST use the ChangeRequest model methods.** Direct manipulation of `related_change_ids`, `compliance_check_ids`, or `drift_violation_ids` fields is prohibited — the model methods enforce bidirectionality and create ChangeLog audit entries automatically.

| Method | What It Links | Bidirectional |
|--------|--------------|---------------|
| `cr.link_related(other_cr_id)` | CR ←→ CR via `related_change_ids` | Yes — sets both sides |
| `cr.link_compliance_checks([check_ids])` | CR ←→ ComplianceCheck via `compliance_check_ids` / `remediation_change_id` | Yes — sets both sides |
| `cr.link_drift_violations([violation_ids])` | CR ←→ DriftViolation via `drift_violation_ids` / `remediation_change_id` | Yes — sets both sides |

<!-- code: correct -->
```python
# Link two CRs — sets related_change_ids on BOTH, creates ChangeLog on source
cr.link_related(other_cr.id, actor="claude@svend.ai", message="Related: compliance expansion")

# Link compliance findings to their fix — sets compliance_check_ids AND remediation_change_id
cr.link_compliance_checks(
    [check.id for check in source_checks],
    actor="claude@svend.ai",
    message="Remediated session_security, error_handling, secret_management warnings",
)

# Link drift violations to their fix
cr.link_drift_violations([violation.id], actor="claude@svend.ai")
```

<!-- code: incorrect -->
```python
# WRONG — no bidirectionality, no audit trail
cr.related_change_ids.append(str(other_cr.id))
cr.save()
```

#### **8.4.2 API Support**

The `api_change_create` endpoint accepts `related_change_ids`, `compliance_check_ids`, and `drift_violation_ids` in the request body — all are linked bidirectionally via the model methods on creation.

The `api_change_transition` endpoint accepts action `"linked"` with details containing any combination of:

```json
{
    "action": "linked",
    "message": "Linking compliance findings to remediation CR",
    "details": {
        "related_change_ids": ["uuid-1", "uuid-2"],
        "compliance_check_ids": ["uuid-3"],
        "drift_violation_ids": ["uuid-4"]
    }
}
```

### **8.5 Compliance Remediation Linking**

<!-- assert: Compliance remediation changes must bidirectionally link to source ComplianceCheck UUIDs | check=chg-remediation-linking -->
<!-- impl: syn/audit/models.py:ChangeRequest -->
<!-- rule: mandatory -->

**When a ChangeRequest remediates a compliance check finding (fail or warning), the bidirectional UUID link MUST be established before the CR transitions to `completed`.**

This is not optional. Every audit fix produces two artifacts — the finding and the fix. If they aren't linked, the chain is broken and the remediation is unverifiable.

**Required step when fixing a compliance finding:**

Call `link_compliance_checks()` on the remediation CR **before** transitioning to `completed`. The method handles both directions and the audit trail in one call (see §8.4.1).

<!-- code: correct -->
```python
# After fixing compliance warnings/failures — BEFORE completing the CR:
cr.link_compliance_checks(
    [str(c.id) for c in source_checks],
    actor="claude@svend.ai",
    message="Remediated session_security, error_handling warnings",
)
```

<!-- code: incorrect -->
```python
# WRONG — manual field manipulation misses reverse link and audit trail
cr.compliance_check_ids = [str(c.id) for c in source_checks]
cr.save()
```

**What gets linked:**

| Scenario | Source Check | Link Direction |
|----------|-------------|----------------|
| Compliance warning fixed by config change | ComplianceCheck (status=warning) | Check → CR via `remediation_change_id` |
| Compliance failure fixed by code change | ComplianceCheck (status=fail) | Check → CR via `remediation_change_id` |
| New checks added that cover a gap | ComplianceCheck (status=pass, first run) | CR → Checks via `compliance_check_ids` |
| Multiple warnings fixed in single CR | All source ComplianceChecks | Both directions for all checks |

**The `change_management` compliance check SHOULD flag completed CRs that reference compliance findings in their description but have empty `compliance_check_ids`.**

## **9. EMERGENCY CHANGES**

### **9.1 Emergency Change Process**

<!-- rule: mandatory -->

<!-- sla: Emergency change retroactive risk assessment | metric=change_velocity | target=24h | window=per_incident | severity=high | measurement=automated | soc2=CC8.1 | check=sla-emergency-retro-review -->
<!-- sla: Post-incident review completion | metric=change_velocity | target=48h | window=per_incident | severity=high | measurement=automated | check=sla-post-incident-review -->

Emergency changes (`hotfix` type with `critical` priority) may bypass the normal approval workflow:

1. **Immediate**: Author creates ChangeRequest with type `hotfix`, status `in_progress`
2. **Concurrent**: Expedited risk assessment (single agent)
3. **Implementation**: Fix applied and deployed
4. **Within 1 hour**: ChangeLog entries documenting what was done and why
5. **Within 24 hours**: Full multi-agent risk assessment retroactively
6. **Within 48 hours**: Post-incident review logged as separate `plan` type change

### **9.2 Emergency Change Tracking**

The `is_emergency` flag on ChangeRequest marks emergency changes. The `change_management` compliance check specifically monitors:
- Emergency changes without retroactive risk assessment (violation after 24h)
- Emergency changes without post-incident review (violation after 48h)

### **9.3 Policy Impact Review**

<!-- rule: recommended -->

When a ChangeRequest modifies files listed in any policy's `<!-- policy-watches: ... -->` tag, the CR description SHOULD note which policies may be affected. The `policy_review` compliance check (CC1.5, CC5.3) automatically detects when watched code files have been modified more recently than the policy's last-updated date, flagging stale policies for review.

This creates a bidirectional link: policies watch code (via `policy-watches` tags), and the compliance system flags when code changes outpace policy updates.

## **11. ANTI-PATTERNS**

### **11.1 Unlogged Changes**

<!-- assert: All codebase changes must have associated ChangeRequest records | check=chg-no-unlogged -->
<!-- impl: syn/audit/compliance.py:check_change_management -->
<!-- rule: mandatory -->

**PROHIBITED:** Making code changes without creating a ChangeRequest.

Every commit in the repository should be traceable to a ChangeRequest. The `change_management` compliance check verifies this by comparing recent commits against ChangeRequest records.

### **11.2 Skipping Risk Assessment**

<!-- rule: mandatory -->

**PROHIBITED:** Implementing `feature`, `migration`, or `critical` risk changes without multi-agent risk assessment.

Risk assessment is not optional for high-impact changes. A `RiskAssessment` record with 4 `AgentVote` records MUST exist before a CR transitions past `approved` (see §6.4.1). The `change_management` compliance check scans all `feature`/`migration` CRs in `approved`/`in_progress`/`testing`/`completed` status and flags any without an associated `RiskAssessment` as a hard failure. If this was missed, follow the retroactive path in §6.4.3.

### **11.3 Unlinked Change Chains**

<!-- rule: mandatory -->

**PROHIBITED:** Change logs without proper linking to source issues, commits, or related changes.

The `details` field on ChangeLog entries MUST contain at minimum a `commit_sha` or `log_md_ref` for implementation checkpoints.

### **11.4 Unlinked Compliance Remediation**

<!-- rule: mandatory -->

**PROHIBITED:** Completing a ChangeRequest that fixes a compliance finding without establishing the bidirectional UUID link to the source ComplianceCheck record(s).

If a CR remediates a compliance failure or warning and transitions to `completed` without setting both `ComplianceCheck.remediation_change_id` and `ChangeRequest.compliance_check_ids`, the audit trail is broken. The finding exists, the fix exists, but nothing connects them — making the remediation unverifiable during SOC 2 audit.

See §8.5 for the required linking process.

### **11.5 Silent Emergency Changes**

<!-- rule: mandatory -->

**PROHIBITED:** Emergency changes without retroactive documentation.

Emergency changes bypass pre-approval but MUST have retroactive risk assessment within 24 hours and post-incident review within 48 hours.

### **11.6 Retroactive CRs for Non-Emergency Changes**

<!-- rule: mandatory -->

**PROHIBITED:** Creating a ChangeRequest after code has already been committed for non-emergency change types (`feature`, `enhancement`, `bugfix`, `migration`, `infrastructure`, `security`, `debt`).

The CR-before-code rule (§5.4.1) exists because the authorization chain is meaningless if the code is already written. A retroactive CR for a migration is especially dangerous — the schema change has already been applied without a rollback plan, risk assessment, or approval on record.

If this does happen, do not falsify the timeline. Add a `comment` log entry acknowledging the gap (§5.4.1). The honest record is more valuable than a clean-looking audit trail.
