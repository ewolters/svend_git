## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

LLM-001 defines the architecture, model selection, rate limiting, prompt construction, and security requirements for all LLM integration in the Svend platform.

**Core Principle:**

> The LLM is a tool, not the product. Deterministic computation handles analysis; the LLM handles translation between human intent and structured output. Every LLM call is rate-limited, tier-gated, logged, and the content encrypted at rest.

### **1.2 Scope**

**Applies to:**
- `agents_api/llm_manager.py` — central LLM manager
- `agents_api/guide_views.py` — AI decision guide
- `agents_api/synara/llm_interface.py` — Synara belief engine LLM bridge
- `chat/models.py` — conversation storage
- Any code calling `LLMManager.chat()`

**Does NOT apply to:**
- Local model inference (Qwen — currently disabled)
- The experimenter agent (deterministic, no LLM)
- Compliance check automation

## **9. ERROR HANDLING**

### **9.1 Failure Modes**

<!-- assert: LLM failures handled gracefully without retry | check=llm-error-handling -->
<!-- impl: agents_api/llm_manager.py -->
<!-- test: syn.audit.tests.test_llm.LLMErrorHandlingTest.test_status_method_returns_dict -->
<!-- test: syn.audit.tests.test_llm.LLMErrorHandlingTest.test_returns_none_when_no_api_key -->
<!-- test: syn.audit.tests.test_llm.LLMResetTest.test_reset_clears_client -->

| Failure | Handling |
|---------|----------|
| API key missing | Returns None, logs warning |
| Package not installed | Logs error |
| API request exception | Logs error, returns None |
| Rate limit exceeded | Returns `{"rate_limited": true}`, HTTP 429 |
| Rate limit check fails | Returns service unavailable, HTTP 503 |
| Timeout (120s) | Exception propagated |

<!-- assert: LLM calls use single-attempt fail-fast with no retry loop | check=llm-no-retry -->
<!-- impl: agents_api/llm_manager.py -->
<!-- test: syn.audit.tests.test_llm.LLMErrorHandlingTest.test_returns_none_when_no_api_key -->

### **9.2 No Retry**

Single attempt per call. No exponential backoff. Fail-fast design — the user can retry manually. This avoids compounding latency on degraded API.

<!-- assert: No degraded-mode LLM fallback — deterministic tools continue without LLM | check=llm-no-fallback -->
<!-- impl: agents_api/llm_manager.py -->
<!-- test: syn.audit.tests.test_llm.LLMErrorHandlingTest.test_status_method_returns_dict -->

### **9.3 No Fallback**

When LLM unavailable, endpoints return error. No degraded-mode response. Deterministic tools (DSW, DOE, SPC) continue functioning without LLM.

## **10. SYNARA BELIEF ENGINE INTEGRATION**

### **10.1 LLM Bridge**

<!-- assert: Synara uses LLM for hypothesis generation and validation | check=llm-synara -->
<!-- impl: agents_api/synara/llm_interface.py -->
<!-- impl: agents_api/synara/belief.py -->

`SynaraLLMInterface` provides 4 integration points:

| Method | Purpose | Temperature |
|--------|---------|-------------|
| `generate_hypotheses_llm()` | Propose hypotheses from expansion signals | 0.3 |
| `validate_graph_llm()` | Find logical issues in causal graph | 0.3 |
| `interpret_evidence_llm()` | Plain-text interpretation of updates | 0.3 |
| `document_findings_llm()` | Structured report generation | 0.3 |

The belief engine itself is purely Bayesian: `P(h|e) = P(e|h) * P(h) / P(e)`. The LLM assists with hypothesis space exploration, not probability calculation.

<!-- test: agents_api.tests.test_llm_service.LLMServiceModelTests.test_enterprise_tier_gets_opus -->

`LLMService` wraps `LLMManager.chat()` with a simpler API and consistent error handling. Returns `LLMResult` dataclass (never None, never raises). Context-based temperature defaults: `analysis=0.3`, `critique=0.5`, `chat=0.7`, `generation=0.7`.

```python
from agents_api.llm_service import llm_service

result = llm_service.chat(request.user, "Analyze this defect pattern",
    system="You are a quality engineer.",
    context="analysis")

if result.success:
    print(result.content)
elif result.rate_limited:
    return JsonResponse({"error": "Rate limit exceeded"}, status=429)
```

**New code SHOULD use `llm_service.chat()`** instead of calling `LLMManager.chat()` directly. `LLMManager` remains the underlying engine; `LLMService` is the public interface.

## **12. ANTI-PATTERNS**

### **12.1 Direct Anthropic Client Usage**

<!-- code: prohibited -->
```python
import anthropic
client = anthropic.Anthropic()
response = client.messages.create(...)
```

<!-- code: correct -->
```python
from agents_api.llm_service import llm_service
result = llm_service.chat(
    user=request.user,
    prompt="Your question here",
    system="You are...",
    context="analysis",
    max_tokens=500,
)
```

All calls through `llm_service.chat()` or `LLMManager.chat()` for rate limiting, usage tracking, and tier-based model selection.

### **12.2 Unbounded Token Limits**

**PROHIBITED:** Setting `max_tokens` above 4096 without justification. Default to 500 for short responses, 2048 for reports.

### **12.3 Storing Plaintext Conversations**

**PROHIBITED:** Storing message content in non-encrypted fields. Use `EncryptedTextField` and `EncryptedJSONField`.

### **12.4 Skipping Rate Limit Check**

**PROHIBITED:** Using `skip_rate_limit=True` outside of internal/staff contexts.
