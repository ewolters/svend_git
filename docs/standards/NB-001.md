## 1. PURPOSE

NB-001 defines the Notebook — the tactical execution layer for continuous improvement. A Notebook is a scrapbook with compute: it persists trials, calculator outputs, simulator results, analysis snapshots, and any other artifact the user generates while improving a process. Structure accumulates from work, not from forms.

### 1.1 The Kaizen Thesis

Kaizen is not a blitz. It is longitudinal attrition against deeply entrenched waste and cultural roots. A blitz assumes the enemy collapses on first contact. Attrition assumes the enemy adapts, and so must you.

The Notebook is the operational log of that campaign. Each trial is a sortie. Some succeed. Some don't. The notebook tracks what was tried, what the data showed, and whether it moved the needle — over weeks, months, or years. It connects the shop floor to strategic intent (Hoshin) and carries learning forward to the next team that faces the same problem (Yokoten).

### 1.2 Design Principles

1. **No forced structure.** A notebook starts with a title. Everything else is optional until the user does the work that creates it.
2. **Before/after is the core trial primitive.** Every trial has a before state and an after state. The delta is computed, not described.
3. **Pages are freeform.** Calculator outputs, simulator results, notes, kanban cards — anything can be a page. Pages don't have to connect to trials. They persist as evidence of thinking.
4. **Verdicts are explicit.** Each trial ends with: improved, no effect, degraded, or inconclusive. This is the data Hoshin needs.
5. **Reflection is built in.** When a notebook concludes, the system prompts for Hansei Kai. Learning is distilled and carried forward.
6. **Yokoten is first-class.** Learnings propagate across notebooks, charters, sites, and tenants. The insight doesn't die when the notebook closes.

### 1.3 What This Replaces

- **Investigation model (CANON-002 §7)** — Deprecated. Notebook absorbs the lifecycle and tool linkage.
- **StudyAction model** — Absorbed into TrialAction.
- **The "Study" label** — Project is the Charter (strategic intent). Notebook is the execution log (tactical). Neither is called "study."

### 1.4 Hierarchy

```
Project (Charter)                    — strategic intent, SMART goal, team, scope
  ├── HoshinProject (Enterprise)     — savings tracking, bowling chart, X-Matrix
  └── Notebook[] (1:many)            — tactical execution logs
        ├── Trial[]                  — structured before/after comparisons
        │     ├── TrialToolLink[]    — linked tool outputs (before/after/supporting)
        │     └── TrialAction[]      — triggered actions (CAPA, audit, etc.)
        ├── Page[]                   — freeform: calculators, simulators, notes, snapshots
        ├── HanseiKai                — reflection on conclusion
        └── Yokoten[]                — learnings carried forward
```

## 2. MODEL DESIGN

### 2.1 Notebook

```
Notebook
  id                UUID (PK)
  project           FK → core.Project (the charter this notebook executes against)
  title             CharField(300)
  description       TextField (optional — can be empty at creation)
  status            open | active | concluded | archived

  # Baseline — the starting point
  baseline_summary  TextField (optional — human description)
  baseline_metric   CharField (optional — e.g., "Cpk", "scrap_rate", "OEE")
  baseline_value    FloatField (optional — e.g., 0.82, 4.7, 62.3)
  baseline_unit     CharField (optional — e.g., "%", "ppm", "index")
  baseline_analysis FK → DSWResult (optional — linked analysis that established baseline)
  baseline_date     DateField (optional)

  # Current state — updated after each adopted trial
  current_value     FloatField (optional — latest post-trial measurement)
  current_analysis  FK → DSWResult (optional — linked analysis for current state)
  current_date      DateField (optional)

  # Active trial quick-access (invisible below Enterprise)
  active_trial      FK → Trial (nullable — the trial currently in progress)

  # Synara causal graph (per notebook, not per charter)
  synara_state      JSONField (default={} — Bayesian belief network)

  # Metadata
  owner             FK → User
  tenant            FK → Tenant (optional)
  created_at        DateTimeField (auto)
  updated_at        DateTimeField (auto)
  concluded_at      DateTimeField (optional)
```

**State machine:**
```
open → active → concluded → archived
```
- `open` — Created, baseline may or may not be set
- `active` — At least one trial or page exists
- `concluded` — User has declared the notebook done (Hansei Kai prompted)
- `archived` — Historical record, read-only

**Transition to `active`** happens automatically when the first trial or page is created.

### 2.2 Trial

```
Trial
  id              UUID (PK)
  notebook        FK → Notebook
  sequence        PositiveInt (auto-increment within notebook — Trial 1, 2, 3...)
  title           CharField(300) — what was changed
  description     TextField (optional — details of the change)

  # Before state
  before_value    FloatField (optional — metric value before this trial)
  before_analysis FK → DSWResult (optional — linked analysis)
  before_date     DateField (optional)

  # After state
  after_value     FloatField (optional — metric value after this trial)
  after_analysis  FK → DSWResult (optional — linked analysis)
  after_date      DateField (optional)

  # Verdict
  verdict         improved | no_effect | degraded | inconclusive | pending
  verdict_narrative TextField (optional — system-generated narrative with stats)
  delta           FloatField (computed: after_value - before_value, nullable)
  delta_pct       FloatField (computed: delta / before_value * 100, nullable)
  adopted         BooleanField (default False — was this change kept?)

  # Evidence linkage (CANON-002 compatible)
  evidence_links  M2M → core.Evidence (optional — Bayesian evidence from analysis bridge)

  # Metadata
  started_at      DateTimeField (auto)
  completed_at    DateTimeField (optional)
  created_by      FK → User
```

**Verdict rules (§2.2.1):**
- `improved` — delta moves toward the charter's goal target
- `degraded` — delta moves away from the charter's goal target
- `no_effect` — delta within noise threshold (configurable, default ±2% or not statistically significant)
- `inconclusive` — insufficient data or analysis not yet run
- `pending` — trial in progress, after state not yet measured

**Verdict narrative (§2.2.2):** When before and after analyses are both linked, the system computes statistical significance (p-value, effect size, confidence interval) and generates a narrative suggestion. Example: *"Statistically significant improvement (p=0.003, Cohen's d=0.84). Scrap rate decreased 1.6pp (4.7% → 3.1%). Effect size is large — this change accounts for most of the observed variance."* The user can accept, modify, or override.

**Delta direction (§2.2.3):** Determined by the charter's goal. If `goal_target > goal_baseline`, positive delta = improvement. If `goal_target < goal_baseline` (reducing defects), negative delta = improvement.

**Trial branching (§2.2.4):** Implicit. If Trial 2 fails, Trial 3 sets `before_value` to Trial 1's `after_value`. No explicit dependency graph — the user picks which baseline to measure from.

### 2.3 NotebookPage

```
NotebookPage
  id              UUID (PK)
  notebook        FK → Notebook
  page_type       calculator | simulator | analysis | note | attachment
  title           CharField(300)
  source_tool     CharField — tool that generated this page:
                    oee, kanban, takt, cpk, bayesian_cpk, sigma, fpy_rty,
                    fmea_rpn, gage_rr, sample_size, control_chart,
                    plantsim, dsw, spc, doe, monte_carlo, tolerance_stackup,
                    variance_propagation, note, other
  inputs          JSONField (frozen — what the user entered)
  outputs         JSONField (frozen — what the tool computed)
  rendered_html   TextField (optional — frozen visual output for display)
  narrative       TextField (optional — system-generated or user-written explanation)
  sequence        PositiveInt (ordering within notebook)

  # Optional trial linkage
  trial           FK → Trial (nullable — if this page is evidence for a specific trial)
  trial_role      before | after | supporting | NULL

  # Metadata
  created_at      DateTimeField (auto)
  created_by      FK → User
```

**Key principle: pages are frozen snapshots.** When you run the OEE calculator with inputs, the output renders in the notebook and stays there. It is not a live link to the calculator — it is a point-in-time record. To re-run with different inputs, create a new page.

**Pages don't have to feed anything.** A kanban card layout, a plant simulation screenshot, a takt time calculation — they just persist in the notebook as evidence of thinking. They can optionally link to a trial (`trial` FK + `trial_role`), but most pages will be standalone.

### 2.4 TrialToolLink

```
TrialToolLink
  id              UUID (PK)
  trial           FK → Trial
  content_type    FK → ContentType
  object_id       UUID
  tool_output     GenericForeignKey (content_type, object_id)
  tool_type       CharField — "spc", "dsw", "doe", "rca", "fmea", etc.
  role            before | after | supporting
  linked_at       DateTimeField (auto)
  linked_by       FK → User
```

**`role` field** distinguishes whether the tool output was used as the before measurement, the after measurement, or supporting evidence.

### 2.5 TrialAction

```
TrialAction
  id              UUID (PK)
  trial           FK → Trial
  action_type     raise_capa | schedule_audit | request_doc_update |
                  flag_training_gap | flag_fmea_update
  target_type     CharField — model name of target
  target_id       UUID
  notes           TextField (optional)
  created_by      FK → User
  created_at      DateTimeField (auto)
```

Absorbs `StudyAction` functionality but attached to Trial instead of Project.

### 2.6 HanseiKai (Reflection)

```
HanseiKai
  id              UUID (PK)
  notebook        OneToOne → Notebook (created on conclusion)
  what_went_well  TextField
  what_didnt      TextField
  what_next       TextField
  key_learning    TextField — one sentence distillation (the yokoten seed)
  carry_forward   BooleanField (default False — should this learning propagate?)

  created_at      DateTimeField (auto)
  created_by      FK → User
```

**When a notebook transitions to `concluded`, the system prompts for Hansei Kai.** This is Eric's practice from directing OpEx at Fort Dearborn: after report-outs, gather the team and ask three questions. The `key_learning` is the one-liner that captures the insight in transferable form.

If `carry_forward=True`, a Yokoten record is automatically created from `key_learning`.

### 2.7 Yokoten (Lateral Transfer of Learning)

```
Yokoten
  id                UUID (PK)
  source_notebook   FK → Notebook (where the learning originated)
  source_trial      FK → Trial (nullable — specific trial that produced the insight)
  learning          TextField (the distilled insight — from HanseiKai.key_learning)
  context           TextField (what conditions made this work — process type, equipment, etc.)
  applicable_to     JSONField (tags for matching: process types, defect categories, industries)

  created_by        FK → User
  created_at        DateTimeField (auto)
```

```
YokotenAdoption
  id                UUID (PK)
  yokoten           FK → Yokoten
  target_notebook   FK → Notebook (the notebook that adopted this learning)
  adopted_by        FK → User
  adopted_at        DateTimeField (auto)
  outcome           helped | not_applicable | adapted
  notes             TextField (optional — how it was adapted or why it didn't apply)
```

**Yokoten visibility by tier (§2.7.1):**

| Scope | Pro | Team | Enterprise |
|-------|-----|------|------------|
| Own notebooks | Yes | Yes | Yes |
| Same charter (sibling notebooks) | Yes | Yes | Yes |
| Same team's charters | — | Yes | Yes |
| Same site | — | — | Yes |
| Cross-site (same tenant) | — | — | Yes |

**Surfacing pattern:** When a user starts a new notebook or creates a new trial, the system checks for relevant yokoten (matched by `applicable_to` tags, charter domain, or keyword similarity) and surfaces them: *"A team at Fort Worth found that adjusting feed rate to 10mm/s reduced scrap by 1.6pp on a similar line. Want to start from their learning?"*

If adopted, a `YokotenAdoption` record is created. If the adoption works, that's tracked. The learning compounds across the organization.

## 3. INTEGRATION POINTS

### 3.1 Analysis Workbench → Notebook

When a user selects a notebook in the analysis workbench toolbar:
- Analysis results are saved as `NotebookPage` (page_type=analysis, frozen snapshot)
- If a trial is in progress (verdict=pending), the result can be attached as before or after measurement
- The analysis bridge creates evidence and links it to the active trial

**Dispatch integration (replaces investigation_id):**
```python
# In analysis/dispatch.py run_analysis():
notebook_id = body.get("notebook_id")
trial_id = body.get("trial_id")  # optional — specific trial
```

### 3.2 Operations Calculators → Notebook

Each calculator (OEE, Takt, Kanban, Cpk, Sigma, FPY/RTY, Gage R&R, FMEA RPN, Sample Size, Control Chart) follows the same pattern:

1. User runs calculator with inputs
2. If a notebook is selected, the system creates a `NotebookPage` with:
   - `inputs` = frozen JSON of what the user entered
   - `outputs` = frozen JSON of what the calculator computed
   - `rendered_html` = frozen visual output (chart, table, cards)
   - `source_tool` = calculator identifier
3. Page appears in the notebook immediately
4. User can optionally link the page to a trial as evidence

**Calculator pages are pure evidence.** They don't trigger Bayesian updates or feed the verdict system. They persist as records of what was calculated and when.

### 3.3 Plant Simulator → Notebook

The discrete event simulator (DES) at `/app/simulator/` creates `NotebookPage` entries with:
- `page_type` = simulator
- `source_tool` = plantsim
- `inputs` = simulation configuration (layout, rates, WIP levels, etc.)
- `outputs` = simulation results (throughput, utilization, bottleneck, queue times)
- `rendered_html` = simulation visualization

**Predicted vs actual:** When a trial has both a supporting simulation page and an after measurement, the narrative engine can compare predicted outcome to actual outcome. This is model validation — good engineering practice but not required for verdict.

### 3.4 Notebook → Hoshin

```
HoshinProject.project ← core.Project → Notebook[] → Trial[]
```

Hoshin reads:
- **Target** from `project.goal_target`
- **Baseline** from `notebook.baseline_value`
- **Current** from `notebook.current_value`
- **Progress** from `(current - baseline) / (target - baseline) * 100`
- **Adopted changes** from `Trial.objects.filter(notebook=nb, adopted=True)`

Bowling chart auto-populated from trial dates and values.

### 3.5 Notebook → A3

A3 reports map to the notebook:
- **Background** → Charter problem statement
- **Current condition** → Notebook baseline
- **Goal** → Charter goal statement
- **Root cause** → Trial evidence (which changes worked, which didn't)
- **Countermeasures** → Adopted trials
- **Implementation plan** → Active/pending trials
- **Follow-up** → Notebook current state vs goal + Hansei Kai reflection

### 3.6 Notebook → VSM

Value Stream Maps identify kaizen bursts. Each burst can generate:
1. A Project (Charter) with the strategic intent
2. A HoshinProject wrapping that charter
3. A Notebook for tactical execution

**Flow:** VSM burst → Charter → HoshinProject + Notebook → Trials → Results → Hoshin bowling chart → X-Matrix

### 3.7 Evidence Bridge (CANON-002 Compatibility)

The evidence weighting system from CANON-002 §2-4 remains unchanged. Trials create evidence through the same bridge. The Bayesian update to notebook hypotheses (Synara state) happens when evidence is created, not when the trial verdict is set. The verdict is a human judgment informed by the evidence — the system suggests, the human decides.

## 8. RESOLVED DECISIONS

1. **Synara state lives on the Notebook.** Each notebook has its own causal graph. Trial outcomes are evidence that updates beliefs. The charter doesn't need its own Synara state.

2. **Multiple notebooks per charter: yes.** One charter can have multiple improvement campaigns. Each notebook tracks its own baseline, trials, and current state independently.

3. **Charter and Notebook link to one active trial.** `Notebook.active_trial` (nullable FK → Trial). Visible at Enterprise tier for Hoshin integration.

4. **System suggests verdicts with narrative.** Statistical significance computed when before/after analyses are linked. User can accept, modify, or override.

5. **Trial branching is implicit.** `before_value` naturally handles branching. No explicit dependency graph.

6. **Pages are frozen snapshots.** Calculator/simulator outputs captured as immutable JSON + optional rendered HTML. Not live links — point-in-time records.

7. **Hansei Kai on Notebook, not Charter.** The notebook is where the work happened, so that's where the reflection belongs.

8. **Yokoten is first-class.** Not a tag, not a comment — a model with adoption tracking and outcome recording.
