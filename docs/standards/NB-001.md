## 1. PURPOSE

NB-001 defines the Notebook and Trial models — the tactical execution layer for continuous improvement. A Notebook is a trial log attached to a Project (Charter). It does not impose structure up front. Structure accumulates from work: baselines measured, changes tried, results compared.

### 1.1 Design Principles

1. **No forced structure.** A notebook starts with a title. Everything else is optional until the user does the work that creates it.
2. **Before/after is the core primitive.** Every trial has a before state and an after state. The delta is computed, not described.
3. **Verdicts are explicit.** Each trial ends with: improved, no effect, degraded, or inconclusive. This is the data Hoshin needs.
4. **Analysis results link, not copy.** Trials point to DSW/SPC/DOE results by reference (FK or UUID). The notebook doesn't store statistical output — it stores the verdict and the link.
5. **Notebooks feed Hoshin.** The charter sets the target. The notebook tracks progress toward it through trials. Hoshin reads current state from the latest successful trial.

### 1.2 What This Replaces

- **Investigation model (CANON-002 §7)** — Deprecated. Notebook absorbs the lifecycle (open/active/concluded) and tool linkage. The Investigation model's versioning and membership features are deferred — they can be added to Notebook if needed.
- **StudyAction model** — Absorbed into Trial. A trial that triggers a CAPA, audit, or doc update records that as a linked action, not a separate model.
- **The "Study" label** — Project remains the charter. Notebook is the execution log. Neither is called "study."

## 2. MODEL DESIGN

### 2.1 Notebook

```
Notebook
  id              UUID (PK)
  project         FK → core.Project (the charter this notebook executes against)
  title           CharField(300)
  description     TextField (optional — can be empty at creation)
  status          open | active | concluded | archived

  # Baseline — the starting point
  baseline_summary    TextField (optional — human description)
  baseline_metric     CharField (optional — e.g., "Cpk", "scrap_rate", "OEE")
  baseline_value      FloatField (optional — e.g., 0.82, 4.7, 62.3)
  baseline_unit       CharField (optional — e.g., "%", "ppm", "index")
  baseline_analysis   FK → DSWResult (optional — linked analysis that established baseline)
  baseline_date       DateField (optional)

  # Current state — updated after each trial
  current_value       FloatField (optional — latest post-trial measurement)
  current_analysis    FK → DSWResult (optional — linked analysis for current state)
  current_date        DateField (optional)

  # Metadata
  owner           FK → User
  tenant          FK → Tenant (optional)
  created_at      DateTimeField (auto)
  updated_at      DateTimeField (auto)
  concluded_at    DateTimeField (optional)
```

**State machine:**
```
open → active → concluded → archived
```
- `open` — Created, baseline may or may not be set
- `active` — At least one trial exists
- `concluded` — User has declared the notebook done (goal met or abandoned)
- `archived` — Historical record, read-only

**Transition to `active`** happens automatically when the first trial is created.

### 2.2 Trial

```
Trial
  id              UUID (PK)
  notebook        FK → Notebook
  sequence        PositiveInt (auto-increment within notebook — Trial 1, 2, 3...)
  title           CharField(300) — what was changed
  description     TextField (optional — details of the change)

  # Before state
  before_value    FloatField (optional — metric value before this trial)
  before_analysis FK → DSWResult (optional — linked analysis)
  before_date     DateField (optional)

  # After state
  after_value     FloatField (optional — metric value after this trial)
  after_analysis  FK → DSWResult (optional — linked analysis)
  after_date      DateField (optional)

  # Verdict
  verdict         improved | no_effect | degraded | inconclusive | pending
  delta           FloatField (computed: after_value - before_value, nullable)
  delta_pct       FloatField (computed: delta / before_value * 100, nullable)
  adopted         BooleanField (default False — was this change kept?)

  # Evidence linkage (CANON-002 compatible)
  evidence_links  M2M → core.Evidence (optional — Bayesian evidence created by analysis bridge)

  # Tool linkage (absorbs InvestigationToolLink pattern)
  tool_links      Reverse FK from TrialToolLink (generic FK to any tool output)

  # Actions triggered (absorbs StudyAction)
  actions         Reverse FK from TrialAction (CAPA raised, audit scheduled, etc.)

  # Metadata
  started_at      DateTimeField (auto)
  completed_at    DateTimeField (optional)
  created_by      FK → User
```

**Verdict rules (§2.2.1):**
- `improved` — delta moves toward the charter's goal target
- `degraded` — delta moves away from the charter's goal target
- `no_effect` — delta within noise threshold (configurable, default ±2% or not statistically significant)
- `inconclusive` — insufficient data or analysis not yet run
- `pending` — trial in progress, after state not yet measured

**Delta direction:** Determined by the charter's goal. If goal_target > goal_baseline, positive delta = improvement. If goal_target < goal_baseline (reducing defects), negative delta = improvement. The notebook inherits this direction from `project.goal_baseline` and `project.goal_target`.

### 2.3 TrialToolLink

```
TrialToolLink
  id              UUID (PK)
  trial           FK → Trial
  content_type    FK → ContentType
  object_id       UUID
  tool_output     GenericForeignKey (content_type, object_id)
  tool_type       CharField — "spc", "dsw", "doe", "rca", "fmea", etc.
  role            before | after | supporting
  linked_at       DateTimeField (auto)
  linked_by       FK → User
```

**`role` field** distinguishes whether the tool output was used as the before measurement, the after measurement, or supporting evidence.

### 2.4 TrialAction

```
TrialAction
  id              UUID (PK)
  trial           FK → Trial
  action_type     raise_capa | schedule_audit | request_doc_update | flag_training_gap | flag_fmea_update
  target_type     CharField — model name of target
  target_id       UUID
  notes           TextField (optional)
  created_by      FK → User
  created_at      DateTimeField (auto)
```

Absorbs `StudyAction` functionality but attached to Trial instead of Project.

## 3. INTEGRATION POINTS

### 3.1 Analysis Workbench → Notebook

When a user selects a notebook in the analysis workbench toolbar:
- Analysis results are linked to the notebook via `DSWResult`
- If a trial is in progress (verdict=pending), the result can be attached as before or after measurement
- The investigation bridge in `analysis/dispatch.py` creates evidence and links it to the active trial

**Dispatch integration (replaces investigation_id):**
```python
# In analysis/dispatch.py run_analysis():
notebook_id = body.get("notebook_id")
trial_id = body.get("trial_id")  # optional — specific trial
```

### 3.2 Notebook → Hoshin

HoshinProject already has FK to `core.Project`. The notebook is also FK to `core.Project`. So:

```
HoshinProject.project ← core.Project → Notebook → Trial[]
```

Hoshin reads:
- **Target** from `project.goal_target`
- **Baseline** from `notebook.baseline_value`
- **Current** from `notebook.current_value`
- **Progress** from `(current - baseline) / (target - baseline) * 100`
- **Adopted changes** from `Trial.objects.filter(notebook=nb, adopted=True)`

This is the closed loop: Charter sets the target → Notebook tracks trials → Hoshin reports progress.

### 3.3 Notebook → A3

A3 reports map to the notebook naturally:
- **Background** → Charter problem statement
- **Current condition** → Notebook baseline
- **Goal** → Charter goal statement
- **Root cause** → Trial evidence (which changes worked, which didn't)
- **Countermeasures** → Adopted trials
- **Implementation plan** → Active/pending trials
- **Follow-up** → Notebook current state vs goal

### 3.4 Notebook → VSM

Value Stream Maps identify kaizen bursts (improvement opportunities). Each burst can generate:
1. A Project (Charter) with the strategic intent
2. A HoshinProject wrapping that charter
3. A Notebook for tactical execution

The flow: **VSM burst → Charter → HoshinProject + Notebook → Trials → Results → Hoshin bowling chart**

### 3.5 Evidence Bridge (CANON-002 Compatibility)

The evidence weighting system from CANON-002 §2-4 remains unchanged. Trials create evidence through the same bridge:

```
Trial.before_analysis → DSWResult → Evidence (weighted per CANON-002 §3)
Trial.after_analysis  → DSWResult → Evidence (weighted per CANON-002 §3)
```

The Bayesian update to charter hypotheses happens when evidence is created, not when the trial verdict is set. The verdict is a human judgment informed by the evidence, not derived from it automatically.

## 4. MIGRATION PATH

### 4.1 From Investigation

1. **Existing investigations** with `synara_state` data migrate to Notebooks with the graph preserved in a `synara_state` JSONField on Notebook (optional field, for Bayesian reasoning continuity)
2. **InvestigationToolLink** records migrate to TrialToolLink (or Notebook-level links if no trial context)
3. **InvestigationMembership** deferred — Notebook initially supports single owner. Team features can be added later via the same pattern
4. **Investigation.exported_to_project** → Notebook.project (direct FK, no export needed — notebook is already on the project)

### 4.2 From StudyAction

1. **StudyAction** records migrate to TrialAction, attached to the most relevant trial or created as standalone notebook-level actions

### 4.3 Database Migration

- New tables: `core_notebook`, `core_trial`, `core_trial_tool_link`, `core_trial_action`
- Deprecate: `core_investigation`, `core_investigation_membership`, `core_investigation_tool_link`, `core_study_action`
- Data migration script moves existing records
- Old tables retained as read-only for 1 release cycle, then dropped

## 7. OPEN QUESTIONS

1. **Synara state on Notebook?** Investigation stored `synara_state` (the causal graph). Should Notebook carry this forward, or should Bayesian reasoning live only on the charter's Project?
2. **Multiple notebooks per project?** A charter could have multiple improvement campaigns. Default: yes, one-to-many.
3. **Trial branching?** If Trial 2 fails, Trial 3 starts from Trial 1's state. The `before_value` handles this naturally (user picks which baseline to measure from), but should we track the dependency graph explicitly?
4. **Automatic verdict suggestion?** If before and after analyses are linked, could the system suggest a verdict based on statistical significance? Or is the verdict always a human call?
5. **Notebook-level Synara graph?** Should each notebook have its own belief network, or share the charter's?
