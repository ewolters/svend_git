## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

SAF-001 defines the HIRARC (Hazard Identification, Risk Assessment, and Risk Control) program architecture for Svend Safety. It covers the data model, audit scheduling, hazard identification vectors, card-to-FMEA pipeline, 5S parallel tracking, management operating system, KPI governance, and security constraints.

**Core Principle:**

> Every hazard that is identified but not assessed is a liability.
> Every risk that is assessed but not controlled is negligence.
> HIRARC closes the loop: Identify → Assess → Control → Verify.

### **1.2 Scope**

**Applies to:**
- Frontier Card observations (primary hazard identification vector)
- Audit scheduling and assignment management
- FMEA central engine (card-to-risk transformation)
- 5S Pareto aggregation with cross-feed to safety
- Management Operating System (MOS) cadence
- Safety KPI dashboard (leading and lagging indicators)

**Does NOT apply to:**
- Workers' compensation case management (external system)
- OSHA recordkeeping 300/300A log (separate compliance module — future)
- Industrial hygiene monitoring (IH sampling, PEL/TLV tracking — future)
- Emergency response planning (separate from routine hazard identification)

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Hazard** | A source, situation, or act with potential for harm in terms of human injury, ill health, or damage to property/environment |
| **Risk** | The combination of the likelihood of an occurrence of a hazardous event and the severity of injury or ill health that can be caused by the event |
| **FMEA** | Failure Mode and Effects Analysis — systematic method for evaluating failure modes, their effects, and criticality (S×O×D = RPN) |
| **RPN** | Risk Priority Number — product of Severity × Occurrence × Detection (1-1000 scale) |
| **HI Vector** | Hazard Identification Vector — a structured mechanism for discovering hazards (Frontier Card is the primary HI vector; Blue Tag, BBSO, and Incident Report are planned) |
| **Frontier Zone** | A deliberately non-standard area targeted for safety audits: transition areas, overhead spaces, below-floor, temporal frontiers (shift changes), hidden infrastructure |
| **Frontier Card** | A structured observation card used during Frontier audits — front: 20 safety observation items; back: 25 5S deficiency tallies; plus 3 operator interaction questions |
| **Cross-feed** | The rule that a 5S deficiency with safety consequence crosses from the 5S parallel track into the FMEA risk register |
| **MOS** | Management Operating System — the cadence of daily, weekly, monthly, and quarterly reviews that govern safety performance |
| **MDI** | Managing for Daily Improvement — the daily huddle cadence for reviewing leading indicators |
| **STOP** | Safety Training Observation Program — a behavioral observation methodology (implemented via Frontier Card) |
| **D-S-O-I-R** | Define-Schedule-Observe-Intervene-Review — the 5-phase audit cycle |
| **Hierarchy of Controls** | E-S-E-A-P: Elimination → Substitution → Engineering Controls → Administrative Controls → PPE (most to least effective) |

## **4. SYSTEM ARCHITECTURE**

### **4.1 Three-Stage Architecture**

The HIRARC system operates as a three-stage pipeline:

```
Stage 1: Hazard Identification (HI)
    Frontier Card audits → at-risk observations → severity tagging
         ↓
Stage 2: FMEA Risk Assessment
    Card findings → FMEA rows → S×O×D = RPN → risk ranking
         ↓
Stage 3: Verified Controls
    Recommended actions → hierarchy of controls → verification audits → RPN re-score
```

Each stage produces traceable, auditable records. No hazard enters Stage 2 without a Frontier Card. No control is verified without a follow-up audit.

### **4.3 FMEA Central Engine**

#### **4.3.1 Hazard-to-Risk Transformation**

The FMEA central engine transforms identified hazards into scored risks using the standard S×O×D formula:

```
RPN = Severity × Occurrence × Detection

Where:
  Severity   (S): 1-10 — how severe the potential injury/consequence
  Occurrence (O): 1-10 — how likely the hazard event occurs
  Detection  (D): 1-10 — how likely current controls detect the hazard before harm

RPN range: 1 (lowest risk) to 1000 (highest risk)
```

FMEA rows are created in `agents_api.models.FMEARow` (shared with QMS-001 FMEA infrastructure). Safety-originated rows are distinguished by `process_step` prefix `[ZoneName]`.

#### **4.3.2 Severity Mapping from Frontier Card**

<!-- assert: FrontierCard.SEVERITY_TO_FMEA maps card severity tags C→10, H→8, M→5, L→2 for FMEA row creation | check=saf-severity-mapping -->
<!-- impl: safety/models.py:FrontierCard.severity_to_fmea_score -->

<!-- rule: mandatory -->

| Card Severity | FMEA Severity Score | Description |
|---------------|---------------------|-------------|
| C (Critical) | 9-10 (mapped: 10) | Potential fatality or permanent disability |
| H (High) | 7-8 (mapped: 8) | Serious injury requiring medical treatment |
| M (Medium) | 4-6 (mapped: 5) | Minor injury, first aid, or reversible health effect |
| L (Low) | 1-3 (mapped: 2) | Negligible injury, near-miss with no contact |

The mapping is stored as `FrontierCard.SEVERITY_TO_FMEA = {"C": 10, "H": 8, "M": 5, "L": 2}` and applied by `FrontierCard.severity_to_fmea_score()`.

#### **4.3.3 Card-to-FMEA Pipeline**

<!-- assert: process_card_to_fmea creates FMEARow records from at-risk observations, marks card as processed, and handles 5S cross-feed items | check=saf-card-to-fmea -->
<!-- impl: safety/models.py:process_card_to_fmea -->

<!-- rule: mandatory -->

Pipeline steps:

1. Iterate `card.safety_observations` — filter to `at_risk=True` entries only
2. For each at-risk observation, create `FMEARow`:
   - `process_step`: `[ZoneName] category`
   - `failure_mode`: observation item text
   - `failure_effect`: notes (or default description)
   - `severity`: mapped via `SEVERITY_TO_FMEA`
   - `occurrence`: default 5 (refined during weekly FMEA review)
   - `detection`: default 5 (refined during weekly FMEA review)
   - `current_controls`: `"Frontier Card audit ({date})"`
   - `recommended_action`: hierarchy of controls reference
   - `responsible`: auditor name
3. If `has_safety_crossfeed=True` and `crossfeed_notes` exist, create additional FMEA row for the 5S cross-feed
4. Mark card: `is_processed=True`, `processed_at=now()`, `processed_by=user`
5. Store created row UUIDs in `card.fmea_rows_created`
6. Return list of created row IDs

### **4.4 Risk Control — Hierarchy of Controls**

<!-- rule: mandatory -->

All recommended actions from FMEA rows MUST reference the hierarchy of controls (E-S-E-A-P), prioritizing higher-order controls:

| Priority | Control Type | Description | Example |
|----------|-------------|-------------|---------|
| 1 | **Elimination** | Remove the hazard entirely | Remove the elevated work requirement |
| 2 | **Substitution** | Replace with less hazardous alternative | Substitute solvent-based cleaner with water-based |
| 3 | **Engineering** | Isolate people from the hazard | Install machine guarding, ventilation, barriers |
| 4 | **Administrative** | Change the way people work | SOPs, training, signage, job rotation |
| 5 | **PPE** | Protect the individual worker | Safety glasses, hearing protection, fall arrest |

Lower-order controls (Administrative, PPE) are acceptable as interim measures but MUST NOT be treated as permanent solutions when higher-order controls are feasible.

### **4.5 5S Parallel Track with Cross-feed Rule**

<!-- assert: aggregate_five_s_pareto aggregates 5S tallies across Frontier Cards into Pareto-sorted data with cumulative percentages, requiring minimum 10 cards | check=saf-five-s-pareto -->
<!-- impl: safety/models.py:aggregate_five_s_pareto -->

<!-- rule: mandatory -->

The 5S track runs parallel to safety observations on every Frontier Card. 5S deficiencies are tallied by pillar (Sort, Set in Order, Shine, Standardize, Sustain) and aggregated site-wide for Pareto analysis.

**Cross-feed Rule:** When a 5S deficiency also carries a safety consequence (e.g., cluttered walkway creating trip hazard), the auditor sets `has_safety_crossfeed=True` and documents the safety implication in `crossfeed_notes`. The `process_card_to_fmea()` pipeline then creates an additional FMEA row for the cross-feed item with default severity 6 (Medium).

**Pareto Aggregation:**
- Requires minimum 10 cards with 5S tallies before generating Pareto
- Tallies are summed across all cards for the site
- Sorted descending by frequency (highest-deficiency pillar first)
- Cumulative percentage calculated for 80/20 analysis
- Drives standardization focus: address the top 1-2 pillars first

### **5.4 Operator Interaction**

<!-- rule: mandatory -->

Every Frontier Card audit SHOULD include an operator interaction. Three questions are asked:

1. **"What is your biggest safety concern in this area?"** → `operator_concern`
2. **"If you could change one thing to make this area safer, what would it be?"** → `operator_improvement`
3. **"Have you experienced or witnessed any near-misses here recently?"** → `operator_near_miss`

The operator's first name is recorded (`operator_name`). No additional PII is captured (see §13.3).

Operator interaction rate is tracked as a leading KPI (§10). Target: 100% of audits include operator engagement.

### **5.5 Immutability**

<!-- rule: mandatory -->

Frontier Cards are **append-only** after submission:

- Once `is_processed=True`, the card's `safety_observations` and `five_s_tallies` MUST NOT be modified
- Cards are never deleted — soft-delete is not implemented; historical integrity is paramount
- The `is_processed` flag, `processed_at`, `processed_by`, and `fmea_rows_created` fields are set exactly once during processing
- If an error is discovered in a submitted card, a new corrective card is created referencing the original

<!-- code: prohibited -->
```python
card = FrontierCard.objects.get(id=card_id)
card.safety_observations = []  # NEVER — destroys audit trail
card.delete()                  # NEVER — no deletion
```

<!-- code: correct -->
```python
# To correct an error, create a new card and reference the original
corrective_card = FrontierCard.objects.create(
    auditor=card.auditor, zone=card.zone, site=card.site,
    audit_date=date.today(),
    safety_observations=corrected_observations,
)
```

### **6.3 Scheduling Discipline**

<!-- rule: mandatory -->

| Parameter | Target | Rationale |
|-----------|--------|-----------|
| Minimum audits per supervisor per week | 2 | Ensures consistent hazard identification frequency |
| Zone rotation | Every supervisor covers every zone within 4 weeks | Prevents blind spots from familiarity |
| Completion rate target | ≥ 95% | Schedule adherence is a leading indicator |
| Ad-hoc audits | Permitted (assignment=null) | Encourages proactive observation beyond schedule |
| Missed assignment review | Required at weekly MOS | Root cause non-completion — workload, access, training |

### **6.4 D-S-O-I-R Audit Cycle**

The five-phase audit cycle governs each Frontier Card observation:

| Phase | Activity | Duration | Output |
|-------|----------|----------|--------|
| **Define** | Area Manager publishes weekly schedule with zone assignments | Monday | AuditSchedule + AuditAssignments |
| **Schedule** | Auditors receive assignments and plan routes | Mon-Tue | Assignment status → `assigned` |
| **Observe** | Auditor walks the zone using Frontier Card (20 safety items + 25 5S items) | Per target_date | FrontierCard record |
| **Intervene** | Immediate corrective action for Critical/High findings; coaching for Medium/Low | During audit | Verbal correction, STOP work authority for imminent danger |
| **Review** | Weekly FMEA review processes unprocessed cards into risk register | Friday | FMEA rows, RPN scores, recommended actions |

## **8. NOTIFICATION & ESCALATION**

### **8.1 Critical/High Findings Auto-Notify**

<!-- assert: Card submission with Critical or High severity findings triggers notification to all site admins and members via NTF-001 notify() helper | check=saf-high-severity-notify -->
<!-- impl: safety/views.py:_notify_high_severity -->

<!-- rule: mandatory -->

When a Frontier Card is submitted with `highest_severity` of `C` (Critical) or `H` (High):

1. Query `SiteAccess` for all users with `admin` or `member` role on the card's site
2. For each user, call `notify()` (NTF-001 §7.1) with:
   - `notification_type="system"`
   - `title`: `"CRITICAL safety finding: {zone_name}"` or `"HIGH safety finding: {zone_name}"`
   - `message`: Auditor name, date, at-risk count, zone
   - `entity_type="frontier_card"`, `entity_id=card.id`
3. Failure to notify does NOT block card submission (try/except with logging)

### **8.2 Unprocessed Card Alerts**

<!-- rule: recommended -->

Cards pending FMEA entry for more than 24 hours SHOULD trigger an alert to the Safety/OpEx Manager. Implementation via `syn.sched` periodic task (future — not yet wired).

Alert criteria:
- `is_processed=False`
- `created_at < now() - 24h`
- `highest_severity in ("C", "H")` for immediate alert; `("M", "L")` for 72h alert

## **13. SECURITY**

### **13.1 Enterprise Tier Feature Gate**

<!-- assert: The safety feature flag in accounts/constants.py is True only for Enterprise tier | check=saf-enterprise-gate -->
<!-- impl: accounts/constants.py -->

<!-- rule: mandatory -->

The `"safety"` key in `TIER_FEATURES` is `True` only for Enterprise ($299/mo). All safety endpoints are wrapped with `@require_feature("safety")` which checks the user's subscription tier and returns HTTP 403 for non-Enterprise accounts.

### **13.2 Tenant Isolation**

<!-- assert: All safety queries are scoped to the user's tenant via get_accessible_sites and get_tenant, preventing cross-tenant data access | check=saf-tenant-isolation -->
<!-- impl: safety/views.py -->

<!-- rule: mandatory -->

Every view function:
1. Calls `get_tenant(request.user)` to resolve the Enterprise tenant
2. Scopes all queries via `site__tenant=tenant` or `get_accessible_sites(request.user, tenant)`
3. Returns HTTP 403 if no tenant is found

No cross-tenant data leakage is possible — all model queries are tenant-scoped before any data is returned.

### **13.3 PII in Card Observations**

<!-- rule: mandatory -->

Frontier Cards MUST NOT contain PII beyond the operator's first name:

- `operator_name`: first name only (no last name, no employee ID)
- `operator_concern`, `operator_improvement`, `operator_near_miss`: free text — auditors trained to record concerns about conditions, not complaints about individuals
- `safety_observations[].notes`: condition descriptions only, no names

<!-- code: prohibited -->
```python
card = FrontierCard.objects.create(
    operator_name="John Smith (EMP-4472)",  # NEVER — last name + ID
    operator_concern="Bob in shipping is always reckless",  # NEVER — names individuals
)
```

<!-- code: correct -->
```python
card = FrontierCard.objects.create(
    operator_name="John",  # First name only
    operator_concern="Forklift traffic in shipping aisle creates pedestrian exposure",  # Condition-focused
)
```
