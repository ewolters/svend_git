## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

This standard defines the **quality contract** for all DSW analysis output. It is subordinate to QUAL-001 (Output Quality Assurance), which defines the overarching three-pillar framework (mathematical correctness, data quality, output coherence), calibration system, and bounds checking. DSW-002 specializes QUAL-001 for the DSW pipeline: exact field shapes, type constraints, depth requirements, chart styling, and rendering expectations. Where DSW-001 section 8 defines the architectural pipeline (post-processor, registry, dispatch), this standard specifies what that pipeline must produce. It exists so that "the key exists" is never confused with "the output is correct."

### **1.2 Scope**

Applies to every analysis result dict returned by `standardize_output()` in `agents_api/dsw/standardize.py`. This covers all 230+ analyses across all modules (stats, bayesian, spc, ml, reliability, simulation, viz, d_type, causal, drift, anytime, bayes_msa, pbs, ishap, quality_econ).

### **1.3 Relationship to DSW-001**

DSW-001 section 8 defines the post-processing pipeline architecture. This standard defines the quality assertions that pipeline must satisfy. Tests linked here verify output quality; tests in DSW-001 verify pipeline structure.

## **3. TERMINOLOGY**

| Term | Definition |
|------|-----------|
| **Canonical output** | The result dict after `standardize_output()` has run — the shape that reaches the frontend |
| **Narrative dict** | `{verdict, body, next_steps, chart_guidance}` — the structured interpretation block |
| **Education entry** | `{title, content}` — collapsible educational context for the analysis |
| **Evidence grade** | One of `"Strong"`, `"Moderate"`, `"Weak"`, `"Inconclusive"` (title-case) — synthesized from p-value, BF10, and effect magnitude |
| **Bayesian shadow** | Bayesian reanalysis of a frequentist result, keyed by shadow_type in the registry |
| **What-if schema** | `{type, parameters[], endpoint, recompute_fields}` — unified interactive exploration format |
| **Color tag** | `<<COLOR:tag>>...<</COLOR>>` markup in summary strings, stripped for narrative and guide_observation |
| **SVEND_COLORS** | 10-color categorical palette defined in `common.py`, used for all chart traces |
| **Trace builder** | Pre-styled functions in `chart_defaults.py` that return Plotly trace dicts |

### **5.4 Auto-Generation**

When an analysis provides `summary` but no `narrative`, the post-processor generates a fallback narrative:
- First non-empty line of summary (after stripping color tags, HTML tags, and box-drawing characters) becomes `verdict`
- Remaining lines become `body`
- `next_steps` and `chart_guidance` default to `""`
- Lines that are purely separator characters (`─`, `═`, `---`, `===`) are dropped

## **11. ANTI-PATTERNS**

### **11.1 String Narrative**

<!-- code: prohibited -->

```python
# WRONG: returning narrative as plain string
result["narrative"] = "The mean is significantly different."
```

<!-- code: correct -->

```python
# RIGHT: returning narrative as structured dict
result["narrative"] = {
    "verdict": "The mean is significantly different from the target.",
    "body": "The sample mean (52.3) exceeds the hypothesized value...",
    "next_steps": "Investigate what is causing the shift.",
    "chart_guidance": "The histogram shows the distribution relative to the target.",
}
```

Note: The post-processor normalizes string narratives as a safety net, but analyses SHOULD return the dict form directly.

### **11.2 Hardcoded Colors**

<!-- code: prohibited -->

```python
# WRONG: hardcoded hex colors
trace["marker"]["color"] = "#ff0000"
```

<!-- code: correct -->

```python
# RIGHT: use palette or semantic colors
from agents_api.dsw.common import SVEND_COLORS, COLOR_BAD
trace["marker"]["color"] = SVEND_COLORS[0]
trace["marker"]["color"] = COLOR_BAD  # for semantic meaning
```

### **11.3 Missing Education**

<!-- code: prohibited -->

```python
# WRONG: analysis returns without education, no entry in education.py
return {"summary": "...", "plots": [...]}
```

<!-- code: correct -->

```python
# RIGHT: add entry to education.py for every new analysis
# In education.py:
EDUCATION_CONTENT[("stats", "new_test")] = {
    "title": "Understanding the New Test",
    "content": "<dl><dt>What is it?</dt><dd>...</dd></dl>",
}
```

### **11.4 Empty What-If Parameters**

<!-- code: prohibited -->

```python
# WRONG: tier 1 analysis with no actual parameters
result["what_if"] = {"type": "slider", "parameters": [], ...}
```

Tier 1 analyses with legacy `power_explorer` or `what_if_data` SHOULD produce non-empty parameter lists. Empty parameters mean the what-if explorer renders but has nothing to interact with.

## **17. ACCEPTANCE CRITERIA**

The output quality standard is met when:

1. All 230+ registered analyses produce correct field types after `standardize_output()`
2. No analysis returns a string narrative after post-processing
3. All narratives have verdict ≥10 characters
4. All guide_observations are 10-300 characters with no color tags
5. All charts have height set (300 or 350), transparent backgrounds, SVEND_COLORS palette
6. All education entries have title ≥15 chars, content ≥200 chars, `<dl>` structure
7. All p-value analyses have evidence_grade in {"Strong", "Moderate", "Weak", "Inconclusive"}
8. Effect classification thresholds match the documented values
9. All what-if parameters satisfy min < max, step > 0, value in [min, max]
10. All diagnostic statuses are in {"pass", "warn", "fail", "info"}
11. Integration tests pass for all engine modules through the full pipeline
12. Registry entries have complete metadata with valid types (§12)
13. Bounds validation catches NaN, Inf, out-of-range values (§13)
14. SPC capability endpoint applies standardize_output (§14.2)
15. Frontend rendering contract verified for all output shapes (§16)
