## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

TST-001 defines the test framework, organization, fixture patterns, and conventions for all projects under the Kjerne standards umbrella. Tests are the verification layer for all other standards.

**Core Principle:**

> A standard without a test is a suggestion. A test without a standard is noise.
> Tests verify that standards are enforced, not that code runs.

### **1.2 Scope**

**Applies to:**
- All `*test*.py` files in any project adopting this standard
- Test fixtures and helper functions
- API test patterns
- Compliance check verification

**Does NOT apply to:**
- Manual QA procedures
- Load testing or performance benchmarks
- Third-party library tests

### **1.3 Cross-Project Applicability**

This standard is a **chassis standard** — it applies uniformly to all projects that adopt the Kjerne standards library. Each project maintains its own test files but follows the same framework, fixture patterns, and enforcement rules.

| Project | Test Root | Compliance Infrastructure |
|---------|-----------|--------------------------|
| **Svend** | `~/kjerne/services/svend/web/` | `syn/audit/` — full check suite |
| **Gemba Exchange** | `~/gemba_exchange/web/` | `syn/audit/` — subset (4 daily critical) |

When updating this standard, verify changes are compatible with all adopting projects. Project-specific test inventories belong in each project's own documentation (MAP-001 or CLAUDE.md), not here.

**Naming conventions:**

| Pattern | Usage | Example |
|---------|-------|---------|
| `<app>/tests.py` | App-level tests | `api/tests.py`, `core/tests.py` |
| `<app>/tests/test_<name>.py` | Split test modules | `syn/audit/tests/test_syslog.py` |
| `<app>/<standard>_tests.py` | Standards-linked tests | `agents_api/qms_phase4_tests.py` |
| `syn/audit/tests/test_<standard>.py` | Compliance tests | `test_compliance_system.py` |

**Inventory tracking:**

Test file inventories go stale in standards documents. Current inventories are maintained by:
- The compliance runner (`run_compliance --standards`) — tracks all assertion-linked tests
- MAP-001 architecture map — tracks file counts per app
- Each standard's own `<!-- test: -->` tags — canonical link from assertion to test

### **5.2 Sparse Areas**

Each project SHOULD track sparse test coverage areas in its own documentation (CLAUDE.md or project-level notes).

<!-- code: correct -->
```python
from rest_framework.test import APIClient

class MyEndpointTest(TestCase):
    def setUp(self):
        self.client = APIClient()
        self.user = _make_user("test@example.com", tier=Tier.PRO)
        self.client.force_authenticate(self.user)

    def test_create(self):
        res = self.client.post("/api/endpoint/", {"key": "value"}, format="json")
        self.assertEqual(res.status_code, 201)
        data = res.json()
        self.assertIn("id", data)
```

### **7.2 SSL Override**

<!-- assert: Tests use @SECURE_OFF to disable SSL redirect | check=tst-ssl-override -->
<!-- impl: api/tests.py -->
<!-- test: api.tests.HealthCheckTest.test_health_returns_ok -->

`@SECURE_OFF` wraps `@override_settings(SECURE_SSL_REDIRECT=False)` — required because Django's `SECURE_SSL_REDIRECT=True` in production settings would 301 all HTTP test client requests.

### **7.3 Mocking**

Standard library `unittest.mock`:

```python

# Throttle bypass for registration tests
p = patch("api.views.RegistrationThrottle.allow_request", return_value=True)
p.start()
self.addCleanup(p.stop)
```

`freezegun` for time-dependent tests (key expiry, token TTLs).

## **10. STANDARDS-LINKED TESTING STRATEGY**

### **10.1 Core Principle: Standards = Spec, Tests = Proof**

<!-- assert: Standards use assert/impl/test tags to link claims to executable proof | check=tst-standards-linked -->
<!-- impl: syn/audit/standards.py:parse_standard -->
<!-- test: syn.audit.tests.test_compliance_system.ParseAllStandardsTest.test_discovers_standards_files -->

The architecture is a triad with standards in the middle:

```
CODE / FEATURE SURFACE / INFRA  ←→  STANDARD  ←→  TEST
```

Code implements what the standard requires. Tests prove the standard is met. The standard is the single source of truth for both sides — it defines what must exist (impl tags) and how to verify it (test tags). Neither code nor tests reference each other directly; they connect through the standard.

Every testable claim in a standard is expressed as:

1. `<!-- assert: [claim] | check=[id] -->` — The testable specification
2. `<!-- impl: [path:symbol] -->` — The implementation under test
3. `<!-- test: [module.Class.method] -->` — The executable proof

The compliance runner (`run_compliance --standards`) is the CI/CD pipeline. It parses every standard, verifies every impl exists, and executes every linked test.

### **10.2 Three-Layer Verification**

<!-- assert: Compliance system operates three verification layers | check=tst-three-layers -->
<!-- impl: syn/audit/compliance.py:check_standards_compliance -->
<!-- impl: syn/audit/standards.py:verify_assertion -->
<!-- test: syn.audit.tests.test_compliance_system.AllChecksRegistryTest.test_expected_checks_registered -->

| Layer | What | How | When |
|-------|------|-----|------|
| **Standards assertions** | Every assert tag verified | Parser + AST + test execution | Daily via `standards_compliance` check |
| **Infrastructure checks** | Runtime/config compliance | All checks registered in ALL_CHECKS | Daily critical + weekday rotation |
| **Drift detection** | Violations tracked as DriftViolation records | Signature-based dedup, SLA enforcement | Automatic on each compliance run |

### **10.3 Test Organization**

Standards-linked test files follow a consistent pattern: each standard owns its test references via `<!-- test: -->` tags, and the compliance runner resolves them. This is the canonical mapping — not a static table in TST-001.

**Chassis tests** (shared infrastructure, live in `syn/audit/tests/`):
- Named `test_<standard_topic>.py` — one file per standard or standard group
- Cover: AUD-001, CMP-001, CHG-001, DOC-001, ERR-001, FE-001, LOG-001, OPS-001, SCH-001, XRF-001, etc.

**Project-specific tests** (live in each project's app directories):
- Named `<app>/tests.py` or `<app>/<feature>_tests.py`
- Cover standards that have project-specific implementations (QMS-001, TRN-001, QMS-002, etc.)

The compliance runner (`run_compliance --standards`) is the authoritative source for which tests map to which assertions. Static test-to-standard tables go stale (§11.5).

### **10.4 CI/CD Architecture**

The compliance runner is the **authoritative CI/CD pipeline**. It is the single source of truth for compliance status. Verification runs via:

- `python manage.py run_compliance --standards` — verify all assertion hooks + test existence
- `python manage.py run_compliance --standards --run-tests` — execute linked tests
- `python manage.py run_compliance --all` — run all registered infrastructure checks
- Daily cron via `syn.sched` at 02:00 UTC — automated enforcement

Drift violations are created automatically when assertions fail and auto-resolved when they pass again.

**Permitted layers beneath the compliance runner:**

| Layer | Purpose | Scope | Authoritative? |
|-------|---------|-------|----------------|
| **Pre-commit hooks** | Local development-time quality gate (linting, formatting) | Developer machine only | No — advisory |
| **GitHub Actions** | Automated lint + file-based compliance subset on push | Repository CI | No — early warning |
| **Compliance runner** | Full compliance verification (DB, ORM, all checks) | Production server | **Yes — single source of truth** |

Pre-commit hooks (STY-001 §11) and GitHub Actions (INIT-010 FEAT-079) are **development-time acceleration** — they catch issues earlier but do not replace the compliance runner. A green CI badge does not mean compliant; only the compliance runner determines compliance status. GitHub Actions MAY run a file-based subset of compliance checks that don't require ORM/DB access.

**Prohibited:**
- Treating GitHub Actions as the authoritative compliance gate
- Skipping compliance runner checks because CI passed
- Adding CI gates that block deployment without compliance runner backing

### **10.5 Coverage Target**

The goal is **96% assertion coverage** — meaning at least 96% of all standards assertions have linked tests that pass. Coverage is measured by the compliance runner and reported as the pass rate across all assertions.

Tests should use a variety of methods to achieve depth, not just breadth:

| Method | Purpose | Example |
|--------|---------|---------|
| **Structural** | Verify file/symbol existence, field presence, naming conventions | DOC-001 section numbering, DAT-001 UUID PKs |
| **Behavioral** | Verify runtime behavior through HTTP requests or function calls | FE-001 theme rendering, API-001 response format |
| **Scenario** | Verify multi-step workflows and state transitions | CHG-001 lifecycle (draft → approved → completed) |
| **Adversarial** | Verify rejection of invalid input, anti-patterns, prohibited code | XRF-001 prose reference detection, ERR-001 bare exception blocking |
| **Invariant** | Verify properties that must always hold across all standards | DOC-001 sequential numbering, XRF-001 no circular dependencies |

A passing test that only checks file existence is weaker than one that exercises the behavior the standard requires. Prefer behavioral and scenario tests where possible.

### **10.6 Existence Tests Do Not Qualify for Symbol Coverage**

<!-- assert: Symbol coverage tests must verify behavior, not just existence — bare import/callable checks are insufficient | check=tst-no-existence-only -->
<!-- impl: syn/audit/tests/test_symbol_governance.py -->
<!-- test: syn.audit.tests.test_symbol_governance.ComplianceCheckSymbolsTest.test_check_architecture_output_schema -->
<!-- test: syn.audit.tests.test_symbol_governance.AuditEventSymbolsTest.test_event_builders_are_callable -->
<!-- test: syn.audit.tests.test_symbol_governance.PermissionSymbolsTest.test_decorators_exist -->

<!-- rule: mandatory -->

**PROHIBITED:** Tests that only verify `assertIsNotNone(symbol)` or `assertTrue(callable(symbol))` without exercising any behavior. A functional test inherently proves existence — a separate existence check adds no value.

Tests linked to `<!-- impl: -->` hooks for symbol coverage MUST verify at least one of:
- **Output schema** — call the function and check the return structure
- **Side effects** — verify state changes (DB writes, context var mutations, signal connections)
- **Rejection behavior** — confirm incorrect input produces expected errors or status codes
- **Field/member presence** — for models and enums, verify specific fields or members exist
- **Roundtrip correctness** — for set/get pairs, verify the value survives the roundtrip

## **11. ANTI-PATTERNS**

### **11.1 Shared Mutable State**

<!-- code: prohibited -->
```python
class MyTest(TestCase):
    shared_user = None  # Class-level mutable state

    @classmethod
    def setUpClass(cls):
        cls.shared_user = _make_user("shared@example.com")
```

<!-- code: correct -->
```python
class MyTest(TestCase):
    def setUp(self):
        self.user = _make_user("test@example.com")
```

Create fresh data per test in `setUp()`. Class-level state leaks between tests.

### **11.2 Testing Implementation Instead of Behavior**

**PROHIBITED:** Testing private methods or internal data structures. Test the public API surface.

<!-- assert: Mock patches cleaned up via addCleanup or context managers | check=tst-mock-cleanup -->
<!-- impl: api/tests.py -->
<!-- test: api.tests.RegistrationTest.test_register_success -->

### **11.3 Ignoring Cleanup**

**PROHIBITED:** Leaving mock patches active. Always use `self.addCleanup(p.stop)` or context managers.

### **11.4 Hardcoded URLs**

<!-- code: prohibited -->
```python
res = self.client.get("http://localhost:8000/api/health/")
```

<!-- code: correct -->
```python
res = self.client.get("/api/health/")
```

Use relative paths. The test client handles host resolution.

### **11.5 Stale Inventories in Standards**

<!-- rule: mandatory -->

**PROHIBITED:** Embedding file counts, line counts, or test method totals in standards.

These go stale with every test addition. Standards define **rules** (what must be true). Inventories are computed dynamically by the compliance runner or tracked in architecture maps (MAP-001). If a count matters for enforcement, express it as a minimum threshold (e.g., "≥96% assertion coverage") rather than a snapshot.
