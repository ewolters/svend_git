## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

SCH-001 defines the architecture, patterns, and operational requirements for `syn.sched` — Kjerne's cognitive task scheduler. This is a purpose-built replacement for Celery that eliminates the broker dependency, integrates governance-aware priority scoring, and provides adaptive backpressure control.

**Core Principle:**

> A task scheduler that cannot explain why it ran a task, when, and what happened is not a scheduler — it's a liability.
> Every task execution is an auditable event with a governance rationale.

### **1.2 Scope**

**Applies to:**
- All files in `services/svend/web/syn/sched/`
- Task handler registration in `svend_tasks.py`
- Management commands that interact with the scheduler
- Any code that calls `CognitiveScheduler.submit()`

**Does NOT apply to:**
- Legacy `tempora/` (deprecated, kept for reference)
- External cron jobs outside Django
- One-off management commands not registered as tasks

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **Cognitive Task** | A unit of work with governance metadata (urgency, confidence, risk). Stored in `CognitiveTask` model. |
| **Handler** | A registered Python callable that executes task logic. Stored in `TaskRegistry`. |
| **Resource Class** | Worker pool classification: CPU_BOUND, IO_BOUND, MIXED, LIGHTWEIGHT. Determines thread allocation. |
| **Backpressure** | Adaptive throttling based on system health (queue depth, DLQ size, CPU, memory). |
| **Temporal Reflex** | Automatic corrective action triggered by governance policy evaluation (pause, adjust priority, compensate). |
| **Dead Letter Queue** | Storage for tasks that exhausted all retry attempts. Requires manual triage. |
| **Circuit Breaker** | Per-service fault isolation. Opens after repeated failures, prevents cascade. |
| **Cascade** | Parent-child task spawning with depth and budget limits. |

## **4. ARCHITECTURE**

### **4.1 No-Broker Design**

<!-- assert: Scheduler operates without external message broker | check=sch-no-broker -->
<!-- impl: syn/sched/core.py -->

`syn.sched` uses an in-memory priority queue backed by PostgreSQL persistence. There is no Redis, RabbitMQ, or external broker.

| Component | Celery | syn.sched |
|-----------|--------|-----------|
| Broker | Redis/RabbitMQ | None — in-memory queue |
| Persistence | Message queue | PostgreSQL (CognitiveTask) |
| Execution | Separate worker processes | Embedded thread pool |
| Scheduling | Celery Beat (separate) | Integrated in scheduler |
| Priority | Numeric | Cognitive scoring with governance |

**Why:** Simpler operational footprint, tighter feedback loops for adaptive control, and medium-scale workloads do not need billion-event streaming.

### **4.2 Execution Stack**

<!-- assert: Task execution uses resource-class worker pools | check=sch-worker-pools -->
<!-- impl: syn/sched/execution/worker_pool.py -->
<!-- impl: syn/sched/execution/executor.py -->
<!-- impl: syn/sched/execution/resource_class.py -->
<!-- test: syn.audit.tests.test_scheduler.TaskStateTest.test_terminal_states -->
<!-- test: syn.audit.tests.test_scheduler.TaskStateTest.test_non_terminal_states -->
<!-- test: syn.audit.tests.test_scheduler.TaskStateTest.test_active_states -->
<!-- test: syn.audit.tests.test_scheduler.TaskStateTest.test_valid_transitions_defined -->

```
CognitiveScheduler
  +-- BackpressureController (monitors health, throttles)
  +-- TemporalController (adaptive control loop)
  +-- WorkerPool (4 resource-class groups)
      +-- WorkerGroup[CPU_BOUND]
      +-- WorkerGroup[IO_BOUND]
      +-- WorkerGroup[MIXED]
      +-- WorkerGroup[LIGHTWEIGHT]
```

Execution flow: submit -> CognitiveTask created in DB -> enqueued to ExecutionQueue (in-memory heap) -> WorkerPool dispatches to resource-class worker -> TaskExecutor runs handler with timeout -> result written to DB + event emitted.

## **6. COGNITIVE PRIORITY SCORING**

### **6.1 Priority Formula**

<!-- assert: Task priority uses cognitive scoring formula | check=sch-cognitive-score -->
<!-- impl: syn/sched/core.py -->
<!-- test: syn.audit.tests.test_scheduler.CognitiveScoringTest.test_priority_score_formula -->
<!-- test: syn.audit.tests.test_scheduler.CognitiveScoringTest.test_low_urgency_lowers_score -->
<!-- test: syn.audit.tests.test_scheduler.CognitiveScoringTest.test_high_risk_lowers_score -->
<!-- test: syn.audit.tests.test_scheduler.CognitiveScoringTest.test_root_correlation_defaults_to_correlation -->
<!-- test: syn.audit.tests.test_scheduler.CognitiveScoringTest.test_attempt_tracking -->
<!-- test: syn.audit.tests.test_scheduler.TaskPriorityTest.test_critical_highest_weight -->
<!-- test: syn.audit.tests.test_scheduler.TaskPriorityTest.test_batch_lowest_weight -->
<!-- test: syn.audit.tests.test_scheduler.TaskPriorityTest.test_weights_monotonically_decrease -->

```
P = (confidence x 0.3) + (urgency x 0.4) + ((1 - governance_risk) x 0.3)
```

| Factor | Range | Weight | Meaning |
|--------|-------|--------|---------|
| Confidence | 0.0-1.0 | 30% | How certain the task should execute |
| Urgency | 0.0-1.0 | 40% | Time sensitivity |
| Governance risk | 0.0-1.0 | 30% | Inverted — lower risk = higher priority |

This is not a simple numeric priority. Tasks carry governance context that affects scheduling decisions.

### **6.2 Queue Affinity**

Tasks route to queues: CRITICAL > CORE > BATCH. Queue type determines scheduling precedence before cognitive score is applied.

## **7. RETRY AND FAILURE HANDLING**

### **7.1 Retry Strategies**

<!-- assert: Failed tasks follow configured retry strategy before DLQ | check=sch-retry-dlq -->
<!-- impl: syn/sched/types.py -->
<!-- impl: syn/sched/models.py -->
<!-- test: syn.audit.tests.test_scheduler.RetryConfigTest.test_exponential_delay_increases -->
<!-- test: syn.audit.tests.test_scheduler.RetryConfigTest.test_none_strategy_zero_delay -->
<!-- test: syn.audit.tests.test_scheduler.RetryConfigTest.test_fixed_strategy_constant -->
<!-- test: syn.audit.tests.test_scheduler.RetryConfigTest.test_max_delay_cap -->

| Strategy | Behavior | Default Config |
|----------|----------|----------------|
| EXPONENTIAL | min(base x 2^attempt, max_delay) + jitter | 5s base, 300s max |
| LINEAR | delay_increment x attempt | 5s increment |
| FIXED | Constant delay between retries | 5s |
| IMMEDIATE | No delay | 0s |
| NONE | Skip retries, go directly to DLQ | — |

Named presets: `transient` (5 attempts, exponential), `permanent` (1 attempt, no retry), `rate_limited` (10 attempts, 60s base).

### **7.2 Dead Letter Queue**

Tasks that exhaust retries move to `DeadLetterEntry` for manual triage. Permanent errors (ValueError, TypeError) skip retries entirely.

<!-- impl: syn/sched/backpressure/throttle.py -->

| Level | Behavior |
|-------|----------|
| NONE | Accept all tasks normally |
| LIGHT | Batch-only, drop low-priority |
| MODERATE | Critical-only, delay batch |
| HEAVY | Queue drains, reduce acceptance |

Health signals: queue depth, DLQ size, CPU utilization, memory usage. Controller evaluates signals continuously and adjusts throttle level.

The schedule processor runs every 10 seconds on a background thread (`_schedule_loop`). It queries `Schedule.objects.filter(is_enabled=True, next_run_at__lte=now)` with `select_for_update(skip_locked=True)` for row-level locking. For each due schedule:

1. Check temporal pause (SCH-006)
2. Create task via `submit()`
3. Call `schedule.record_run()` to update metadata
4. Emit `scheduler.schedule.triggered` event

## **13. ANTI-PATTERNS**

### **13.1 Bypassing the Registry**

<!-- code: prohibited -->
```python
# Calling task handler directly without scheduler
from syn.sched.svend_tasks import send_email_handler
send_email_handler({"to": "user@example.com"}, None)
```

<!-- code: correct -->
```python
from syn.sched.core import get_scheduler
scheduler = get_scheduler()
scheduler.submit("api.send_email", payload={"to": "user@example.com"})
```

All task execution MUST go through the scheduler for audit, retry, and governance tracking.

### **13.2 Unbounded Cascades**

<!-- code: prohibited -->
```python
# Spawning child tasks without parent tracking
for item in large_list:
    scheduler.submit("process_item", payload={"id": item.id})
```

<!-- code: correct -->
```python
parent = scheduler.submit("batch_process", payload={"items": item_ids})
for item_id in item_ids:
    scheduler.submit("process_item", payload={"id": item_id}, parent_task=parent)
```

### **13.3 Ignoring Backpressure**

**PROHIBITED:** Catching `ThrottledError` and retrying immediately. Respect the backpressure signal — it exists for a reason.
