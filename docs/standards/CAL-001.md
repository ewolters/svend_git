## §1 SCOPE AND PURPOSE

### §1.1 Purpose

CAL-001 treats Svend's codebase as a calibrated measurement device. In metrology, a calibrated device has: known accuracy (traceable to a reference), measurement uncertainty (bounded error), calibration certificates (documented proof), recalibration schedule (periodic re-verification), traceability (chain to national standard), and out-of-tolerance detection (immediate drift alerts). This standard maps those concepts to software quality.

**Core Principle:** If you can't measure it, you can't improve it. If you can't prove it's correct, it isn't.

### §1.2 Scope

Applies to all code under `services/svend/web/`:
- All Python modules (`.py` files)
- All analysis engine functions (`agents_api/dsw/*.py`)
- All view endpoints (`*_views.py`)
- All models (`models.py`)
- All infrastructure (`syn/`, `core/`, `accounts/`)

### §1.3 Metrology Analogy

| Metrology Concept | Software Equivalent | Implementation |
|---|---|---|
| Calibration certificate | Monthly `CalibrationReport` | `generate_calibration_cert` command |
| Measurement uncertainty | Tolerance on golden file assertions | `assertAlmostEqual(delta=)` |
| Traceability | Test → assertion → standard → SOC 2 | `<!-- test: -->` → `<!-- assert: -->` chain |
| Out-of-tolerance (OOT) | Calibration case failure | `DriftViolation` record |
| Recalibration interval | Daily/weekly/monthly/quarterly | Compliance rotation |
| Reference standard | Known-answer dataset | Golden files in `agents_api/tests/golden/` |
| Calibration lab | Compliance runner + test suite | `run_compliance --all` + `manage.py test` |
| NIST traceability | Verified against R/Minitab/textbook | `test_stats_reference.py` |
| Calibration sticker | Dashboard badge (last cert date/result) | Internal dashboard Calibration tab |
| Drift | Previously correct analysis producing wrong output | `DriftViolation` auto-created |

## §3 TERMINOLOGY

| Term | Definition |
|---|---|
| **Calibration cycle** | A scheduled run of tests + coverage measurement + calibration cases, producing a `CalibrationReport` |
| **Coverage baseline** | The minimum overall line coverage %, stored in `CalibrationReport.ratchet_baseline` |
| **Coverage ratchet** | Mechanism ensuring coverage never decreases — each report becomes the new floor |
| **Golden file** | A JSON file containing known input data, analysis config, and exact expected output values |
| **Measurement uncertainty** | The tolerance band (delta) used in `assertAlmostEqual` for floating-point comparisons |
| **Out-of-tolerance (OOT)** | A calibration case that fails — output differs from expected beyond tolerance |
| **Recalibration schedule** | The frequency at which calibration cycles run (daily: 8 cases, weekly: coverage, monthly: certificate) |
| **Reference standard** | A known-answer dataset with analytically or textbook-proven correct results |
| **Calibration certificate** | A `CalibrationReport` with `is_certificate=True`, generated monthly, containing full system state |
| **Complexity budget** | The maximum lines × cyclomatic complexity product allowed per module |
| **Smoke test** | Minimal test proving an endpoint responds correctly to auth + valid input + invalid input |
| **Module tier** | Risk classification (1-4) determining coverage targets and calibration requirements |

Thresholds are aspirational targets approached via ratchet:

| Metric | Phase 1 | Phase 2 | Phase 3 | Phase 4 |
|---|---|---|---|---|
| Overall coverage | Measured | 15% | 30% | 40%+ |
| Tier 1 coverage | Measured | 40% | 60% | 70%+ |
| Tier 2 coverage | Measured | 25% | 40% | 50%+ |
| Tier 3 coverage | Measured | 10% | 20% | 30%+ |
| Tier 4 coverage | Measured | 5% | 10% | 20%+ |

The ratchet mechanism: each `CalibrationReport` stores `ratchet_baseline`. The compliance check `cal-coverage-ratchet` verifies `current_coverage >= ratchet_baseline`. Coverage can only go up.

### §5.4 Coverage Exclusions

Lines excluded from coverage measurement (via `# pragma: no cover`):
- `__repr__` methods (display-only)
- `if __name__ == "__main__"` blocks (CLI entry points)
- `raise NotImplementedError` (abstract methods)
- `except ImportError` (optional dependency handling)
- `if TYPE_CHECKING` (type hint imports)

Everything else counts. No blanket exclusions for "hard to test" code.

**File format:** Each golden file is a JSON document:
```json

  "description": "One-sample t-test: known N(100,15) n=50 vs mu=95",

  "config": {"var1": "x", "mu": 95, "conf": 95},
  "expected": {
    "statistics.p_value": {"value": 0.0312, "tolerance": 0.005},
    "statistics.t_statistic": {"value": 2.21, "tolerance": 0.1},
    "statistics.cohens_d": {"value": 0.333, "tolerance": 0.05}

}
```

**Golden file naming:** `{analysis_type}_{analysis_id}_{scenario}.json`

### §6.2 Tolerance Bands

| Statistic Type | Tolerance (delta) | Rationale |
|---|---|---|
| p-value | 0.005 | Sufficient for decision boundary (0.05) |
| Effect size (Cohen's d, η², V) | 0.05 | Within conventional interpretation bands |
| R² | 0.02 | Regression accuracy |
| Confidence interval bounds | 1.0 | Wider tolerance for interval endpoints |
| Bayes Factor (BF₁₀) | 20% of value | Log-scale metric, proportional tolerance |
| Correlation (r) | 0.02 | Tight for directional accuracy |
| Control limits (UCL/LCL) | 0.5 | SPC chart precision |
| Capability indices (Cp/Cpk) | 0.05 | Quality decision boundary (1.33) |

## §9 REGRESSION PREVENTION

### §9.1 No-Regression Rule

<!-- assert: Coverage percentage never decreases between calibration cycles | check=cal-no-regression -->
<!-- impl: syn/audit/models.py:CalibrationReport -->
<!-- test: syn.audit.tests.test_calibration.NoRegressionTest.test_ratchet_enforced -->

Once a `CalibrationReport` records a coverage percentage, that becomes the floor. The `check_calibration_coverage` compliance check compares current coverage against the stored baseline. If current < baseline, the check FAILs.

### §9.2 Golden File Immutability

Golden files must not be modified to make failing tests pass. If a golden file test fails:
1. Investigate whether the analysis function changed
2. If the analysis was improved (corrected), update the golden file AND document the change in a CHG-001 CR
3. If the analysis regressed (got worse), fix the analysis, not the golden file
4. Every golden file modification must be justified in the commit message

## §11 CALIBRATION REPORTS

### §11.1 CalibrationReport Model

```python
class CalibrationReport(models.Model):
    """Point-in-time calibration snapshot."""
    date = models.DateField()
    overall_coverage = models.FloatField(null=True)
    tier1_coverage = models.FloatField(null=True)
    tier2_coverage = models.FloatField(null=True)
    tier3_coverage = models.FloatField(null=True)
    tier4_coverage = models.FloatField(null=True)
    calibration_pass_rate = models.FloatField(null=True)
    calibration_cases_run = models.IntegerField(default=0)
```

## §12 ANTI-PATTERNS

### §12.1 Coverage Gaming

**PROHIBITED:** Writing tests that execute code without verifying behavior, solely to increase coverage numbers.

### §12.2 Tolerance Widening

**PROHIBITED:** Widening golden file tolerances to make failing tests pass. If a test fails within its tolerance band, the analysis function has a bug.

### §12.3 Skipping Calibration

**PROHIBITED:** Disabling, skipping, or reducing calibration cycles. The daily 8-case rotation is the minimum.
