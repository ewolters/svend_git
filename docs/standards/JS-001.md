## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

JS-001 codifies JavaScript implementation conventions for the Svend web application. It documents the calculator architecture, workbench patterns, simulator conventions, DOM manipulation patterns, event handling, async patterns, chart integration, modal lifecycle, data persistence, and toast notification system used across 20,000+ lines of inline JavaScript in 70+ templates.

**Core Principle:**

> FE-001 says *what* (no framework, CSRF, API pattern). JS-001 says *how* (naming, state, events, architecture). Together they govern the full frontend.

### **1.2 Scope**

**Applies to:**
- All inline `<script>` blocks in `services/svend/web/templates/`
- All static JavaScript files in `services/svend/web/static/js/`
- All JavaScript functions, variables, event handlers, and DOM operations

**Does NOT apply to:**
- Python code (see STY-001)
- CSS / HTML structure (see FE-001)
- Third-party CDN libraries (see FE-001 §7)
- Marketing site JavaScript (`services/svend/site/`)

### **1.3 Relationship to FE-001**

FE-001 §6 defines **policy-level** JavaScript rules:
- §6.1: No SPA framework
- §6.2: CSRF auto-injection
- §6.3: API call pattern
- §6.4: Global utilities (SvendTheme, SvendChart, svToast, svendTrack)
- §6.5: 403 upgrade interceptor

JS-001 defines **implementation-level** conventions: naming, architecture patterns, state management, event handling, calculator/simulator/workbench-specific patterns, and anti-patterns. FE-001 policy rules are **non-negotiable**; JS-001 conventions apply to all new code and are enforced on modified code.

### **5.3 Calculator Function Pattern**

<!-- assert: Calculator functions follow input-calc-output-chart-publish pattern | check=js-calc-pattern -->
<!-- impl: templates/calculators.html -->
<!-- test: syn.audit.tests.test_javascript.CalculatorArchitectureTest.test_calc_function_pattern -->
<!-- test: syn.audit.tests.test_javascript.FMEATest.test_rpn_calculation -->

Every calculator function follows this 7-step pattern:

```javascript
function calcTakt() {
    // 1. INPUT COLLECTION — parseFloat/parseInt with fallback to 0
    const available = parseFloat(document.getElementById('takt-available').value) || 0;
    const demand = parseFloat(document.getElementById('takt-demand').value) || 1;

    // 2. CALCULATION — pure math, no DOM access
    const taktMin = (available - breaks) / demand;
    const taktSec = taktMin * 60;

    // 3. OUTPUT RENDERING — innerHTML with result-unit spans
    document.getElementById('takt-result').innerHTML =
        `${taktMin.toFixed(2)}<span class="calc-result-unit">min</span>`;

    // 4. DERIVATION DISPLAY — step-by-step math for transparency
    document.getElementById('takt-derivation-body').innerHTML = `
        <div class="step">
            <div class="step-num">Step 1: Net Available Time</div>
            Net Time = ${available} − ${breaks} = <strong>${net} min</strong>
        </div>`;

    // 5. CHART GENERATION — Plotly with theme-aware config
    Plotly.newPlot('takt-chart', [{ type: 'indicator', ... }],
        { paper_bgcolor: 'transparent', ... },
        { responsive: true, displayModeBar: false });

    // 6. DATA PUBLICATION — publish to SvendOps for other calcs
    SvendOps.publish('takt', taktSec, 'sec', 'Takt Time');

    // 7. NEXT STEPS — guide user to related calculators
    renderNextSteps('takt-next-steps', [
        { title: 'Staff the Line', calcId: 'rto' },
        { title: 'Balance Stations', calcId: 'yamazumi' },
    ]);
}
```

Not all calculators use all 7 steps. Steps 4 (derivation), 6 (publish), and 7 (next steps) are optional. Steps 1-3 and 5 are mandatory for any calculator that produces a result.

### **5.4 SvendOps Data Bus**

<!-- assert: SvendOps provides publish/pull/get for inter-calculator data flow | check=js-svendops -->
<!-- impl: templates/calculators.html -->
<!-- test: syn.audit.tests.test_javascript.CalculatorArchitectureTest.test_svendops_bus -->

`SvendOps` is the publish/subscribe bus for inter-calculator data transfer:

```javascript
const SvendOps = {
    values: {},

    publish(key, value, unit, source) {
        this.values[key] = { value, unit, source, timestamp: Date.now() };
        this.updateLinkIndicators();
    },

    pull(key, targetId) {
        const val = this.get(key);
        if (val !== null) {
            const el = document.getElementById(targetId);
            el.value = val;
            el.dispatchEvent(new Event('input'));
        }
    },

    get(key) {
        return this.values[key]?.value ?? null;
    },

    updateLinkIndicators() {
        document.querySelectorAll('[data-link-from]').forEach(el => {
            const key = el.dataset.linkFrom;
            if (this.values[key]) el.classList.add('has-link');
        });
    }
};
```

HTML integration:

```html
<button class="calc-pull-btn" data-link-from="takt"
    onclick="SvendOps.pull('takt', 'rto-takt')">Pull</button>
```

### **6.3 API Helpers**

<!-- assert: Workbench API calls use apiPost/apiCall helpers | check=js-api-helpers -->
<!-- impl: templates/workbench_new.html -->
<!-- test: syn.audit.tests.test_javascript.WorkbenchArchitectureTest.test_api_helpers -->
<!-- test: syn.audit.tests.test_javascript.AnalysisWorkbenchTest.test_csrf_handling -->
<!-- test: syn.audit.tests.test_javascript.InternalDashboardTest.test_api_fetch_helper -->
<!-- test: syn.audit.tests.test_javascript.HoshinTest.test_csrf_handling -->
<!-- test: syn.audit.tests.test_javascript.ModelsTest.test_csrf_handling -->
<!-- test: syn.audit.tests.test_javascript.ModelsTest.test_file_upload -->

```javascript
async function apiCall(endpoint, method = 'GET', data = null) {
    const opts = { method, credentials: 'include', headers: { 'X-CSRFToken': getCSRFToken() } };
    if (data) { opts.headers['Content-Type'] = 'application/json'; opts.body = JSON.stringify(data); }
    const r = await fetch(`/api/workbench/${endpoint}`, opts);
    if (r.status === 401 || r.redirected) return { error: 'auth', workbenches: [] };
    const ct = r.headers.get('content-type') || '';
    if (!ct.includes('application/json')) return { error: 'bad_response', workbenches: [] };
    return r.json();
}

async function apiPost(url, data, isFormData = false) {
    const opts = { method: 'POST', credentials: 'include', headers: { 'X-CSRFToken': getCSRFToken() } };
    if (isFormData) opts.body = data;
    else { opts.headers['Content-Type'] = 'application/json'; opts.body = JSON.stringify(data); }
    const r = await fetch(url, opts);
    if (r.status === 401) { window.location.href = '/login/?next=' + encodeURIComponent(window.location.pathname); }
    const ct = r.headers.get('content-type') || '';
    if (!ct.includes('application/json')) return { response: r, data: { error: 'Unexpected response' } };
    return { response: r, data: await r.json() };
}
```

Key features:
- CSRF auto-injection via `getCSRFToken()`
- 401 → redirect to login with `next` param
- Content-type validation before `.json()` parse
- FormData support for file uploads

### **7.2 Simulation State Machine**

<!-- assert: Simulation uses requestAnimationFrame tick loop with state machine | check=js-sim-state -->
<!-- impl: templates/simulator.html -->
<!-- test: syn.audit.tests.test_javascript.SimulatorArchitectureTest.test_state_machine -->

States: `idle` → `running` → `paused` → `scenario`

```javascript
let simState = 'idle';
let speedScale = 1.0;
let lastFrameTime = Date.now();

function tick() {
    const now = Date.now();
    const deltaTime = (now - lastFrameTime) * speedScale / 1000;
    lastFrameTime = now;

    // Advance simulation
    state.simTime += deltaTime;
    updateSimulationStep();
    renderCanvas();
    updateMetrics();

    if (simState === 'running') requestAnimationFrame(tick);
}

function startSimulation() { simState = 'running'; lastFrameTime = Date.now(); requestAnimationFrame(tick); }
function pauseSimulation() { simState = 'paused'; }
function resetSimulation() { simState = 'idle'; reinitializeSimulation(); renderCanvas(); }
```

### **7.3 Discrete Event Engine**

- **Jobs**: arrival, processing, transport, completion tracking
- **WIP limits**: configurable max buffer between stations
- **Bottleneck detection**: highest-utilization station
- **Starvation/blocking**: stations waiting for input or blocked by downstream

### **8.1 Element Access**

<!-- assert: DOM access uses getElementById or querySelector, not jQuery | check=js-dom-access -->

<!-- impl: templates/workbench_new.html -->

- `document.getElementById(id)` — for known, unique element IDs
- `document.querySelector(selector)` — for CSS selector-based lookup
- `document.querySelectorAll(selector)` — for multiple elements
- `element.closest(selector)` — for ancestor traversal

No jQuery. No `$()`. No `document.getElementsByClassName()`.

### **8.2 Dynamic Content**

- `innerHTML` — for static template strings (no user input)
- `document.createElement()` — for interactive elements with event listeners
- `textContent` — for user-supplied text (XSS-safe)

### **8.3 Class Toggling**

```javascript
element.classList.add('active');
element.classList.remove('active');
element.classList.toggle('open');
element.classList.contains('selected');
```

Never assign `element.className = '...'` directly (overwrites all classes).

### **8.4 Visibility**

<!-- assert: Visibility toggled via classList or display property, not visibility hidden | check=js-visibility -->
<!-- impl: templates/workbench_new.html -->
<!-- test: syn.audit.tests.test_javascript.DOMPatternTest.test_visibility_pattern -->

Two patterns for showing/hiding elements:

1. **CSS class toggle** (preferred): `.active` class with `display: none` / `display: flex|block`
2. **Direct style**: `element.style.display = 'none'` / `'flex'` / `'block'`

Do NOT use `visibility: hidden` (element still takes space).

### **8.5 XSS Prevention**

<!-- assert: User-supplied content uses textContent, not innerHTML | check=js-xss-prevention -->
<!-- impl: templates/base_app.html -->
<!-- test: syn.audit.tests.test_javascript.SecurityPatternTest.test_no_eval -->

row.innerHTML = `<td>${userInput}</td>`;  // XSS risk
```

<button onclick="showCalc('takt')">Takt Time</button>
<button onclick="calcOEE()">Calculate</button>
<button onclick="SvendOps.pull('takt', 'rto-takt')">Pull</button>
```

Do NOT use `onclick` for:
- Complex event handling (drag, resize, keyboard combinations)
- Handlers requiring cleanup (temporary listeners)
- Handlers with closure-captured state

### **9.2 addEventListener**

Use `addEventListener` for complex interactions:

```javascript
// Drag operations
handle.addEventListener('mousedown', (e) => {
    document.addEventListener('mousemove', onResize);
    document.addEventListener('mouseup', stopResize);
});

// Input events
distSel.addEventListener('change', () => recalculate());

// Tab initialization
document.addEventListener('DOMContentLoaded', () => { initialize(); });
```

### **9.3 Event Delegation**

For dynamic lists, use `querySelectorAll().forEach`:

```javascript
document.querySelectorAll('.ops-nav-item').forEach(el => {
    el.addEventListener('click', () => { showCalc(el.dataset.calcId); });
});
```

### **9.4 Keyboard Handling**

All dialogs/panels MUST close on Escape:

```javascript
document.addEventListener('keydown', function esc(e) {
    if (e.key === 'Escape') {
        closeModal();
        document.removeEventListener('keydown', esc);
    }
});
```

### **9.5 DOMContentLoaded**

Initialization code runs inside `DOMContentLoaded`:

```javascript
document.addEventListener('DOMContentLoaded', () => {
    initializeRibbon();
    loadProjects();
    hookAutoSave();
});
```

### **9.6 Cleanup**

Temporary event listeners (drag, resize) MUST be removed when done:

```javascript
function onMouseUp() {
    document.removeEventListener('mousemove', onDragMove);
    document.removeEventListener('mouseup', onMouseUp);
}
```

### **10.2 CSRF Token**

CSRF is handled two ways:
1. **Auto-injected** by FE-001 §6.2 monkey-patch (covers most cases)
2. **Explicit** via `getCSRFToken()` when building custom headers

```javascript
function getCSRFToken() {
    const m = document.cookie.match(/csrftoken=([^;]+)/);
    return m ? m[1] : '';
}
```

### **10.3 Response Validation**

Always check response status and content-type before parsing:

```javascript
const resp = await fetch(url, opts);
if (resp.status === 401) { redirectToLogin(); return; }
if (!resp.ok) throw new Error(`HTTP ${resp.status}`);
const ct = resp.headers.get('content-type') || '';
if (!ct.includes('application/json')) throw new Error('Non-JSON response');
const data = await resp.json();
```

### **10.4 Error Display**

- **User-facing**: `showToast(message)` or `svToast(message, 'error')`
- **Debug**: `console.error(context, error)` with meaningful context prefix
- **Both**: Show toast to user AND log to console

### **10.5 Loading States**

Long-running operations show loading indicators:

```javascript
showLoading(true, 'Running analysis...');
try {
    const result = await apiPost('/api/dsw/analyze/', data);
    renderResult(result);
} finally {
    showLoading(false);
}
```

### **13.2 Auto-Save**

<!-- assert: Auto-save uses function wrapping, not polling | check=js-autosave -->
<!-- impl: templates/calculators.html -->
<!-- test: syn.audit.tests.test_javascript.PersistencePatternTest.test_autosave_wrapping -->
<!-- test: syn.audit.tests.test_javascript.PersistencePatternTest.test_file_export_pattern -->
<!-- test: syn.audit.tests.test_javascript.AnalysisWorkbenchTest.test_localstorage_persistence -->

Auto-save wraps existing functions to trigger save after execution:

```javascript
function hookAutoSave() {
    const fns = ['updateSMED', 'addYamazumiStation', ...];
    fns.forEach(fnName => {
        if (typeof window[fnName] === 'function') {
            const original = window[fnName];
            window[fnName] = function(...args) {
                const result = original.apply(this, args);
                saveAutoState();
                return result;
            };
        }
    });
}
```

This pattern is preferred over polling (`setInterval`) because it saves only when state actually changes.

### **13.3 File Export**

Standard pattern for CSV/JSON download:

```javascript
function exportCSV(data, filename) {
    const blob = new Blob([data], { type: 'text/csv' });
    const a = document.createElement('a');
    a.href = URL.createObjectURL(blob);
    a.download = filename;
    a.click();
    URL.revokeObjectURL(a.href);
}
```

### **15.1 jQuery**

No jQuery. The codebase is 100% vanilla JavaScript per FE-001 §6.1.

<!-- code: prohibited -->
```javascript
$('#element').click(handler);    // No jQuery
$.ajax({ url: '...' });          // No jQuery AJAX
```

### **15.2 document.write**

<!-- code: prohibited -->
```javascript
document.write('<script>...');   // Blocks parsing, breaks streaming
```

### **15.3 eval / Function Constructor**

<!-- code: prohibited -->
```javascript
eval('code');                    // CSP violation, injection risk
new Function('return ' + str);  // Same risk
```

### **15.4 Hardcoded Colors in JS**

<!-- code: prohibited -->
```javascript
ctx.fillStyle = '#ff0000';       // Hardcoded — breaks themes
```

<!-- code: correct -->
```javascript
ctx.fillStyle = getComputedStyle(document.documentElement)
    .getPropertyValue('--accent-primary');
```

### **15.5 Synchronous XHR**

<!-- code: prohibited -->
```javascript
const xhr = new XMLHttpRequest();
xhr.open('GET', url, false);     // Synchronous — blocks UI thread
```

### **15.6 Global Namespace Pollution**

Avoid creating unnecessary globals. Group related state into objects:

<!-- code: correct -->
```javascript
const SvendOps = { values: {}, publish() {}, pull() {} };
```

<!-- code: prohibited -->
```javascript
// Scattered globals for the same concern
let svendOpsValues = {};
function svendOpsPublish() {}
function svendOpsPull() {}
```

### **15.7 innerHTML with User Input**

See §8.5. Never interpolate user-supplied strings into `innerHTML`.
