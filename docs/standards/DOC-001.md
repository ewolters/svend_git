## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

DOC-001 defines the canonical structure for all technical standards in the Kjerne repository. Every standard follows this template so that humans can find information predictably and tooling can validate compliance automatically.

**Core Principle:**

> Standards are living technical contracts. Structure is non-negotiable.
> A standard that can't be found, understood, and verified in 30 seconds is not a standard.

### **1.2 Scope**

**Applies to:**
- All technical standards in `docs/standards/`
- Architecture decision records (ADRs) promoted to standards
- Process and governance documents

**Does NOT apply to:**
- Informal documentation (READMEs, inline comments)
- Auto-generated API documentation
- Planning documents (`docs/planning/`)
- Vendor/integration notes (`docs/anthropic/`, `docs/stripe/`, etc.)

### **1.3 Document Lifecycle**

```
DRAFT → REVIEW → APPROVED → DEPRECATED
  ↓                 ↓            ↓
REJECTED        REVISION    SUPERSEDED
```

## **6. FORMATTING GUIDELINES**

### **6.1 Markdown Rules**

- Use `##` for section headers (§-level), `###` for subsections
- All tables use GitHub-Flavored Markdown pipe syntax
- Code blocks specify language: ` ```python `, ` ```yaml `, ` ```bash `
- Internal section refs use `§N` or `§N.M` notation (see XRF-001 §4)
- External standard refs use `CODE-NNN §N` notation (see XRF-001 §4)

## **7. MACHINE-READABLE HOOKS**

### **7.1 Purpose**

Standards are both human-readable contracts and machine-parseable compliance definitions. HTML comments embedded in standard markdown serve as hooks for the compliance parser (`syn/audit/standards.py`). The parser extracts assertions, links them to implementation code, and generates automated compliance checks.

**Core Principle:**

> If a standard says "X must happen," there must be a hook that lets a machine verify it does.
> Hooks are invisible to readers but first-class citizens for the compliance engine.

### **7.2 Hook Syntax**

All hooks use HTML comments with a keyword prefix and structured attributes:

```markdown
<!-- keyword: value | attr=value | attr=value -->
```

**Rules:**
- Hooks appear on their own line, immediately before the content they annotate
- Attributes are separated by ` | ` (space-pipe-space)
- Values containing spaces do not need quotes
- A hook's scope extends to the next hook of the same keyword, the next section header, or end of file

### **7.3 Hook Vocabulary**

| Hook | Purpose | Generates |
|------|---------|-----------|
| `<!-- assert: [claim] -->` | Testable compliance assertion | ComplianceCheck entry |
| `<!-- impl: [path] -->` | Links assertion to implementation file | Code-to-standard traceability |
| `<!-- check: [id] -->` | Named compliance check ID | Registerable check in ComplianceCheck |
| `<!-- test: [module.Class.method] -->` | Links assertion to executable test method | Test verification |
| `<!-- code: correct -->` | Marks following code block as canonical pattern | Positive test fixture |
| `<!-- code: prohibited -->` | Marks following code block as anti-pattern | Negative test fixture |
| `<!-- control: [framework] [id] -->` | Maps to external compliance control | ComplianceReport mapping |
| `<!-- table: [type] -->` | Identifies table semantics for parsing | Structured data extraction |
| `<!-- rule: mandatory -->` | Marks section as mandatory requirement | Enforcement flag |
| `<!-- rule: recommended -->` | Marks section as recommended practice | Advisory flag |
| `<!-- sla: [description] -->` | Service level agreement definition | SLA compliance tracking |

### **7.4 Hook Details**

#### **7.4.1 `assert` — Testable Claim**

Declares a verifiable assertion. The compliance parser generates a `ComplianceCheck` from it.

```markdown
<!-- assert: All SynaraError subclasses map to correct HTTP status | check=err-http-mapping -->
<!-- impl: syn/err/exceptions.py -->

| Exception | Category | HTTP |
|-----------|----------|------|
| `ValidationError` | VALIDATION | 400 |
```

**Attributes:**
- `check=[id]` — (optional) explicit check ID; auto-generated from assertion text if omitted

#### **7.4.2 `impl` — Implementation Link**

Points to the source file (or file:function) that implements the assertion. Multiple `impl` hooks can follow a single `assert`.

```markdown
<!-- assert: ErrorEnvelopeMiddleware wraps unhandled exceptions -->
<!-- impl: syn/api/middleware.py:ErrorEnvelopeMiddleware -->
<!-- impl: syn/err/types.py:ErrorEnvelope -->
```

**Format:** `path/to/file.py` or `path/to/file.py:ClassName` or `path/to/file.py:function_name`

#### **7.4.3 `test` — Linked Test Method**

Points to a Django test method that verifies the assertion. Multiple `test` hooks can follow a single `assert`. The compliance runner verifies the test exists (importable) and can optionally execute it via `--run-tests`.

Example (shown in FE-001.md):

    &lt;!-- assert: Theme system uses CSS custom properties | check=fe-themes --&gt;
    &lt;!-- impl: templates/base_app.html --&gt;
    &lt;!-- test: syn.audit.tests.test_frontend.ThemeSystemTest.test_six_themes_defined --&gt;
    &lt;!-- test: syn.audit.tests.test_frontend.ThemeSystemTest.test_css_variable_namespaces --&gt;

**Format:** `module.path.ClassName.method_name` — fully qualified Python dotted path. The compliance parser splits on the last two dots to resolve module, class, and method.

#### **7.4.4 `check` — Named Compliance Check**

Registers a compliance check by ID. Used when the check is broader than a single assertion.

```markdown
<!-- check: aud-chain-integrity | soc2=CC7.2 | nist=AU-9 -->
```

**Attributes:**
- `soc2=[control]` — SOC 2 control mapping
- `nist=[control]` — NIST SP 800-53 control mapping
- `iso=[clause]` — ISO 27001 clause mapping

#### **7.4.5 `code` — Code Block Classification**

Marks the immediately following code block as a correct pattern or a prohibited pattern. The parser extracts these as test fixtures.

```markdown
<!-- code: correct -->
```python
try:
    process()
except SynaraError:
    raise
except Exception as e:
    raise wrap_exception(e, correlation_id=corr_id)
```

<!-- code: prohibited -->
```python
try:
    process()
except:
    pass
```
```

#### **7.4.6 `control` — External Control Mapping**

Maps a section to an external compliance framework control.

```markdown
<!-- control: SOC 2 CC7.2 -->
<!-- control: NIST SP 800-53 AU-2 -->
<!-- control: ISO 27001:2022 A.12.7 -->
```

#### **7.4.7 `table` — Table Semantics**

Identifies table type for structured extraction.

```markdown
<!-- table: compliance-mapping -->
| Requirement | External Standard | Section |
```

**Known types:** `compliance-mapping`, `error-codes`, `field-definitions`, `acceptance-criteria`, `http-mapping`, `terminology`

#### **7.4.8 `sla` — Service Level Agreement**

Declares a measurable SLA target. The SLA parser (`syn/audit/standards.py:parse_sla_definitions`) extracts these into `SLADefinition` objects, and the `sla_compliance` check verifies them against operational data.

```markdown
<!-- sla: Platform availability | metric=availability | target=99.9% | window=monthly | severity=critical | soc2=CC9.2 -->
```

**Required Attributes:**
- `metric` — SLA type: `availability`, `response_time`, `durability`, `incident_response`, `compliance`, `change_velocity`
- `target` — Measurable target value with unit (e.g., `99.9%`, `500ms`, `24h`)
- `window` — Measurement period: `monthly`, `quarterly`, `annually`, `per_incident`
- `severity` — Business impact: `critical`, `high`, `medium`, `low`

**Optional Attributes:**
- `check` — Explicit SLA ID for deduplication (auto-generated from description if omitted)
- `measurement` — `automated`, `manual`, `derived` (default: `automated`)
- `soc2` — SOC 2 control mapping

Full specification: SLA-001 §4.

### **7.5 Parser Contract**

The compliance parser (`syn/audit/standards.py`) performs:

1. **Scan** all `.md` files in `docs/standards/` for hook comments
2. **Extract** structured data: assertions, implementation links, check IDs, test links, code fixtures, control mappings, SLA definitions
3. **Validate** implementation links point to existing files/functions (AST-based)
4. **Verify** linked test methods exist and are importable
5. **Register** checks in `ComplianceCheck` model (AUD-001 §7.1)
6. **Report** orphan hooks (assert without impl), broken impl links, missing test methods

### **7.6 Retrofitting Existing Standards**

When adding hooks to an existing standard:
1. Add `<!-- assert: ... -->` before each testable claim in domain sections
2. Add `<!-- impl: ... -->` linking assertions to source files
3. Add `<!-- code: correct/prohibited -->` before code blocks in anti-patterns sections
4. Add `<!-- control: ... -->` in the Compliance Mapping section
5. Add `<!-- table: ... -->` before structured tables

Hooks are additive — they do not change the human-readable content of the standard.

## **8. ANTI-PATTERNS**

### **8.1 Wall of Text**

**PROHIBITED:** Standards that read like essays without structure.

Every requirement MUST be in a numbered section with a clear title. If you can't express it as a table or a code block, it probably doesn't belong in a standard.

### **8.2 Aspirational Requirements**

**PROHIBITED:** Standards that describe a desired future state as if it exists.

If a capability isn't implemented, the standard MUST say so clearly (e.g., "Phase 2: Not yet implemented"). Do not document imaginary systems.

### **8.3 Duplicate Coverage**

**PROHIBITED:** Two standards covering the same domain.

Split by concern (DOC-001 structure, XRF-001 cross-refs) rather than duplicating. Use cross-references per XRF-001.

### **8.4 Missing Enforcement**

**PROHIBITED:** Standards with no acceptance criteria or compliance path.

Every standard MUST have measurable acceptance criteria (§N-1) and at least one enforcement mechanism (manual review, automated check, or test).
