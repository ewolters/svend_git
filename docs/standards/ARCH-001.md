## **1. SCOPE AND PURPOSE**

### 1.1 Purpose

Define the canonical directory structure, layer boundaries, module organization rules, growth limits, and deprecation patterns for the Kjerne/Svend Django monolith. This standard is the structural foundation that all other standards assume.

### 1.2 Scope

- All directories under `services/svend/web/`
- All apps in `INSTALLED_APPS`
- All infrastructure modules under `syn/`
- Root-level files in `web/`

Does NOT apply to: `staticfiles/` (generated by collectstatic), `media/` (user uploads), `__pycache__/` (runtime), migration files (auto-generated), `logs/` (runtime).

### 1.3 5S Alignment

This standard operationalizes STANDARD.md 5S methodology:
- **SEIRI (Sort)**: Prohibited directories list (Section 8)
- **SEITON (Set in Order)**: Canonical structure (Section 4)
- **SEISO (Shine)**: Growth boundaries (Section 7)
- **SEIKETSU (Standardize)**: Module organization rules (Section 6)
- **SHITSUKE (Sustain)**: Compliance check enforcement (Section 12)

## **3. TERMINOLOGY**

| Term | Definition |
|------|-----------|
| **Layer** | Conceptual boundary grouping modules by responsibility level |
| **Infrastructure layer** | `syn/` — shared services below application logic |
| **Domain layer** | `core/` — target data models |
| **Feature layer** | Top-level app directories (agents_api, chat, workbench, etc.) |
| **Integration layer** | `api/` — external-facing API and internal dashboard |
| **Prohibited directory** | A directory that MUST NOT exist, typically a removed legacy module |

Four layers with strict import direction:

```
Infrastructure (syn/) --> Domain (core/) --> Feature (apps) --> Integration (api/)
```

### 5.1 Import Rules

| Source | May Import From | Must NOT Import From |
|--------|----------------|---------------------|
| `syn/` | `syn/`, `core/`, `accounts/` (models only) | Feature apps |
| `core/` | `syn/`, `core/` | Feature apps, `api/` |
| Feature apps | `syn/`, `core/`, `accounts/` | Each other (prefer core/ interfaces) |
| `api/` | Any layer | — |

### 5.2 Exemptions

- `syn/audit/compliance.py` — introspects all layers by design (compliance checks query app models)
- `syn/audit/tests/` — test files may import from any layer
- `syn/audit/standards.py` — parses standard files across the repo
- `syn/sched/svend_tasks.py` — registers task handlers from all layers by design

Each Django app follows this file structure:

| File | Contents |
|------|----------|
| `models.py` or `models/*.py` | Django models only |
| `views.py` or `{feature}_views.py` | View functions/classes |
| `urls.py` | URL configuration |
| `apps.py` | Django AppConfig |
| `admin.py` | Admin site registration |
| `constants.py` | App-level constants |
| `utils.py` | Shared utilities within the app |
| `signals.py` | Django signals |
| `tasks.py` | Scheduled task handlers |
| `tests/__init__.py` | Test package init (required) |
| `tests/test_*.py` | Tests (canonical location) |

Cross-reference: STY-001 Section 4 (naming conventions).

| Metric | Warning | Failure |
|--------|---------|---------|
| Lines per `.py` file | > 2000 | > 3000 |
| `.py` files per app | > 20 | > 30 |

When thresholds are reached:
- `models.py` > 500 lines: split into `models/` package
- `views.py` > 500 lines: split into `{feature}_views.py` files
- App > 30 files: evaluate splitting into sub-apps

Known exceptions tracked in `.kjerne/DEBT.md` (e.g., dsw_views.py — P2 debt).

## **9. DEPRECATION PATTERN**

To sunset a module:

1. Create a CHG-001 ChangeRequest with `change_type="infrastructure"`
2. Redirect all imports to the replacement module
3. Remove from `INSTALLED_APPS` (if present)
4. Add the directory to Section 8 prohibited list
5. Remove the directory and contents
6. Update MAP-001 module map
7. Update STANDARD.md and CLAUDE.md directory trees

**Prohibited structures:**

1. Nested duplicate directories (e.g., `agents/agents/`)
2. Empty directories with no `__init__.py` or purpose
3. Python packages outside the canonical structure
4. Root files beyond the allowed set: `manage.py`, `pyproject.toml`, `gunicorn.conf.py`, `.gitignore`, `.env*`, `Caddyfile`
5. Infrastructure layer (`syn/`) importing from feature apps
6. Circular app dependencies
