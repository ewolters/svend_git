## **1. SCOPE AND PURPOSE**

### **1.1 Purpose**

QMS-001 defines the quality management system — the compliance, audit, and regulatory framework that operates ON the process graph (GRAPH-001). QMS tools interact with the graph and investigation system (LOOP-001) to detect, investigate, standardize, and verify. The graph is the persistent knowledge base; QMS is the quality control structure applied to graph operations.

> **Architectural note (Object 271):** QMS does not own FMEA or RCA. FMEA is a filtered view of the ProcessGraph (GRAPH-001 §7.3). RCA is an investigation tool that builds graph edges (GRAPH-001 §8). QMS owns compliance concerns: NCR tracking, CAPA documentation, audit scheduling, training matrix, management review. This section covers QMS-specific aspects of each tool; see GRAPH-001 and CANON-002 for graph and evidence integration.

**Core Principle:**

> Quality tools that don't connect are filing cabinets. The value is in the closed loop:
> Detect → Investigate → Solve → Measure → Improve. Every module feeds the next.
> The graph is the persistent substrate. The X-Matrix is the scoreboard.

### **1.2 Scope**

**Applies to:**
- FMEA (Failure Mode and Effects Analysis) — risk quantification
- RCA (Root Cause Analysis) — special cause causal investigation with AI critique
- Ishikawa (Fishbone Diagram) — common cause contributor mapping (6M categories)
- C&E Matrix (Cause & Effect) — weighted scoring of inputs vs outputs
- A3 (Toyota Problem Solving) — structured problem resolution
- VSM (Value Stream Mapping) — process analysis and improvement identification
- Hoshin Kanri — strategy deployment, project tracking, savings calculation
- X-Matrix — strategy-to-execution correlation and KPI rollup
- CAPA (Corrective and Preventive Action) — **DEPRECATED per LOOP-001 §5.** CAPA is now a generated compliance artifact from investigation data, not a standalone lifecycle. CAPAReport model retained for backward compatibility; new implementations use Investigation → Standardize → Verify loop.
- ActionItem — **SUPERSEDED per LOOP-001 §3.** Replaced by three mechanisms: Signals (event entry), Mode Transitions (system mechanics), Commitments (accountability contracts). Legacy ActionItem model retained for existing integrations.
- Evidence hooks — Bayesian integration from QMS findings

**Does NOT apply to:**
- DSW statistical analysis engine (separate concern — calculator, not QMS)
- SPC control charts and capability studies (feeds into QMS via KPI aggregation)
- Whiteboard collaboration (integration point, not QMS module)
- Knowledge graph and entity modeling (core platform, not QMS)

## **3. TERMINOLOGY**

| Term | Definition |
|------|------------|
| **RPN** | Risk Priority Number = Severity × Occurrence × Detection (1-1000 scale) |
| **Severity** | Impact of failure mode on end user or downstream process (1-10, AIAG scale) |
| **Occurrence** | Likelihood of failure cause happening (1-10, AIAG scale) |
| **Detection** | Ability of current controls to detect failure before reaching customer (1-10, AIAG scale) |
| **Causal Chain** | Ordered sequence of cause-effect claims in RCA, each individually critiqued |
| **Counterfactual Test** | "If we fix this cause, does the event stop happening?" — RCA validation criterion |
| **A3** | Single-page (A3 paper size) problem-solving document following Toyota PDCA structure |
| **PDCA** | Plan-Do-Check-Act cycle — the backbone of continuous improvement |
| **Takt Time** | Available production time ÷ customer demand — the heartbeat of a value stream |
| **PCE** | Process Cycle Efficiency = (value-add time ÷ total lead time) × 100% |
| **Kaizen Burst** | Identified improvement opportunity on a value stream map |
| **X-Matrix** | Hoshin Kanri deployment tool correlating strategic objectives, annual objectives, projects, and KPIs in four quadrants |
| **Hoshin Kanri** | "Compass management" — Japanese strategy deployment methodology |
| **Fiscal Year Rollover** | Process of copying X-Matrix structure to a new fiscal year |
| **Effective Actual** | Computed KPI value based on aggregation mode (sum, weighted_avg, latest, manual) |

FMEA quantifies risk. Each `FMEARow` records a failure mode with:
- **Severity** (1-10): impact on customer or downstream process
- **Occurrence** (1-10): frequency of the failure cause
- **Detection** (1-10): effectiveness of current controls at catching it
- **RPN** = S × O × D (auto-computed on save)

**Risk buckets:**

| RPN Range | Risk Level | Action |
|-----------|-----------|--------|
| ≥ 200 | Critical | Immediate corrective action required |
| 100-199 | High | Action plan within 30 days |
| 50-99 | Medium | Monitor and improve |
| < 50 | Low | Acceptable risk |

**Revised scoring:** After corrective action, revised S/O/D scores produce a post-action RPN (pRPN). The delta RPN → pRPN quantifies improvement effectiveness.

**Study Status** (`FMEA.Status`) — lifecycle of the FMEA document:

RCA investigates why failures happen. Each `RCASession` stores:
- **Event**: the observed failure or nonconformity
- **Chain**: ordered list of `{claim, critique, accepted, error_labels}` entries
- **Root cause**: the deepest systemic cause (validated by counterfactual test)
- **Countermeasure**: the proposed fix (evaluated for effectiveness level)

**State machine** (enforced on status transitions):

```
draft → investigating → root_cause_identified → verified → closed
                                                  ↘ draft (reopen with justification)
                                          closed ↗
```

| Transition | Condition |
|------------|-----------|
| draft → investigating | Always allowed |
| investigating → root_cause_identified | `root_cause` must be non-empty |
| root_cause_identified → verified | `countermeasure` must be non-empty |
| verified → closed | `evaluation` must be non-empty |
| verified/closed → draft | Requires `reopen_reason` |

<!-- test: agents_api.qms_wiring_tests.RCACritiqueTest.test_critique_endpoint_validates_input -->

**AI Critique Engine** (enterprise only):
- **Claim critique**: evaluates each causal step against 11 reasoning error patterns
- **Chain evaluation**: assesses overall causal logic and identifies gaps
- **Countermeasure critique**: stress-tests proposed fixes against 5-level hierarchy

**Reasoning error labels:**

| Label | Pattern | Example |
|-------|---------|---------|
| `HUMAN_ERROR_COP_OUT` | Blaming the operator instead of the system | "Operator didn't follow procedure" |
| `TRAINING_FALLACY` | Assuming training solves systemic issues | "Retrain the team" |
| `PROCEDURE_WORSHIP` | Writing a procedure instead of fixing the system | "Add a checklist step" |
| `HINDSIGHT_BIAS` | Claiming the cause was obvious after the fact | "They should have known" |
| `STOPPING_TOO_EARLY` | Accepting a proximate cause as root | "The sensor failed" (why?) |
| `SINGLE_CAUSE_FALLACY` | Assuming only one cause when multiple interact | "The only issue was..." |
| `COUNTERFACTUAL_FAILURE` | Proposed cause fails the "if we fix it" test | Fix this and it still happens |

**Countermeasure hierarchy** (highest → lowest effectiveness):

| Level | Type | Decay Risk | Example |
|-------|------|-----------|---------|
| 1 | Elimination | None | Remove the hazard entirely |
| 2 | Substitution | Low | Replace with less hazardous alternative |
| 3 | Engineering | Low | Physical barrier, automation, poka-yoke |
| 4 | Administrative | Medium | Procedures, training, scheduling |
| 5 | PPE/Warnings | High | Signs, alarms, protective equipment |

### **4.4 VSM — Value Stream Mapping**

VSM models the flow of material and information from supplier to customer:

| Component | Model | Key Fields |
|-----------|-------|-----------|
| Process steps | `process_steps[]` | cycle_time, changeover_time, uptime%, operators, batch_size |
| Inventory | `inventory[]` | quantity, days_of_supply, before_step_id |
| Information flow | `information_flow[]` | from_id, to_id, type (electronic/manual/schedule) |
| Material flow | `material_flow[]` | from_step_id, to_step_id, type (push/pull/fifo) |
| Kaizen bursts | `kaizen_bursts[]` | text, priority (high/medium/low), position |

<!-- assert: VSM auto-calculates total_lead_time, total_process_time, and PCE | check=qms-vsm-metrics -->
<!-- impl: agents_api/models.py:ValueStreamMap -->

**Computed metrics:**
- `total_lead_time` = Σ(inventory days_of_supply) + total_process_time (in days)
- `total_process_time` = Σ(process step cycle_times) in seconds
- `pce` = (total_process_time / total_lead_time) × 100%

**Bottleneck detection:** For parallel machines in a work center, effective CT = 1 / Σ(1/CT_i). Steps are flagged with `is_bottleneck` when effective CT exceeds takt time.

**Current ↔ Future state pairing:** Every VSM can be paired with a future-state copy. The `/compare/` endpoint calculates lead time reduction, process time reduction, PCE improvement, and inventory reduction between the two.

### **9.1 Disconnected Tools**

**PROHIBITED:** QMS modules that operate in isolation without evidence hooks or cross-linking.

evidence = Evidence.objects.create(
    project=hypothesis.project,
    source_type="data_analysis",
    source_tool="fmea",
    summary=f"FMEA failure mode: {row.failure_mode} (RPN {row.rpn})",
    confidence=confidence,
)
EvidenceLink.objects.create(
    hypothesis=hypothesis,

### **9.2 RPN Without Action**

**PROHIBITED:** Critical or high RPN scores (≥ 100) without a recommended action or linked ActionItem.

FMEA is a risk prioritization tool. If you score but don't act, you've quantified your negligence.

### **9.3 Root Cause = Human Error**

**PROHIBITED:** Accepting "human error," "operator mistake," or "training issue" as a root cause.

These are symptoms. The RCA critique engine flags `HUMAN_ERROR_COP_OUT` for exactly this reason. Dig into system conditions: why was the error possible? What barrier was missing?

### **9.4 A3 Without Data**

**PROHIBITED:** A3 reports where `current_condition` has no quantitative data.

The Plan phase requires measurement. "Things are bad" is not a current condition. Charts, control limits, defect rates, cycle times — something measurable.

### **9.5 VSM Without Future State**

**PROHIBITED:** Value stream maps that stay in current-state analysis forever.

A current state map without a paired future state is a diagnosis without a treatment plan. The `future_state()` endpoint exists for this reason.

### **9.6 Hoshin Without Monthly Actuals**

**PROHIBITED:** Hoshin projects with `hoshin_status=active` but no monthly_actuals entries beyond 60 days.

Active projects must report monthly. The dashboard shows YTD savings — if the data isn't entered, the X-Matrix rollup is wrong, and strategic objective tracking is fiction.
