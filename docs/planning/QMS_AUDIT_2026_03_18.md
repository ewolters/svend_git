## Bugs

### BUG-1: FMEA revised score validation gap

**Severity:** Medium
**Location:** `agents_api/models.py` FMEARow, line ~1000
**Description:** `FMEARow` enforces "all three revised scores or none" in `save()`, but API-level validation doesn't catch partial revised scores before hitting the model. POSTing `{"revised_severity": 5}` without `revised_occurrence` and `revised_detection` will trigger a model-level error that surfaces as a 500, not a clean 400 with a meaningful message.
**Fix:** Add pre-save validation in the FMEA view (`fmea_views.py`) to return 400 if any revised score is set without the other two.
**ISO Reference:** Not directly ISO — but poor UX erodes trust in the tool.

## ISO Compliance Gaps

### ISO-1: No customer complaint register (§9.1.2)

**Severity:** High
**Location:** Missing model
**Description:** ISO 9001 §9.1.2 requires monitoring customer perception, including complaints. CAPA `source_type` includes `customer_complaint` as an option, but there's no `CustomerComplaint` model to track complaints independently. Customer feedback appears only as free text in Management Review inputs. An auditor would cite this: "Show me your customer complaint register" → doesn't exist.
**Fix:** Create a `CustomerComplaint` model with: source, product/service, date received, date acknowledged, assigned_to, status, root_cause, resolution, satisfaction_followup. Wire to NCR and CAPA as source.
**Priority:** High — this is one of the most commonly cited ISO 9001 findings.

### ISO-2: No risk register (§6.1)

**Severity:** High
**Location:** Missing model — RISK-001 standard exists but no implementation
**Description:** ISO 9001 §6.1 requires determining risks and opportunities and planning actions to address them. Currently, risks are mentioned only in Management Review inputs as free text. There's no structured risk tracking, no risk mitigation actions, no risk re-evaluation schedule. The RISK-001 standard is listed in the standards library but has no implementation.
**Fix:** Create `Risk` model with: description, category (operational/quality/compliance/strategic), likelihood (1-5), impact (1-5), risk_score, owner, mitigation_actions, review_date, status. Link to FMEA (FMEA is risk assessment for product/process, this is organizational risk).
**Priority:** High — second most commonly cited finding after customer complaints.

### ISO-3: No calibration equipment register (§7.1.5)

**Severity:** Medium
**Location:** Missing model
**Description:** ISO 9001 §7.1.5.2 requires measurement traceability — calibrated instruments with known status. Svend has Gage R&R in the DSW but no model to track measurement instruments, calibration schedules, calibration certificates, or out-of-calibration events. An auditor asking "Show me your calibration register" gets nothing.
**Fix:** Create `MeasurementEquipment` model with: name, serial_number, type, location, calibration_interval_months, last_calibration_date, next_calibration_due, calibration_provider, status (in_service/out_of_service/out_of_calibration), linked Gage R&R studies. Add to ISO QMS tab.
**Priority:** Medium — critical for manufacturing, less so for service orgs.

### ISO-4: Management review completion without required inputs (§9.3.2)

**Severity:** Medium
**Location:** `iso_views.py` — review status transitions
**Description:** The one Management Review in the system is `scheduled` with populated `data_snapshot` but empty `inputs`, empty `outputs`, and empty `attendees`. The status can advance to `complete` without populating the ISO 9001 §9.3.2 required inputs (status of previous actions, changes in external/internal issues, customer feedback, process performance, audit results, supplier performance, resource adequacy, risk/opportunity actions). This means a user could mark a review complete with nothing filled in — which is exactly the kind of finding a registrar loves.
**Fix:** Add transition gate: `complete` status requires at least `attendees` (non-empty), `outputs` (non-empty), and a minimum set of `inputs` keys populated.
**Priority:** Medium — easy fix, high audit value.

### ISO-5: Audit finding → NCR cascade not enforced (§9.2.2)

**Severity:** Medium
**Location:** `iso_views.py` — audit finding creation
**Description:** `AuditFinding` has `ncr` FK and `status` choices including `corrective_action_required`, but there's no enforcement that major NCs generate NCRs. The cascade is manual. ISO 9001 §9.2.2 requires audit findings to "result in correction and corrective action" — the system should at minimum flag major findings without linked NCRs.
**Fix:** Two options: (a) Auto-create NCR when finding type is `nc_major`, or (b) Add a dashboard warning showing major findings without linked NCRs. Option (b) is lighter and more flexible.
**Priority:** Medium — the link exists, just needs enforcement.

## Integration Gaps

### INT-1: SPC → NCR closed loop missing

**Severity:** High
**Location:** `spc_views.py` / `iso_views.py` — no connection
**Description:** The landing page and marketing copy claims "closed-loop QMS" — audit findings auto-create NCRs, NCRs auto-create corrective actions with statistical evidence. But there's no path from an SPC out-of-control signal to NCR creation. The FMEA ↔ SPC bridge exists (`spc-update`, `spc-cpk-update` endpoints), but SPC alarms don't trigger NCR creation. A control chart violation should optionally create an NCR with source=`process`.
**Fix:** Add an endpoint or automation: when SPC detects a special cause (rule violation), offer/auto-create an NCR with pre-populated fields (measurement, chart type, rule violated, timestamp). Link back to the DSW result.
**Priority:** High — this is the central "closed loop" claim. Without it, the marketing overpromises.

### INT-2: FMEA → CAPA direct path missing

**Severity:** Medium
**Location:** `fmea_views.py`
**Description:** FMEA rows can promote to action items and link to hypotheses, but can't directly raise CAPAs. For high-RPN failure modes (especially S≥9), ISO/IATF expects corrective action — not just an action item. The path is: high RPN → action item → hope someone creates a CAPA separately.
**Fix:** Add `promote-capa` endpoint on FMEA rows (similar to `promote-action`). Pre-populate CAPA with failure mode, root cause from FMEA, severity mapping.
**Priority:** Medium — especially important for IATF 16949 (automotive).

### INT-3: Hoshin KPIs don't pull QMS metrics

**Severity:** Low
**Location:** `hoshin_views.py` / `xmatrix_views.py`
**Description:** HoshinKPI tracks dollar savings and operational metrics but can't pull from QMS data (NCR open/close counts, CAPA closure rate, audit finding counts, training completion %). A quality director running Hoshin Kanri would expect quality KPIs alongside operational ones in the X-matrix.
**Fix:** Add `qms_*` calculator_result_types to HoshinKPI: `qms_ncr_open`, `qms_capa_closure_rate`, `qms_audit_findings`, `qms_training_completion`. These query ISO models directly for the `latest` aggregation mode.
**Priority:** Low — nice-to-have for enterprise, not blocking.

## Design Issues

### DES-1: Supplier disqualification is terminal

**Severity:** Low
**Location:** `agents_api/models.py` SupplierRecord state machine
**Description:** Once a supplier is `disqualified`, there's no transition path back — not even to `pending` for re-evaluation. In practice, suppliers do get re-qualified after corrective action (especially sole-source or strategic suppliers). The state machine makes this impossible without direct database manipulation.
**Fix:** Add transition: `disqualified → pending` (requires `notes` explaining re-qualification rationale). This matches ISO 9001 §8.4 which doesn't prohibit re-qualification.

### DES-2: Training requirement → document version not linked

**Severity:** Medium
**Location:** `agents_api/models.py` TrainingRequirement
**Description:** `TrainingRequirement` has `document` FK and `document_version` field, but there's no trigger to flag when the linked document is revised. If a procedure changes from v1.0 to v2.0, all training records against that requirement should be flagged for renewal. Currently the system is passive — version mismatches go undetected.
**Fix:** On `ControlledDocument` approval (version bump), check for `TrainingRequirement` records pointing to that document. If `document_version != document.current_version`, flag affected training records as needing renewal. Surface as dashboard warning.

### DES-3: Electronic signatures not wired to workflows

**Severity:** Medium
**Location:** `agents_api/models.py` ElectronicSignature — 0 records
**Description:** The `ElectronicSignature` model exists with hash chain backing and 21 CFR Part 11 references, but no QMS workflow actually requires or creates signatures. Document approval, NCR closure, CAPA verification, management review sign-off — none trigger e-signature creation. The model is dead code.
**Fix:** Wire e-signatures into at minimum: (a) ControlledDocument `approved` transition, (b) NCR `closed` transition, (c) CAPA `closed` transition. Make optional for Team tier, required for Enterprise.

### DES-4: NCR can reach verification without documented corrective action

**Severity:** Medium
**Location:** `agents_api/models.py` NonconformanceRecord.can_transition()
**Description:** The NCR state machine allows `capa → verification` with only `approved_by` required. Neither `corrective_action` (text field) nor `capa_report` (FK) are required. ISO 9001 §10.2.1(b) requires "implement any action needed" before evaluating effectiveness. A user can advance an NCR through the full lifecycle with no corrective action documented.
**Fix:** Add gate on `verification` transition: require either `corrective_action` (non-empty) or `capa_report` (non-null). At least one must be populated.

## What's Strong (Keep / Protect)

These are genuine differentiators — don't break them in pursuit of gaps:

1. **FMEA ↔ Hypothesis evidence bridge (CANON-001)** — No competitor has this. FMEA findings create Bayesian evidence that updates hypothesis probabilities. Protect this.
2. **RCA AI critique engine** — Claude challenges root cause chains for blame framing, circular logic, and premature stops. Unique in the market.
3. **State machines with transition gates** — NCR, CAPA, Document, Supplier all have proper workflow enforcement. Well-designed.
4. **QMSFieldChange audit trail** — Field-level change tracking across all QMS records. Auditors love this.
5. **Management review auto-snapshot** — Automatically captures QMS metrics at review time. Smart design.
6. **Full SPA in iso.html** — 2430 lines, all 7 modules, CRUD + detail views + status workflows. Comprehensive.
